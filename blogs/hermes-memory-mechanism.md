# Hermes 的记忆机制：不是一个脑子，是三只抽屉

> 作者：Shoa Lin
>
> 发布日期：2026年5月31日

---

Hermes 的记忆，不是"我全都记得"的魔法大脑，更像一组分工明确的货架。最上层是**短期上下文**：刚刚聊过什么、工具跑了什么、当前任务做到哪一步，都先堆在这里。它反应快，但也容易爆仓，像桌面上临时摊开的便签。

中间是**文件记忆**：`MEMORY.md` 和 `USER.md`。前者记环境、项目习惯、工具坑；后者记用户偏好、沟通方式、长期习惯。比如"用户喜欢中文""不要一上来就大改代码"，这种就值得放进去。它不是流水账，更像冰箱门上的重要贴纸。

底层是**会话历史**：过去聊过的内容会进数据库，需要时可以搜索回来。它不一定每天自动浮现，但你问"上次那个问题怎么处理的"，Hermes 就能去旧档案柜里翻一翻。

![三层记忆货架](blogs/images/hermes-memory/01-three-layer-memory-shelf.png)

有趣的是，Hermes 的长期记忆虽然会立刻写入文件，但不会马上改掉当前对话里的"系统提示词快照"。这像把一张说明书冻在玻璃罩里：你可以在旁边继续写新便签，但玻璃罩里的那张纸，要等下一次重建或压缩时才会更新。

这不是健忘，而是省钱又稳定。因为系统提示词一旦频繁变化，模型缓存就会失效，成本和延迟都可能上来。Hermes 的做法是：先把记忆安全落盘，当前会话别乱晃，下一轮再优雅地带上。

![冻结的玻璃罩](blogs/images/hermes-memory/02-frozen-system-prompt-snapshot.png)

那外部记忆提供商怎么办？比如 mem0、Supermemory 之类。它们不会直接冲进系统提示词里改装修，而是走"旁门"：先根据当前问题召回相关内容，再临时贴到这次用户消息后面。

也就是说，系统提示词保持不动，召回内容只作为这次对话的背景材料出现。像小黑从侧门递来一张纸条："喂，这个用户以前说过这个。"递完就走，不抢主位。

![召回内容走旁门](blogs/images/hermes-memory/03-prefetch-side-door.png)

所以，Hermes 的记忆机制可以简单理解为：

**当前上下文负责临场反应，文件记忆负责长期习惯，会话历史负责旧事检索，外部 provider 负责临时召回。**

它不是一个永远清醒的超级大脑，而是一套有点认真、有点好笑的记忆后勤系统。小黑在里面爬货架、敲玻璃、贴纸条，努力让你少重复一句："我不是早说过了吗？"
