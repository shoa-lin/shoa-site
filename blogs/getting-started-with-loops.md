最近很多人都在讨论一件事：不要只是不断 prompt 编码 agent，而是设计 loop，让 agent 在一套边界里自己推进工作。问题在于，loop 这个词很容易被说得很玄。Claude Code 团队在这篇文章里给了一个更工程化的定义：loop 是 agent 重复执行工作循环，直到满足停止条件。

这个定义的重点有两个。

第一，loop 不只是“自动化”。自动化强调触发，loop 还必须关心停止条件、验证方式、任务边界和成本。

第二，loop 不一定很复杂。很多任务只需要最简单的 turn-based loop，也就是你发一个 prompt，Claude 读代码、改代码、检查结果，然后把结果交还给你。只有当任务变长、变重复、需要明确退出条件，或者需要长期监听外部系统时，才值得升级到更强的 loop 形态。

## 四种 loop

原文把 Claude Code 里的 loop 大致分成四类：Turn-based loop、Goal-based loop、Time-based loop 和 Proactive loop。可以按三个问题理解它们：

1. 谁触发它？
2. 它什么时候停？
3. 哪类任务值得用它？

### Turn-based loop

Turn-based loop 是最常见的使用方式：你发起一个 prompt，Claude 完成一轮 agentic work，再把结果交还给你。

它适合短任务、探索型任务、一次性的改动，或者还没想清楚成功标准的任务。比如你让 Claude 加一个按钮，它会读代码、修改、运行测试，然后给你一个它认为可用的结果。接下来你检查，再决定下一步怎么 prompt。

这类 loop 的风险是验证容易靠人肉补位。你可以把常见的人工检查写进 `SKILL.md`，让 Claude 每次做 UI 改动后都知道要启动本地服务、打开页面、点击控件、检查 console、截图对比，甚至跑性能 trace。这样做不是为了让 Claude 完全替代你，而是把你反复做的检查标准化。

我自己的理解是：turn-based loop 的核心不是“多聊几轮”，而是把每一轮变得更可验证。如果你发现自己每次都在提醒 agent “先跑测试”“看浏览器”“不要只说改好了”，那就应该把这些提醒沉淀进 skill 或项目规则。

### Goal-based loop

Goal-based loop 对应 Claude Code 的 `/goal`。它解决的是一个很具体的问题：单轮不够，但你又知道“完成”长什么样。

例如，你可以给出一个明确目标：把首页 Lighthouse 分数提高到 90 以上，最多尝试 5 次。Claude 会不断尝试，每次想停下来时，由评估模型检查目标是否真的满足。如果没有满足，就继续回去工作，直到达标或达到你设置的轮次上限。

这里的关键是停止条件必须具体。测试全部通过、lint 清空、性能分数超过阈值、某个 API 返回正确状态，这些都是好条件。相比之下，“优化一下”“让它更好”“尽量漂亮”都太模糊，会让 loop 要么过早停下，要么消耗失控。

我会把 goal-based loop 理解成“把停止权从 agent 的感觉，转移到可检查的条件”。它特别适合 bugfix、性能优化、测试补全、迁移收尾这类有明确验收标准的任务。

### Time-based loop

Time-based loop 对应 `/loop` 和 `/schedule`。它适合那些输入会随时间变化的任务。

比如每天早上总结 Slack 消息，每隔 5 分钟检查一个 PR 有没有新 review 或 CI 失败，定期扫 issue 队列，或者观察某个外部系统的状态变化。这个时候，任务本身不是一次性完成，而是按时间间隔重复触发。

`/loop` 更像本机运行的周期任务：你的电脑关掉，它也就停了。`/schedule` 则把 routine 放到云端，更适合长期运行。

这类 loop 最容易带来 token 浪费。因为它会按时间醒来，所以间隔设计很重要。一个小时变化一次的队列，不需要每分钟检查。更好的策略是按真实变化频率设置周期，或者在能做事件触发时优先事件触发。

### Proactive loop

Proactive loop 是更完整的主动循环。它通常由事件或 schedule 触发，不需要人在实时对话里盯着。每个子任务都有自己的目标，整个 routine 会一直运行，直到你关闭它。

原文给的形状是：用 `/schedule` 定期检查新的反馈或 bug report，用 `/goal` 定义“每条反馈都要被分诊、处理、回应”，用 dynamic workflows 编排多个 agent 去并行分诊、修复和 review，再配合 auto mode 减少人工许可阻塞。

这其实就是从“我让 agent 做一件事”，升级成“我设计一个小型工作系统”。它适合边界清楚、重复出现、有明确处理流程的工作流，例如 bug report、issue triage、依赖升级、批量迁移、反馈处理。

但 proactive loop 也最危险。它越主动，就越需要严格的权限边界、验证步骤和人工抽检。否则你得到的不是自动化工程，而是自动化地制造混乱。

## 质量来自系统，不来自一次聪明回答

文章很强调一点：loop 的输出质量取决于围绕它的系统。

如果代码库本身混乱，Claude 会学到混乱。如果没有验证路径，它只能靠猜。如果文档拿不到，它会用旧知识填空。如果让同一个 agent 既实现又评价自己的实现，它会有自我偏好。

所以一个健康的 loop 至少需要这些支撑：

1. 清晰的代码模式和项目约定。
2. 可执行的验证方式，例如测试、lint、浏览器检查、性能分数。
3. 易访问的框架和库文档。
4. 独立 review agent，最好带新上下文和明确 rubric。
5. 失败后的规则沉淀，而不是只修单次问题。

最后一点很重要。loop 运行失败时，不要只问“这次怎么修”。更应该问：这个失败能不能变成下一轮不会再犯的项目规则、skill、测试或脚本？

## Token 成本怎么管

Loop 的风险不只是质量，也包括成本。越主动、越并行、越多 agent，token 消耗就越容易膨胀。

原文给出的几个原则很实用：

1. 为任务选对 primitive 和模型。小任务不要上多 agent 或复杂 workflow。
2. 设定明确成功标准和停止标准。越具体，越少空转。
3. 大规模运行前先小范围试跑。尤其是 dynamic workflows，可能一次拉起很多 agent。
4. 能用脚本解决的确定性工作，就不要让模型每次重新推理。
5. 周期任务不要跑得比真实变化更频繁。
6. 经常看 usage，知道 token 花在 skills、subagents、MCP 和 workflow 的哪个部分。

这和我之前对 loop engineering 的理解是一致的：loop 不是“让 AI 一直跑”，而是把不确定的人工推动，变成有边界、有停止条件、有观测指标的系统。

## 怎么开始

可以从一个很小的问题开始：找一件你已经反复做、而且你自己正在成为瓶颈的事。

然后问三个问题：

1. 我能不能把检查步骤写清楚？
2. 这个任务有没有明确的完成条件？
3. 这个工作是不是按时间或事件反复出现？

如果只是探索，先用 turn-based loop，把验证 skill 写好。

如果完成条件很明确，用 goal-based loop。

如果任务按周期到来，用 time-based loop。

如果任务重复、边界清楚、可以标准化处理，再考虑 proactive loop，把 schedule、goal、workflow、review 和权限边界组合起来。

我的结论是：loop 的价值不在于它看起来多自动，而在于它把人类工程师的判断前置到了系统设计里。你不是少思考了，而是把思考从“下一句 prompt 怎么写”，移动到了“触发、停止、验证、成本和权限怎么设计”。

这也是为什么 loop 比 prompt 更像工程：prompt 是一次请求，loop 是一个运行系统。
