エージェントの記事の多くは、**モデルにタスクを完了させる方法**と**システムを確実に作成する方法**という 2 つのことをまとめて説明しています。その結果、たくさんの名詞を覚えてしまい、実際にデザインを始めるときに、どの問題を最初に解決すればよいのかわかりません。

この記事では、エージェントが実際の環境でどのように判断し、行動し、協力し、制御可能であり続けるかという最初の点のみに答えます。これはタスク マップとして考えることができます。つまり、1 人のエージェントの推論ループから始まり、徐々に複数人のコラボレーション、記憶、知識が加わり、最後に本番運用に必要な安全性とフォールト トレランスが追加されます。

読むときにスキーマ名を覚える必要はありません。各セクションが本当に答えたいのは次のとおりです: **この種のタスクではどのような種類の不確実性を排除する必要がありますか? ** 時には次に何をすべきか、時には誰がそれを行うか、時にはどうやって思い出すか、時にはどうやって取り返しのつかない結果を回避するかが重要です。

## 最初の層: シングルエージェントコアモード

### 01 ReAct — 論理と行動が絡み合う

**論文**: Yao et al.、2022 — _ReAct: 言語モデルにおける推論と行動の相乗効果_

💡 核となるアイデア

考えてからやるのではなく、やりながら考えてください。 LLM は **思考** と **行動** を交互に生成し、結果を観察した後推論を続けます。

> 感想: ユーザーは今日の北京の天気を知りたいと考えています。探す必要があります。
> アクション: search("2026 年 4 月 9 日の北京の天気")
> 観測：晴れ、18℃、北の風レベル3
> 感想：気象データはあるので答えられます。
> 回答：今日の北京は晴れ、気温は18℃、北風はレベル3です。

**機能する理由**:

- **推論が行動を導く**: 盲目的な試みを減らすために、各行動ステップには「なぜこれを行う必要があるのか」を説明する考えがあります。
- **アクションは推論をフィードバックします**: 観察により現実世界の情報が得られ、その後の推論がより正確になります。
- **説明可能性**: 思考リンク自体は自然な「意思決定ログ」です。

**該当するシナリオ**: 複数ステップの推論 + ツールの呼び出しを必要とするタスク (質疑応答、分析、デバッグ)

**制限事項**: 長いチェーンは「逸脱」する傾向があり、後続の観測は元のターゲットから逸脱する可能性があります。方向を制約するには、リフレクション モードまたはプランニング モードと連携する必要があります。

### 02 計画と実行 — 行動を起こす前に明確に考える

💡 核となるアイデア

完全な実行計画を生成し、それを段階的に監視するのではなく、段階的に実行します。 ReAct の「考えて実行する」とは異なり、**計画**と**実行**が 2 つの明確な段階に分かれています。

> [計画段階]
> ユーザー: 競合製品 HireGo の最新開発の分析を手伝ってください
>計画:
> 1. HireGo公式サイトを検索して製品情報を入手する
> 2. Google Play / App Store のレビューを検索する
> 3. 過去 30 日間の関連レポートをテクノロジー メディアで検索する
> 4. LinkedIn で採用に関する最新情報を検索する (チーム拡大の方向性を決定するため)
> 5. 上記の情報に基づいて、競合製品に関するインテリジェンスに関する説明を作成します。
>
> [実行フェーズ]
> ステップ 1 を実行 → 観察 → ステップ 2 を実行 → 観察 → ... → 最終レポート

**キーデザイン**:

- **計画は動的に調整できます**: ステップ 2 の実行中に新しい手がかりが見つかった場合は、戻ってステップ 1 の検索を補足できます。
- **ReAct を補完**: 計画と実行は「グローバルなビジョン」を提供し、ReAct は「ローカルな柔軟性」を提供します。
- **実際のプロジェクトではよく混合されます**: まず大まかな計画を生成し、それを各ステップで ReAct モードで実行します

**適用可能なシナリオ**: 明確なプロセスを伴う複数ステップのタスク (調査レポート、データ処理パイプライン、コード リファクタリング)

**制限事項**: 当初の計画は完璧ではない可能性があります。環境が大幅に変化すると、静的な計画は時代遅れになります。

### 03 振り返り — 終わって振り返って

💡 核となるアイデア

エージェントに **自身の出力を評価**させ、欠陥を見つけた場合は自ら修正します。これは最も「人間的」なモードかもしれません。問題を終了し、再度確認して間違いを見つけて修正します。

> [1回目の出力]
> 「NVIDIA の 2026 年第 1 四半期の収益は 280 億米ドルで、前年比 78% 増加」
>
> [反省]
> 待って、確認させてください: 昨年の第 1 四半期の 280 億対 260 億...
> 280/260 - 1 = 7.7%、78% ではありません。 2024 年第 1 四半期との対前年比は 78% です。
> 比較のベースラインを明確にラベル付けする必要があります。
>
> [補正出力]
> 「NVIDIA の 2026 年第 1 四半期の収益は 280 億米ドルで、前年比 7.7% (前四半期比) 増加、前年比 78% (前年比) 増加しました。」

**Andrew Ng の典型的なケース**: LLM にコードを書かせて、テストを実行して、エラーを LLM にフィードバックし、LLM がコードを修正して、再度テストします。このサイクルは、合格するまで複数回繰り返すことができます。

**実装ポイント**:

- フィードバック信号は **外部** (テスト結果、ユーザー フィードバック、ツールの戻り値) または **内部** (LLM 自己評価) にすることができます。
- 反映の頻度は制御する必要があります。毎回反映すると時間の無駄になります。(コードが記述されてレポートが生成された後) キーノードでのみ反映されます。
- 明確な「完了基準」が必要です。そうでないと、エージェントは無限ループで「反映」されてしまいます。

**該当するシナリオ**: コード生成、作成、データ分析 - 「品質が重要」なタスク

### 04 ツールの使用 — エージェントに腕と脚を成長させます

💡 核となるアイデア

LLM 自体は、**関数呼び出し / ツールの使用** を通じて現実世界のツールで動作できるようにする単なるテキスト ジェネレーターです。これはすべてのエージェント システムの **インフラストラクチャ**です。ツールの使用がなければ、エージェントは存在しません。

> ユーザー: 「明日の朝 9 時の会議についてリマインドするようにスケジュールされたリマインダーを作成してください」
>
> LLM 内部の意思決定:
> 呼び出す必要があります → cron_schedule(
> メッセージ: 「リマインダー: 9 時に集合」、
> 時間: "2026-04-10T09:00:00+08:00"
>)
>
> → 通話成功 → 「リマインダーが設定されました: 明日の 9:00 — 会議をリマインドするためです」

**ツールの分類**:

|カテゴリー |例 |特徴 |
| --- | --- | --- |
| **情報検索** | web\_search、web\_fetch、read\_file |読み取り専用、副作用なし |
| **アクションの実行** | \_メッセージの送信、\_ファイルの書き込み、実行 |副作用がありますので注意してください |
| **対話型ツール** |ブラウザ、メッセージ、キャンバス |双方向コミュニケーション |
| **電卓** |電卓、コード\_インタープリター |確定的な出力 |

**エンジニアリングの実践**:

- **ツールの説明の品質によって通話の品質が決まります**: LLM がツールを正しく選択できるかどうかは、ツールの説明が適切に記述されているかどうかに 90% 依存します。
- **最小特権の原則**: エージェントは現在のタスクに必要なツールのみを必要とし、それらを一度にすべて与えないでください。
- **エラー処理が核心**: ツール呼び出しの失敗は日常茶飯事であり、エージェントは失敗を適切に処理できなければなりません (再試行、フォールバック、レポート)。

### 05 思考連鎖と思考樹

**思考の連鎖**: LLM に、答えを直接与えるのではなく、「思考プロセスを書き出す」よう依頼します。

> 悪い習慣: 「答えは 42」 (ブラックボックス)
> 良い習慣: 「x を... として、式に代入して... を得る。したがって、答えは 42 になる」 (プロセスは表示されます)

これは独立したエージェント モードではありませんが、すべてのモードの基礎となるインフラストラクチャ (ReAct の思考、計画と実行の計画、およびリフレクションの評価) はすべて CoT に依存しています。

**思考の木**: CoT のアップグレード バージョン - 単なる 1 つの思考チェーンではなく、**複数のパスを探索し、最善のものを選択します**。

> 質問: 「検索 API の遅延を 2 秒から 200 ミリ秒に短縮するにはどうすればよいですか?」
>
> パス A: より高速な API プロバイダーに変更 → コストが 3 倍増加 → 非推奨
> パス B: キャッシュ層を追加 → ヒット率が不明 → 評価が必要
> パス C: 複数の API を同時にリクエストして最速のリターンを得る → 複雑さが増す → 試す価値がある
>
> 評価: コストが制御可能であればパス C で目標を達成できる可能性がある → パス C を選択

**該当するシナリオ**: 複数のソリューションを検討する必要がある複雑な意思決定 (アーキテクチャ設計、ソリューションの選択)

## 第 2 レベル: マルチエージェント コラボレーション モード

1 人のエージェントでは十分な能力がない場合、複数の専門エージェントが協力する必要があります。これはオーケストレーションの領域に入ります。

### 06 スーパーバイザー — スーパーバイザーがタスクを割り当てる

「スーパーバイザーエージェント」は、ユーザーの意図を理解し、タスクを細分化し、実行のために専門のエージェントに配布する責任があります。

> ユーザー: 「AI 業界に関する週次レポートの作成を手伝ってください」
>
> スーパーバイザーエージェント:
> → 情報収集エージェント：「今週のAI業界の重大ニュースを収集します」
> → 金融エージェント「今週のAI関連企業の株価動向をまとめて」
> → 研究エージェント: 「今週重要な論文を集めてください」
> ← すべてのエージェントの出力を要約 → 週次レポートに統合 → ユーザーに送信

💡 **一般的な方法**は、スーパーバイザーエージェントが情報収集、財務、調査などの専門エージェントを調整することです。

**利点**: 責任が明確で、各エージェントは自分の分野に集中します。
**短所**: スーパーバイザが単一点のボトルネックになります。タスクの分解が間違っていると、パイプライン全体が軌道から外れてしまいます。

### 07 階層 — 階層管理

スーパーバイザのアップグレード バージョン - **マルチレベルのコマンド チェーン**。

```text
协调 Agent
├── 产品负责人 Agent — 负责产品相关任务
│   ├── 竞品分析 Agent — 数据搜集
│   └── 用户研究 Agent — 数据搜集
├── 技术负责人 Agent — 负责技术任务
│   ├── 前端开发 Agent
│   └── 后端开发 Agent
└── 运营负责人 Agent — 负责运营任务
    ├── 数据分析 Agent
    └── 内容创作 Agent
```

**スーパーバイザーとの違い**: スーパーバイザーは「フラット」です。1 人の上司が全員を直接管理します。階層は「ツリー状」であり、より大きなタスクを処理できる中間管理層があります。

### 08 Swarm — 分散型ミツバチの群れ

中央制御はなく、エージェント間のコラボレーションは **ハンドオフ** を通じて完了します。

> ユーザー: 「北京から上海までの航空券を予約したいのですが」
>
> ルーティング エージェント:
> 意図を分析 → 「フライト予約」として識別される
> → チケット係員にカードを渡します
>
> チケット販売代理店:
> フライトを検索 → ログインが必要であることが判明
> → ハンドカードを認証エージェントに渡します
>
> 認証エージェント:
> ユーザーにログインを誘導 → ログイン成功
> → ハンドカード返却チケット取扱店
>
> チケット販売代理店:
> 予約手続きを続ける → 完了

**ハンドオフ**: エージェント A がその部分を完了すると、エージェント B にコンテキストを渡して続行します。グローバルマネージャーは存在せず、各エージェントは自らの責任のみを負います。

**利点**: 柔軟でスケーラブルで、単一点のボトルネックがない
**欠点**: プロセスを追跡するのが困難です。ハンドオフ ロジックが適切に設計されていない場合、タスクはエージェント間で「ピンポン」を起こします。

**典型的な実装**: OpenAI Swarm フレームワークは、このパターンに基づいて設計されています。

### 09 黒板 — 共有黒板

複数のエージェントが共有黒板を中心に作業し、各エージェントが黒板に情報を書き込んだり、他の人の情報を読んだりします。

```text
[共享黑板]
┌─────────────────────────────────┐
│ 用户需求: "做一个天气预报 App"    │
│                                 │
│ [搜索 Agent 写入]               │
│ 天气API: OpenWeatherMap 免费    │
│ 地理API: Nominatim              │
│                                 │
│ [设计 Agent 写入]               │
│ 技术栈: React + Node.js        │
│ 架构: 三层（UI/API/数据）        │
│                                 │
│ [代码 Agent 读取黑板后开始编码]  │
│ ...                            │
└─────────────────────────────────┘
```

**スーパーバイザーとの違い**: スーパーバイザーは「コマンド駆動型」です。スーパーバイザーは誰が何をするかを指示します。 Blackboard は「データ駆動型」です。処理できる情報を見た人は誰でも、積極的にその情報を処理します。

### 10 パイプライン / DAG — パイプライン

タスクは固定ステージに分割され、各ステージはエージェントによって処理され、データはそのステージを順番に流れます。

> 入力 → [エージェント収集] → [エージェント分析] → [エージェント書き込み] → [エージェント監査] → 出力
> 生データ 構造化分析 レポート作成 品質チェック

- 各ステージは並列処理可能 (ステージ間に依存関係がない場合)
- 明確なインターフェイスを介してステージ間でデータを渡す
- モニタリングとデバッグが容易 (各ステージの入力と出力が決定されます)

**制限事項**: 柔軟性が低い - プロセスは固定されており、中間結果に基づいて動的に調整することはできません。

## 第 3 レベル: 記憶と知識モデル

エージェントには「本当の記憶」がなく、すべての会話が新鮮に始まります。メモリ モードは、セッション間での情報保持の問題を解決します。

### 11 短期記憶 - 現在の会話のコンテキスト

**本質**: LLM のコンテキスト ウィンドウです。

> 現在の会話:
> ユーザー: 「NVIDIA の分析を手伝ってください」
> エージェント: [データを取得するために検索ツールを呼び出しています]
> ユーザー: 「AMD についてはどうですか?」 ← エージェントは短期記憶により、「分析」が財務分析を指すことを認識しています

**エンジニアリングの課題**:

- **限られたコンテキスト ウィンドウ**: 長い会話では前の内容が「忘れ」られます (GPT-4 128K、Gemini 1M、GLM 128K)
- **ノイズの蓄積**: 会話が長くなるにつれて、無関係な情報が増え、推論の質に影響します。
- **コスト**: トークンの数 = お金、長いコンテキスト = 高コスト

**エンジニアリングの実践**: 会話の圧縮 (定期的な要約)、スライディング ウィンドウ (最新の N ラウンドのみが保持される)、検索の強化 (オンデマンドで関連するフラグメントの検索)

### 12 長期記憶 - セッション全体にわたる知識の保持

エージェントがセッション間で情報を保持できるようにします。つまり、あなたが誰であるか、何をしたか、そしてあなたの好みを覚えておくことができます。

|方法 |原則 |利点 |デメリット |
| --- | --- | --- | --- |
| **ファイルメモリ** | MEMORY.md / .learnings/ の読み取りと書き込み |シンプル、透明性、監査可能 |大まかな手動メンテナンス |
| **ベクトル メモリ** |情報は埋め込み後にベクトルデータベースに保存されます。意味検索、自動関連付け |追加のインフラストラクチャが必要です |
| **構造化記憶** |ナレッジ グラフ、リレーショナル データベース |正確なクエリ、強力な推論能力 |高い建設費 |

💡 **ドキュメント記憶の一般的な実践**: 長期記憶ドキュメントを使用して安定した事実を保存し、元のコンテキストを日付ごとにファイルされたレコードとして保存し、学習ログを使用してエラーと改善を蓄積します。

### 13 RAG — 検索拡張の生成

LLM にすべての知識を「記憶」させるのではなく、必要に応じて外部の知識ベースから関連情報を取得し、それをプロンプトの単語に挿入します。

> ユーザー: 「2026 年 4 月 2 日に何を話し合いましたか?」
>
> → ベクトルデータベース検索: query="2026-04-02 Discussion"
> → 取得: 関連する記憶の断片/2026-04-02.md
> → プロンプトに挿入: 「次のコンテキストに基づいてユーザーの質問に答えます: [検索結果]」
> → LLM が回答を生成します

💡 **RAG はメモリモードの「検索エンジン」** - 「保存方法」を解決するのではなく、「必要な情報を素早く見つける方法」を解決します。

## 第 4 層: 実稼働レベルのエンジニアリング モード

学術的なプロトタイプと使用可能な生産システムの間には、エンジニアリングの実践があります。

### 14 ガードレール — 安全ガードレール

エージェントの動作に**境界制約**を設定して、エージェントが実行すべきでないことを実行しないようにします。

```text
[输入护栏]                          [输出护栏]
用户输入 → ┌──────────┐ → LLM → ┌──────────┐ → 最终输出
          │ 过滤敏感词  │         │ 验证事实   │
          │ 检测注入    │         │ 检查格式   │
          │ 限制话题    │         │ 过滤有害内容│
          └──────────┘         └──────────┘
```

|レイヤー |例 |
| --- | --- |
| **入力確認** |プロンプト インジェクションを検出し、機密性の高い命令をフィルターします。
| **ツールの権限** | exec コマンド ホワイトリスト、ファイル書き込み操作には確認が必要です。
| **出力監査** |ファクトチェック、フォーマット検証、機密情報のフィルタリング |
| **行動の制約** |エージェントが特定のデータ ソースにのみアクセスし、外部メッセージを送信できないように制限します。

### 15 Human-in-the-Loop — ループ内の人々

エージェントは完全に自律的ではないため、**重要な決定には人間の確認が必要です**。

> エージェント: 「合計 342 個のファイルを含む /tmp/old-data/ ディレクトリを削除します。削除してもよろしいですか?」
> 人間: [確認] / [拒否] / [変更: .log ファイルのみ削除]

|決定の種類 |自動化の程度 |理由 |
| --- | --- | --- |
|ファイルの読み取りと検索 |全自動 |副作用なし |
|指定した場所にファイルを書き込む |半自動 |既存のコンテンツを上書きする可能性があります |
|他の人にメッセージを送信する |確認が必要です |外部からの影響がある |
|データの削除 |要確認 |不可逆的な操作 |
|シェルコマンドを実行する |リスクごとに分類 |システムのセキュリティに影響を与える可能性があります |

**エンジニアリングの実装**: 承認ゲートは、Human-in-the-Loop の典型的な実装です。

### 16 フォールバックと再試行 — フォールト トレランスとダウングレード

エージェントのすべてのステップが失敗する可能性があるため、システムは失敗を適切に処理できなければなりません。

```text
搜索请求 → web_search(Gemini)
              ↓ 429 限流
           unified-search(Tavily)
              ↓ 无结果
           unified-search(Exa)
              ↓ 全部失败
           返回提示: "搜索服务暂时不可用，请稍后重试"
```

💡 **堅牢な検索レイヤー** は、プライマリ検索ソースに結果がない場合に自動的にバックアップ ソースに切り替え、意味のあるダウングレード結果をユーザーに提供します。

**フォールト トレラント設計原則**:

1. **Fail Fast**: 失敗することが予想される操作を長く待ちすぎないでください。
2. **意味のあるダウングレード**: フォールバック ソリューションは「何もできない」わけにはいきませんが、いくつかの機能を提供する必要があります。
3. **再試行には上限があります**: 無限再試行 = 無限ループ。最大再試行回数を設定する必要があります。
4. **失敗の理由を記録します**: その後の分析と改善のために、失敗を構造化エラー キューに書き込みます。

### 17 自己改善 — 自己進化

エージェントは自分の間違いから学び、継続的に改善することができます。

```text
执行任务 → 出错 → 记录错误 → 分析模式 → 提炼规则 → 下次避免
                                                    ↑
                                              注入到行为中
```

**典型的な実装**:

- ツールの呼び出し後にエラーを自動的に検出し、構造化エラー キューに書き込みます
- プロンプト単語を作成する前に保留中の学習リマインダーを挿入します
- 定期的なレビューを通じてエラーをアーカイブし、繰り返し発生するパターンを検出し、ルールを安定した構成に促進します

💡 **重要な洞察**: 自己進化の核心は「学習」そのものではありません。LLM は本質的にコンテキストから学習します。中心となるのは、手動のリマインダーに頼るのではなく、学習を継続的、体系的かつ自動化することです。

## セレクションガイド

|タスクの特性 |推奨モード |理由 |
| --- | --- | --- |
|簡単なQ&A | **ReAct + ツールの使用** | 1 つまたは 2 つのステップで解決でき、複雑な計画は必要ありません。
|多段階の研究 | **計画と実行 + 対応** |グローバルなビジョンが必要ですが、各ステップでの柔軟性が必要です |
|コード生成 | **ツールの使用 + 振り返り** |実際にコードを実行して修正する必要があります。
|多分野連携 | **スーパーバイザー/階層** |専門的な分業が必要 |
|高度な同時処理 | **スウォーム / パイプライン** |中央の調整は必要なく、並列化可能 |
|知識集約 | **RAG + 長期記憶** |大量の知識から検索する必要がある |
|セキュリティに注意 | **ガードレール + 人間参加型** |制約と確認が必要 |
|長期運用 | **自己改善 + フォールバック** |間違いから学び、耐障害性が必要 |

## 概要: 最初にタスク モードを選択し、次にコード構造について説明します。

この時点で、エージェントがどのように機能するかを決定できるはずです。実行中に考える必要があるのか、それとも計画してから実行する必要があるのか。監督者によって調整される必要があるか、それとも専門のエージェントによって自律的に引き継がれる必要があるか。何を記憶する必要があるか、確認のためにどの主要なアクションを停止する必要があるか。

これらの選択によって、システムの動作の境界が決まります。次のステップは、これらの機能を、置換可能、監視可能、および回復可能なコード構造に編成することです。
