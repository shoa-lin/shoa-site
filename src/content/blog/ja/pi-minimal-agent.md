*2026 年 1 月 31 日執筆*

OpenClaw は ClawdBot や MoltBot など、いくつかの名前で急速に広まりました。その本質は、通信チャネルに接続され、コードを実行できるエージェントです。

OpenClaw の内部では、**Pi** という小さなコーディングエージェントが使われています。Armin Ronacher は、現在ほぼ Pi だけをコーディングエージェントとして使っていると述べ、その意図的に小さく保たれた設計がなぜ魅力的なのかを説明しています。

Pi を作ったのは **Mario Zechner** です。Mario の地に足のついたアプローチは、Peter の「少し狂気を帯びた SF」とは異なります。それでも Pi と OpenClaw は、LLM はコードの記述と実行を非常に得意としており、システムはその能力を積極的に活かすべきだ、という前提を共有しています。

## Pi とは何か

コーディングエージェントは数多くありますが、Pi を際立たせる特徴は二つあります。Armin は **AMP** についても、単に洗練されたインターフェースで包むのではなく、エージェントによるプログラミングを本気で試してきた人々が作ったプロダクトだと評価しています。

Pi が興味深い主な理由は次の二つです。

- **コアが非常に小さい。** システムプロンプトは例外的に短く、コアが公開するツールは Read、Write、Edit、Bash の四つだけです。
- **強力な拡張システムを持つ。** 拡張は振る舞いを追加できるだけでなく、独自の状態をセッションへ永続化できます。

実用面の利点もあります。Pi は丁寧に作られたソフトウェアらしく、安定していてメモリ消費が少なく、画面のちらつきや予期しない障害で注意をそらしません。

Pi は、別のエージェントを作るための小さなコンポーネント群でもあります。OpenClaw はこれらのコンポーネント上に構築され、Armin は Telegram ボットに、Mario は `mom` に利用しました。Pi 自身のコードと `mom` のような例を Pi に読ませれば、目的の連携方法に合わせた別のエージェントを組み立てる手助けをさせられます。

## Pi に含まれないもの

Pi を理解するには、意図的に含まれていないものも理解する必要があります。コアには**内蔵の MCP サポートがありません**。ただし、MCP を利用できないわけではありません。拡張で追加することも、MCP 呼び出しを CLI または TypeScript バインディングとして公開する **mcporter** をエージェントから使うこともできます。

この省略は Pi の思想を反映しています。エージェントに機能が足りないとき、既製の拡張をマーケットプレイスで探すことを既定の行動にはしません。コードを書いて実行し、自らを拡張するようエージェントに依頼します。

拡張のダウンロード自体はサポートされています。違いは文化的なものです。既存の拡張を不変の依存関係として扱うのではなく、ローカルな要件に合わせてエージェントが作り替えるための参考資料として利用できます。

## エージェントを作るエージェントのための設計

自らを作り替えるソフトウェアには、基盤となるいくつかの能力が必要です。

第一に、Pi の AI SDK では、一つのセッションに異なるモデルプロバイダーからのメッセージを含められます。セッションが完全には移植可能でないことを認めながらも、プロバイダー固有の機能への不要な依存を避けています。

第二に、セッションファイルにはモデルのメッセージと並べて独自メッセージを保存できます。拡張はそれを状態の永続化に使えます。また、その情報をモデルへ一切送らない、あるいは一部だけ含める、といった判断もシステム側で行えます。

第三に、拡張の状態をディスクへ保存でき、拡張はホットリロードに対応しています。エージェントは拡張を書き、再読み込みし、テストして、さらに反復できます。第四に、Pi には自己拡張時にエージェントが読めるドキュメントとサンプルが付属します。第五に、セッションはツリー構造です。ユーザーは別タスクへ分岐し、メインブランチのコンテキストを消費せずに壊れたツールを修正してから元へ戻れます。その際、別のブランチで何が起きたかを Pi に要約させられます。

これらの設計はツールにも影響します。多くのモデルプロバイダーでは、MCP ツールやその他の LLM ツールがセッション開始時にシステムコンテキストまたはツール欄へ読み込まれます。後から定義を完全に置き換えると、キャッシュを破壊したり、以前の呼び出し方法について矛盾した記憶をモデルに残したりする可能性があります。

## モデルコンテキストの外に置くツール

Pi の拡張は LLM から呼び出せるツールを登録でき、Armin も時折この方法を使います。彼がローカルで構築した課題管理ツールが一例です。エージェントが TODO を直接管理する必要があるため、CLI ではなく追加ツールを一つ公開しています。現在、彼がモデルコンテキストへ読み込んでいる追加ツールはこれだけです。

追加機能の大半は、ツールスキーマとしてモデルコンテキストを占有する必要がありません。それらはスキル、または人間の作業体験を改善する TUI 拡張です。Pi の拡張は、スピナー、進捗バー、ファイル選択画面、表、プレビュー画面をターミナル内に直接描画できます。Mario は TUI 上で Doom を動かすデモまで披露しました。実用的ではありませんが、インターフェースの柔軟性を示す好例です。

以下の拡張は、固定のパッケージではなく例です。意図されているのは、エージェントに一つを参照させ、必要な振る舞いへ作り替えさせる使い方です。

### `/answer`

Armin は Plan Mode を使いません。構造化された質問ダイアログに固定するより、エージェントの自然な文章の中に説明や図を織り交ぜながら、生産的にやり取りする方法を好みます。

ただし、文章内の質問が増えると一つずつ答えにくくなります。そこで `/answer` は、エージェントの直前の応答を読み、質問を抽出し、回答しやすい一つの入力ボックスへ整形します。

![/answer 拡張が質問ダイアログを表示している画面](https://lucumr.pocoo.org/static/pi-answer.png)

### `/todos`

Armin は Beads の実装を批判していますが、エージェント向け TODO リストの有用性は認めています。`/todos` は `.pi/todos` 以下に Markdown ファイルとして保存された項目を開きます。ユーザーとエージェントのどちらも編集でき、セッションがタスクを引き受けて進行中としてマークすることもできます。

### `/review`

エージェントがより多くのコードを書くようになれば、未完了の作業は人間へ渡す前にエージェントによるレビューを受けるべきです。Pi のセッションはツリー構造なので、Armin は新しいレビュー用コンテキストへ分岐して指摘事項を集め、その修正をメインセッションへ持ち帰れます。

![/review 拡張がレビュー対象のプリセットを表示している画面](https://lucumr.pocoo.org/static/pi-review.png)

インターフェースは Codex を参考にしており、コミット、差分、未コミットの変更、リモートのプルリクエストなどをレビュー対象にできます。レビュープロンプトは、新しく追加された依存関係を明示的に指摘することなど、Armin が重視するフィードバックを強調します。

### `/control`

これは Armin の日常的なワークフローの一部ではなく、実験的な拡張です。一つの Pi エージェントから別の Pi エージェントへプロンプトを送信し、複雑なオーケストレーション層を使わずに小規模なマルチエージェント構成を作れます。

### `/files`

この拡張は、セッション内で変更または参照されたファイルを一覧表示します。Finder で表示する、VS Code で差分を確認する、Quick Look で開く、プロンプトから参照する、といった操作が可能です。`shift+ctrl+r` を押すと直前に言及されたファイルを Quick Look できるため、エージェントが PDF を生成したときに便利です。

ほかの開発者も拡張を作っています。Nico のサブエージェント拡張や、動作を確認できる TUI オーバーレイ内で Pi が対話型 CLI を自律実行できる `interactive-shell` などがあります。

## ソフトウェアがソフトウェアを作る

重要なのは、Armin がこれらの拡張を手書きしていないことです。彼が要望を説明し、Pi が構築しました。Pi のコアには MCP も、同梱されたコミュニティスキルもありませんが、エージェントは所有者に合わせた機能を作り、保守できます。一例として、ブラウザ自動化 CLI や MCP 連携の代わりに、CDP と直接通信するスキルを使っています。

彼のエージェントには多くのスキルがありますが、どれも使い捨てられます。ほかのエンジニアが共有した Pi セッションをコードレビューのために読むものもあれば、コミットメッセージ、コミット時の振る舞い、変更履歴の更新方法を整えるものもあります。以前のスラッシュコマンドの一部をスキルへ移行しつつあり、`uv` の利用を促すスキルと、`pip` や `python` の呼び出しを `uv` へリダイレクトする拡張を組み合わせてもいます。

これこそが、Pi のようなミニマルなエージェントの魅力です。「ソフトウェアがソフトウェアを作る」ことを、日常的な仕事の進め方にするのです。OpenClaw はローカル UI を取り除き、エージェントをチャットへ接続することで、この考えをさらに先へ進めています。Armin の結論は、あらゆる詳細が決着したというものではありません。ただ、この方向性がソフトウェアの未来の一部にますます見えるようになっている、ということです。
