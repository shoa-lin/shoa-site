## 先に結論

**AIが文章を生成するだけなら、Retryは別の回答を再生成する操作です。しかしAIが現実の環境に働きかけるようになると、Retryはタスクそのものの再実行になり得ます。**

この違いは、CodexのようなAgent製品で従来の「再生成」ボタンが前面に出なくなった背景を説明します。

- ChatbotのRetryは、通常は一つの文章を置き換えるだけです。
- Agentの一回の実行では、すでにファイル変更、コマンド実行、外部ツール呼び出しが行われているかもしれません。
- 二回目の実行が始まる時点で、世界は一回目の前と同じ状態ではありません。
- そのため、漠然とRetryするより、フィードバックを追加する、失敗したステップだけを再実行する、明示的な状態からForkする方が安全です。

ただし、Codexが「状態が複雑になったためRetryを廃止した」と公式に説明しているわけではありません。本稿は、公開されているThread、Turn、Item、Forkの仕組みに基づくアーキテクチャ上の考察です。

![別の回答を生成するRetryから、履歴を残した分岐へ](/assets/blog/ai-agent-retry-state/retry-becomes-fork-ja.png)

以前のChatbotでは、**Retry（再試行、再生成）**という小さな機能にすっかり慣れていました。

質問を送るとAIが答える。回答に満足できなければ、質問を書き直さずにRetryを押すだけで、別の回答が返ってきます。

あまりに自然な操作だったため、Codexを使い始めたときに疑問が生まれました。**なぜCodexでは、見慣れた「再生成」ボタンが目立たないのでしょうか。**

単に実装されていないのでしょうか。それとも、Agent製品には別の操作モデルが必要なのでしょうか。

私が現在もっとも納得している説明は、次のものです。

> AIが文章を生成するだけなら、Retryは別の回答を再生成する操作です。しかしAIが現実の環境に働きかけるようになると、Retryはタスクそのものの再実行になり得ます。

違いはボタン一つに見えますが、その背後にはまったく異なる二つのシステムがあります。

## Chatbot時代のRetryはなぜ簡単だったのか

従来のChatbotのやり取りは、おおよそ次の形です。

```text
用户问题 → 模型生成 → 回答 A
                   └→ Retry → 回答 B
```

最初の回答が良くなければ、同じ会話コンテキストを保ったまま、モデルにもう一度生成させられます。古い回答を捨てても、深刻な結果は通常起こりません。

Chatbotが変えたのは、多くの場合、画面上の文章だけだからです。

- ローカルファイルは変更していません。
- コマンドは実行していません。
- Gitブランチは変わっていません。
- 外部システムへデータを書き込んでいません。
- 取り消せないメールも送っていません。

同じ問題を別の言い方で答え直してもらうようなものです。最初の紙が気に入らなければ、丸めて新しく書けばよい。

したがってChatbotのRetryは、**入力を残し、出力を捨て、別の候補を生成すること**と理解できます。

## Codexが返すのは「回答」だけではない

CodexのようなAgentは、最終回答を書く前に、すでに多くの作業を終えている可能性があります。

Codex App Serverでは、Threadが複数のTurnを持ち、Turnが複数のItemを持ちます。ItemにはユーザーとAIのメッセージだけでなく、コマンド実行、ファイル変更、ツール呼び出しも含まれます。

つまり、画面に表示される最後の文章は、作業全体の要約にすぎないかもしれません。

```text
用户任务
   │
   ▼
理解仓库与上下文
   │
   ▼
读取文件 → 运行命令 → 调用工具 → 修改代码 → 执行测试
   │
   ▼
最终回复：“已经修改了这些内容……”
```

最終回答に満足できないとき、Retryは何をやり直すべきでしょうか。

要約だけを書き直すのでしょうか。推論からやり直すのでしょうか。すべてのコマンドを再実行するのでしょうか。それとも、すでに行われた変更を元に戻してから最初から実行するのでしょうか。

ここでRetryは単純な操作ではなくなります。

## 二回目の実行では、世界はすでに変わっている

Codexに次のように頼んだとします。

> このバグを修正し、テストを実行して、Pull Requestを作成してください。

一回目の実行で、Codexはすでに次の作業を終えているかもしれません。

1. コードを読み、原因を特定する。
2. 三つのファイルを変更する。
3. テストを実行する。
4. 新しいブランチを作る。
5. 変更をコミットしてpushする。
6. Pull Requestを作る。

この時点でRetryを押すと、何が起きるでしょうか。

- 文章だけを再生成すると、実際に行われた操作と説明が一致しなくなるかもしれません。
- 推論からやり直す場合、二回目はすでに変更されたリポジトリから始まります。
- ツールを再実行すると、重複コミット、競合するブランチ、二つ目のPRが作られるかもしれません。
- 先にロールバックしようとしても、外部操作は元に戻せない場合があります。

ファイルは復元できても、メールはすでに送信済みかもしれません。ローカルブランチは削除できても、外部システムの承認、メッセージ、取引を痕跡なく巻き戻せるとは限りません。

本当の問題は、単に「途中の情報が複雑」ということではありません。

> 一回のAgent実行が、現実の因果関係を作ります。二回目の実行は、一回目より前の世界から始まるわけではありません。

![文章は書き直せても、Agentが変更した現実の状態は消えない](/assets/blog/ai-agent-retry-state/text-vs-world-state-ja.png)

## AgentのRetryは少なくとも4種類の状態に関わる

システムの観点から見ると、一つのAgentタスクは四つの状態に影響します。

### 1. 会話状態

ユーザーの要求、過去のメッセージ、確認済みの制約、推論コンテキスト、ツールの結果が含まれます。

Retryするなら、どの履歴を残し、どの履歴を捨てるかを決めなければなりません。

### 2. 実行状態

ローカルファイル、Gitの作業ツリー、実行中のプロセス、テスト成果物、一時ファイル、インストール済み依存関係などです。

一回目の実行ですでに変わっている可能性があります。

### 3. 外部状態

GitHubのPull Request、データベースレコード、送信済みメッセージ、クラウドジョブ、フォーム、第三者システムなどです。

外部操作は完全にロールバックできないことが多く、もっとも危険です。

### 4. 権限と監査状態

ユーザーが承認した操作、呼び出されたツール、発生時刻、責任を追跡するための記録です。

Retryが一連の操作を黙って再生する場合、二回目は最初の承認の範囲内なのでしょうか。それとも新しい承認が必要でしょうか。監査ログでは二つの実行をどう関連付けるべきでしょうか。

四つの状態が重なると、Retryは単なるボタンではなく、ロールバック、重複防止、分岐管理の仕組みになります。

## Retryは何を意味し得るのか

Agent製品で「もう一度」は、少なくとも四つの異なる操作を意味します。

```text
再写一次    → 保留已完成的工作，只重写最终回答
重新规划    → 保留当前环境，但换一条推理和执行路线
重新执行    → 再运行一次失败的命令或工具
回到分叉点  → 保留原始历史，从某个状态创建一条新分支
```

これらのリスクはまったく異なりますが、一つのRetryボタンでは違いが隠れてしまいます。

Agent製品にも再試行は必要です。ただし、より明確な意味を持つ操作に分けるべきです。

- 最終回答だけを再生成する。
- 現在の状態を保ったまま修正を続ける。
- 失敗したステップだけを再実行する。
- ここから新しい分岐を作る。
- チェックポイントへ戻ってから再実行する。

ユーザーとシステムの双方が、何を繰り返すのか理解して初めて、Retryは信頼できる操作になります。

## Codexは一つのRetryではなくForkを提供する

公式資料から確認できるのは、Codexが作業をThread、Turn、Itemとして扱い、`fork`の仕組みを提供していることです。Codex CLIの`/fork`は現在のタスクを新しいタスクへ複製します。App Serverの`thread/fork`も、元の履歴を残したまま新しいThreadを作ります。

ForkとRetryは似ていますが、伝えている意味は異なります。

- Retry：「さっきの試行はなかったことにして、もう一度。」
- Fork：「さっきの試行は実際に起きた。ここから別の道を試す。」

文章を生成するだけのChatbotなら、前者で十分なことが多いでしょう。

しかし、ファイルを変更し、コマンドを実行し、外部ツールを呼ぶAgentでは、後者の方が正直で追跡しやすい設計です。

Forkは履歴が存在することを認めます。新しい試行は古い過程を上書きせず、独自の分岐と識別子を持ちます。

## 多くの場合、Retryよりフィードバックの方が有効

Agentの結果に満足できないとき、タスク全体をやり直したいとは限りません。多くの場合、どこが違うかを伝えたいのです。

- 調査結果は残し、結論だけをもっと直接的にする。
- APIは変えず、内部実装だけを修正する。
- テスト結果は残し、根本原因をもう一度調べる。
- 現在のコードを戻さず、別のUI案を試す。
- 実行を止め、まず現在の変更を見せる。

このフィードバックは正しい作業を再利用し、前回のずれを明確に伝えます。

従来のRetryはランダム性に期待し、次は良くなることを願う操作でした。良いAgent協働は同僚との仕事に近く、ずれを指摘し、正しい部分を残し、現在の状態から収束させます。

## 小さなボタンが示す製品の境界

Codexで従来のRetryが目立たないことは、必ずしも機能が失われたことを意味しません。

それは、AIが**回答する機械**から**行動するシステム**へ移る境界を示しています。

AIが文章だけを生成するなら、履歴は置き換えられることがあります。しかしAIが行動し始めると、履歴はシステム状態の一部になります。

重要なのは「もう一回答えられるか」だけではありません。

- 前回の実行が何をしたか確認できるか。
- 正しい成果を残し、間違った部分だけを直せるか。
- 明確な状態から別の道へ分岐できるか。
- ツールや外部操作の二重実行を防げるか。
- 各試行の因果関係を追跡できるか。

> Retryボタンが薄れていくことは、見慣れた操作が消えるだけではありません。AIが現実を変えられるようになったとき、「もう一度」は、どの状態から何を繰り返すのかという、より重い問いに答えなければなりません。
