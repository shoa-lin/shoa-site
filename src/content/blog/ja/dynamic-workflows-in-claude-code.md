先週、Claude Code で[ダイナミック・ワークフロー](https://code.claude.com/docs/en/workflows)をリリースしました。Claude は、目の前のタスクに合わせて独自の[ハーネス](https://code.claude.com/docs/en/glossary#agentic-harness)をその場で記述できるようになりました。

> **ハーネス**とは、プロンプトの組み立て、ツールの連携、コンテキスト管理、エラーからの回復など、AI モデルの周囲にある制御層を指します。Claude Code は **モデル + ハーネス**と捉えられます。この記事では、以降「ハーネス」と表記します。

Claude Code のデフォルト・ハーネスはコーディング向けに作られていますが、多くのタスクは突き詰めるとコーディングタスクに似ているため、ほかのさまざまな仕事にも役立ちます。一方で、最高の性能を引き出すには Claude Code の上に専用のハーネスが必要だった仕事もあります。[Research](https://support.claude.com/en/articles/11088861-using-research-on-claude)、[セキュリティ分析](https://support.claude.com/en/articles/11932705-automated-security-reviews-in-claude-code)、[agent teams](https://code.claude.com/docs/en/agent-teams)、[Code Review](https://code.claude.com/docs/en/code-review) などです。

ワークフローを使うと、Claude は Claude Code の上に、こうしたタスク専用のハーネスを動的に作成できます。作成したワークフローは保存、共有、再利用も可能です。

この記事では、ワークフローを使って得られた初期の経験と学びを紹介します。ベストプラクティスはまだ発展途上です。ダイナミック・ワークフローは多くのトークンを使うことが多く、複雑で価値の高いタスクに最も適しています。

## プロンプト例

技術的な詳細に入る前に、どのような可能性があるのかを示すプロンプトをいくつか紹介します。

「このテストは50回に1回くらい失敗する。再現するためのワークフローを構築して。競合状態について互いに競う仮説を立て、証拠によって1つの仮説だけが残るまで止めないで。」

「ワークフローを使って最近の50セッションを調べ、私が繰り返し行っている修正を見つけ、再発するものを `CLAUDE.md` のルールにして。」

「ワークフローを使って Slack の #incidents の過去6か月分を調べ、繰り返し発生しているのに誰もチケットを作成していない根本原因を見つけて。」

「私の事業計画を使い、投資家、顧客、競合他社それぞれの視点から別々のエージェントが徹底的に批判するワークフローを実行して。」

「ここに80件の履歴書がある。ワークフローを使ってバックエンド職への適合度で順位付けし、上位10件を再確認して。評価基準を決めるため、AskUserQuestion ツールで私に質問して。」

「この CLI ツールの名前が必要だ。ワークフローを使って幅広い案を出し、トーナメントを実行して上位3案を選んで。」

「ワークフローを使って、あらゆる場所にある User モデルを Account に改名して。」

「ワークフローを使って私のブログ記事の下書きを確認し、技術的な主張をすべてコードベースと照合して。誤った内容は一切公開したくない。」

## ダイナミック・ワークフローの仕組み

ダイナミック・ワークフローは、[サブエージェント](https://code.claude.com/docs/en/sub-agents)を生成して連携させるための特別な関数をいくつか含む JavaScript ファイルを実行します。

![ダイナミック・ワークフローがサブエージェントを生成して連携させる仕組み](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f1684f559cc83ff4b465b_image1.png)

データ処理には JSON、Math、Array などの標準 JavaScript オブジェクトも利用できます。

ダイナミック・ワークフローは、各エージェントが使うモデルと、サブエージェントを独自の worktree で実行するかどうかを選べます。これにより Claude は、各ステップに適した知能レベルと分離の度合いを選択できます。

ユーザーの操作やターミナルの終了によってワークフローが中断しても、セッションを再開すれば停止した場所から続行できます。

## ダイナミック・ワークフローが必要な理由

Claude Code のデフォルト・ハーネスは、同じコンテキストウィンドウの中でタスクの計画と実行を行う必要があります。これは多くのコーディングタスクで非常にうまく機能しますが、長時間実行される作業、大規模な並列処理、高度に構造化された作業、対抗的な検証を伴う作業では破綻することがあります。

Claude が1つのコンテキストウィンドウで複雑なタスクに取り組む時間が長くなるほど、次のような失敗モードが起こりやすくなります。

- **Agentic laziness**：複雑で複数部分からなるタスクを終える前に作業を止め、部分的な進捗だけで成功を宣言します。たとえば、セキュリティレビューの50件の指摘のうち35件だけを処理するようなケースです。
- **Self-preferential bias**：Claude は自分自身の結果を好む傾向があります。とりわけ、評価基準に照らして自分の結果を検証、評価するよう求められた場合に顕著です。
- **Goal drift**：多くのターンを重ねるうちに、元の目的への忠実さが徐々に失われます。特にコンパクションの後に起こりやすい問題です。要約のたびに情報が失われるため、エッジケースの要件や「X をしないこと」といった制約が消える可能性があります。

ワークフローは、それぞれ独自のコンテキストウィンドウと、焦点を絞って分離された目標を持つ Claude のサブエージェントを連携させることで、こうした失敗モードに対抗します。

## ダイナミック・ワークフローと静的ワークフロー

Claude Agent SDK や `claude -p` を使い、複数の Claude Code インスタンスを連携させる静的ワークフローをすでに構築したことがあるかもしれません。

静的ワークフローは、あらゆるエッジケースを事前に考慮しなければならないため、汎用的になりがちです。[Claude Opus 4.8](https://www.anthropic.com/news/claude-opus-4-8) とダイナミック・ワークフローを使えば、Claude は現在のユースケースに合わせた専用のハーネスを代わりに記述できます。

![静的ワークフローとダイナミック・ワークフローの比較](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f3a0e17e2844bed86f22a_image9.png)

## ダイナミック・ワークフローに役立つパターン

まずは Claude にダイナミック・ワークフローを作成するよう頼むだけでも構いません。意図を明示するには、トリガーワード `ultracode` も使えます。

よく使われるパターンを概念モデルとして理解しておくと、ワークフローが役立つ場面を見極め、プロンプトで Claude を導きやすくなります。

Claude は、次のようなパターンを利用し、組み合わせることができます。

![ダイナミック・ワークフローでよく使われるパターンの概要](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f16d86247e586b929a407_image10.png)

### 分類して実行する

分類エージェントを使ってタスクの種類を特定し、異なるエージェントや動作へ振り分けます。最後に分類エージェントを実行し、出力の処理方法を決めることもできます。

### 分散して統合する

タスクを小さなステップに分け、各ステップでエージェントを実行し、その結果を統合します。ステップ数が多い場合や、それぞれのステップで干渉や相互汚染を防ぐクリーンなコンテキストウィンドウが有効な場合に、特に役立ちます。統合ステップは障壁として機能します。分散したすべてのエージェントが完了するまで待ち、それぞれの構造化された出力を1つの結果にまとめます。

### 対抗的検証

出力を生成するエージェントごとに独立したエージェントを起動し、評価基準や明示的な条件に照らして、その出力を厳しく検証させます。

### 生成して絞り込む

多数のアイデアを生成し、評価基準や検証ステップで絞り込み、重複を除去して、検証済みの最も有力な候補だけを返します。

### トーナメント

作業を分割する代わりに、エージェントを競わせます。異なる方法で同じタスクに取り組む N 個のエージェントを生成し、審査エージェントまたはモデルで結果を2つずつ比較して、勝者が1つになるまで続けます。

### 完了するまでループする

作業量が不明なタスクでは、固定された試行回数を選ぶのではなく、新しい発見がなくなる、ログにエラーが残らなくなるといった停止条件を満たすまでエージェントを生成し続けます。

## ユースケース

Claude Code にダイナミック・ワークフローを依頼する場面を、柔軟に考えてみてください。ワークフローは、コーディングよりも非技術的な仕事でさらに役立つ場合があります。

### 移行とリファクタリング

[Bun](https://bun.com/) は、ワークフローを使って Zig から Rust に書き直されました。[Jarred の X thread](https://x.com/jarredsumner/status/2060050578026189172) では、チームがどのように取り組んだかが説明されています。

重要なのは、移行を呼び出し箇所、失敗しているテスト、モジュールなどの具体的な単位に分割することです。それぞれの修正について worktree 内でサブエージェントを起動し、別のエージェントに対抗的なレビューを行わせてから、変更をマージします。必要に応じてエージェントにリソースを大量に消費するコマンドを避けるよう伝えれば、より多くの作業をマシン上で並列実行できます。

### ディープリサーチ

Claude Code には、ダイナミック・ワークフローで構築されたディープリサーチ Skill、`/deep-research` が含まれています。Web 検索を分散して実行し、情報源を取得し、その主張を対抗的に検証して、引用付きのレポートに統合します。

同じパターンは Web 検索以外にも使えます。Claude は Slack のコンテキストから状況レポートをまとめたり、コードベースを深く調査して機能の仕組みを明らかにしたりできます。

### 詳細な検証

![詳細な検証のワークフロー](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f1721824a27cf13da87f4_image2.png)

すでにレポートがあり、事実に関する主張を1つずつ確認したい場合は、1つのエージェントに主張を抽出させ、別々のサブエージェントに各項目を調査させるワークフローを作成します。さらに別の検証役を使って、引用された各情報源に十分な信頼性があるかを評価できます。

### 並べ替え

![並べ替えのワークフロー](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f173ce727a972001584cc_image3.png)

Claude Code が評価できる定性的な尺度でリストを順位付けするとします。たとえば、サポートチケットをバグの深刻度順に並べるケースです。1つのプロンプトで1,000行を超えるデータを並べ替えようとすると、品質が低下し、有効なコンテキストの範囲も超えてしまいます。代わりにトーナメント、2項比較を行うエージェントのパイプライン、または並列のバケット別ランキングとその後のマージを使います。絶対評価よりも比較による判断のほうが信頼性が高く、比較ごとに独立したコンテキストウィンドウを使えます。

### メモリとルールの遵守

![メモリとルール遵守のワークフロー](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f17517076bb59050d90bb_image8.png)

ルールを `CLAUDE.md` に記述しても Claude が繰り返し見落とす場合は、ルールごとに1つの検証役を割り当てるワークフローを作成します。懐疑的なサブエージェントにルール自体を確認させれば、誤検知を減らせます。

逆方向にも使えます。最近のセッションやコードレビューのコメントから繰り返し行われている修正を抽出し、並列エージェントでクラスタリングします。候補となる各ルールが実際のミスを防げたかを対抗的に検証し、残ったものを `CLAUDE.md` に戻します。

### 根本原因の調査

デバッグでは、複数の独立した仮説を立て、それぞれを検証する方法が最も効果的です。1つのコンテキストウィンドウだけを使うと、self-preferential bias が起こりやすくなります。

ワークフローを使えば、ログ、ファイル、データなど、互いに重ならない証拠を別々のエージェントに割り当てることで、この問題を構造的に防げます。その後、各仮説を独立した検証役と反証役に検証、反証させます。

このパターンはコードに限りません。営業分析、データエンジニアリングの障害、あらゆる事後調査に適用できます。

### 大規模なトリアージ

![大規模なトリアージのワークフロー](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f1778dc00d34cca70819d_image6.png)

どのチームにも、サポートキュー、バグレポートなど、人間だけでは処理しきれないバックログがあります。

トリアージワークフローは各項目を分類し、すでに追跡されている項目と重複していないか確認し、修正を試みるか人にエスカレーションするかなど、適切なアクションを実行します。

ここでは隔離が有効なパターンです。信頼できない公開コンテンツを読むエージェントには高い権限を必要とする操作を許可せず、別のエージェントが得られた情報に基づいて行動します。

トリアージワークフローを [`/loop`](https://claude.com/blog/getting-started-with-loops) と組み合わせれば、Claude に継続的に実行させられます。

### 探索とセンス

デザインや命名のように最終的な選択がセンスに左右され、評価基準で表現できる場合、ワークフローは複数の解決策を探索するのに役立ちます。

Claude に多数の選択肢を探索させ、良い解決策の条件を記した評価基準をレビューエージェントに与えます。レビュー担当が条件を満たしたと判断した時点でタスクは終了します。トーナメントを使い、評価基準に沿って候補を順位付けまたは選定することもできます。

### 評価

worktrees で独立したエージェントを起動し、比較エージェントに評価基準と照らして出力を採点させることで、軽量な評価を実行できます。特定の基準に沿って Skill を評価し、改善する用途にも使えます。

### モデルと知能のルーティング

タスクに合わせて調整した分類エージェントを作成し、モデルを選択させます。事前調査やツール呼び出しによって、実際の実行にどの程度の知能が必要かが分かるタスクに役立ちます。

たとえば、「auth モジュールの仕組みを説明して」というタスクに適したモデルは、そのモジュールの規模と構造によって変わります。分類エージェントはモジュールを調べ、予想される複雑さに応じて Sonnet または Opus にタスクを振り分けられます。

## ダイナミック・ワークフローを使わない場面

ワークフローは新しい機能です。非常に大きな成果を生むこともありますが、すべてのタスクに必要なわけではなく、トークン使用量が大幅に増える可能性があります。

並列化、専門化、対抗的なチェックが連携コストに見合う場合に使ってください。一般的なコーディングタスクの多くは、5人のレビュー担当からなるパネルを必要としません。アーキテクチャのレベルで[マルチエージェントとシングルエージェントのシステム](https://claude.com/blog/building-multi-agent-systems-when-and-how-to-use-them)のどちらを選ぶかについても、同じ判断が当てはまります。

## ダイナミック・ワークフローを構築するヒント

### プロンプト

関連するワークフローパターンを明示した詳細なプロンプトほど、良い結果を得られます。

ワークフローは大規模なタスクだけに限定されません。1つの前提を短時間で対抗的にレビューするような「簡易ワークフロー」を依頼することもできます。

### `/goal` と `/loop` を組み合わせる

トリアージ、調査、検証などの反復可能なワークフローでは、[`/loop`](https://claude.com/blog/getting-started-with-loops) と [`/goal`](https://code.claude.com/docs/en/workflows) を組み合わせ、一定間隔で実行しながら、厳密な完了要件を設定します。

### トークン使用量の予算

ダイナミック・ワークフローには明示的なトークン予算を設定できます。「use 10k tokens」のようなプロンプトは、タスクに10kトークンの上限を設定します。

### ダイナミック・ワークフローの保存と共有

ワークフローメニューで `s` を押すと、ワークフローを保存できます。`~/.claude/workflows` にチェックインするか、Skill を通じて配布できます。

![ワークフローメニューからワークフローを保存する](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f17b1ca20533e666c867c_image4.png)

Skill を通じて共有するには、JavaScript のワークフローファイルを Skill フォルダに置き、`SKILL.md` から参照します。柔軟性を高めるには、逐語的に実行すべきスクリプトではなく、テンプレートとしてワークフローを扱うよう Claude に指示します。

![Skill を通じてワークフローを共有する](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f17cb835cf4f9fd5da921_image7.png)

## 発見のための新しい出発点

ワークフローは、Claude Code を拡張する新しく有用な方法です。Claude が仕事を支援できる新たな方法を発見するための出発点として活用してください。うまく使いこなす方法については、まだ多くの学びが残されています。

ハーネスに何を含めるべきかについては、Anthropic の[3つのハーネス設計パターン](https://claude.com/blog/harnessing-claudes-intelligence)を参照してください。
