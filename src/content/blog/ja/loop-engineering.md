**Loop Engineering（ループ・エンジニアリング）とは、自分でエージェントにプロンプトを与える代わりに、その役割を担うシステムを設計することです。** ここでいうループは再帰的な目標として捉えられます。目的を定義すると、AI が完了まで反復します。これは、今後コーディングエージェントと仕事をする方法になり得ると私は考えています。ただし、まだ初期段階であり、私は懐疑的でもあります。利用可能なトークン予算によって使用パターンが大きく変わるため、トークンコストには必ず[注意](https://x.com/weswinder/status/2063700289710964906)しなければなりません。そこで、この設計手法が何であり、何を意味するのかを整理してみます。

---

Peter Steinberger は最近、[こう述べました](https://x.com/steipete/status/2063697162748260627)。「もうコーディングエージェントにプロンプトを書くべきではない。エージェントへプロンプトを与えるループを設計すべきだ」。同じく Anthropic の Claude Code 責任者 Boris Cherny も、[こう述べています](https://x.com/rohanpaul_ai/status/2063289804708835412)。「私はもう Claude にプロンプトを与えていない。Claude にプロンプトを与え、何をすべきか判断するループを動かしている。私の仕事はループを書くことだ」

では、これは実際にはどういう意味なのでしょうか。

この 2 年ほど、コーディングエージェントから役立つ成果を得るには、良いプロンプトを書き、十分なコンテキストを与える必要がありました。何かを入力し、返ってきた内容を読み、次を入力する。エージェントは、作業の最初から最後まで一ターンずつ操作する道具でした。その時代は基本的に終わった、少なくともそう考える人が出てきています。

いまは、仕事を見つけ、割り当て、確認し、完了内容を記録し、次に何をするかを決める小さなシステムを構築します。自分でエージェントを動かす代わりに、そのシステムへ任せるのです。私はこれまでに、近い考え方を二つ紹介しました。一つは、単一エージェントが動く環境を設計する [エージェント・ハーネス・エンジニアリング（agent harness engineering）](https://addyosmani.com/blog/agent-harness-engineering/)。もう一つは、ソフトウェアを構築する仕組みである [ファクトリーモデル（factory model）](https://addyosmani.com/blog/factory-model/) です。ループ・エンジニアリングはハーネスの一段上に位置します。一定間隔で動き、補助エージェントを生成し、結果を次のサイクルへ渡すハーネスです。

意外だったのは、これがもはやツール単体の問題ではないことです。1 年前なら、ループを作るために大量の bash スクリプトを書き、長期にわたって保守し、ほかの誰も持っていない専用システムを抱える必要がありました。いまは、その構成要素が製品に組み込まれています。Steinberger のリストは Codex app の機能とほぼ完全に重なり、Claude Code にもほぼそのまま当てはまります。構造が同じだと気づけば、ツールについて議論するのをやめ、どの製品上でも動くループの設計へ集中できます。

## 5 つの構成要素と状態に関する補足

[ループ](https://x.com/reach_vb/status/2063713960495558940) には 5 つの要素と、状態を記憶する場所が必要です。詳しく対応づける前に、まず一覧にします。

1. **Automations（自動化）**：スケジュールに従って動き、作業の発見と振り分けを自律的に処理する。
2. **Worktrees（分離作業ツリー）**：並行作業する二つのエージェントが互いのファイルを壊さないようにする。
3. **Skills（スキル）**：エージェントが推測せずに済むよう、プロジェクト知識を記録する。
4. **Plugins / connectors（プラグインと接続機能）**：すでに使っているツールへエージェントを接続する。
5. **Sub-agents（補助エージェント）**：一方が解決策を提案し、もう一方が確認する。

そして 6 番目の要素がメモリです。Markdown ファイルでも Linear のボードでも、単一の会話より長く存続し、完了したことと次にすべきことを記録できれば何でも構いません。重要とは思えないほど単純に聞こえます。しかし、これは [long-running agents](https://addyosmani.com/blog/long-running-agents/) で説明したように、長時間稼働するすべてのエージェントが依存する手法です。モデルは実行の間にすべてを忘れるため、メモリはコンテキストではなくディスク上に置く必要があります。エージェントは忘れますが、リポジトリは忘れません。

現在は、どちらの製品にも 5 つの要素がすべて用意されています。

| 基本機能 | ループでの役割 | Codex app | Claude Code |
| --- | --- | --- | --- |
| **Automations（自動化）** | 定期的な発見 + 振り分け | [Automations タブ](https://developers.openai.com/codex/app/automations)：プロジェクト、プロンプト、実行間隔、環境を選択。結果は Triage inbox（振り分け受信箱）に入り、`/goal` が完了まで実行する動作を提供 | Scheduled tasks と cron、`/loop`、`/goal`、hooks、GitHub Actions |
| **Worktrees（分離作業ツリー）** | 並行する機能開発の分離 | スレッドごとに組み込みの worktree | `git worktree`、`--worktree`、補助エージェントの `isolation: worktree` |
| **Skills（スキル）** | プロジェクト知識の形式知化 | [Agent Skills](https://developers.openai.com/codex/skills)（`SKILL.md`）。`$name` で呼び出すか暗黙的に起動 | [Agent Skills](https://addyosmani.com/blog/agent-skills/)（`SKILL.md`） |
| **Plugins / connectors（プラグインと接続機能）** | ツールとの接続 | Connectors（MCP）と配布用プラグイン | MCP サーバーとプラグイン |
| **Sub-agents（補助エージェント）** | 解決策の作成と検証 | [Subagents](https://developers.openai.com/codex/subagents)。`.codex/agents/` に TOML で定義 | `.claude/agents/` の subagents（補助エージェント）と agent teams（エージェントチーム） |
| **状態** | 完了した作業の追跡 | Markdown、または connector 経由の Linear | Markdown（`AGENTS.md`、進捗ファイル）、または MCP 経由の Linear |

名称は少し異なりますが、機能は同じです。細部は重要です。ループが一つの仕組みとして維持されるか、気づかないうちにあちこちで破綻するかを左右するからです。

## Automations（自動化）：ループを定期実行する中枢

Automations（自動化機能）があることで、ループは一度きりの実行ではなく、本当の意味で繰り返し動く仕組みになります。Codex app では Automations タブから作成し、プロジェクト、プロンプト、実行間隔、ローカルの作業コピーとバックグラウンドの worktree のどちらで実行するかを選びます。何かを発見した実行は Triage inbox（振り分け受信箱）に入り、何も見つからなかった実行は自動的にアーカイブされます。OpenAI では、日々の issue の振り分け、CI 失敗の要約、コミットの概要作成、前週に混入したバグの調査といった定型作業に自動化機能を社内利用しています。自動化処理はスキルも呼び出せるため、再利用する動作を保守しやすい形にできます。誰も更新しない長大な指示をスケジュールへ貼るのではなく、`$skill-name` を起動すれば済みます。

Claude Code はスケジュール機能と hooks（フック）で同じ地点へ到達します。`/loop` でプロンプトやコマンドを一定間隔で実行し、cron タスクを設定し、エージェントのライフサイクルの特定時点でシェルコマンドを実行するフックを使えます。ノート PC を閉じた後も続けたいなら、処理全体を GitHub Actions へ移せます。考え方は同じです。自律的なタスクと実行間隔を定義し、すべてのシステムを自分で確認する代わりに、検出した内容が届くようにします。

この記事の核心にさらに近い、セッション内の基本機能もあります。`/loop` は設定した間隔で反復します。`/goal` は、あなたが記述した条件が実際に真になるまで作業を続けます。各ターンの後には独立した小さなモデルが条件を確認するため、コードを書いたエージェントが自分の作業を採点することはありません。「test/auth のすべてのテストが通り、lint が clean」のような条件を与えたら、その場を離れられます。Codex にも同じく `/goal` という基本機能があります。検証可能な停止条件が成立するまでターンをまたいで継続し、pause、resume、clear をサポートします。両方のツールに同じ基本機能があるという構図は、この記事全体に共通しています。

ここまでが作業を表面化する部分です。ループの残りの要素が、その作業に対応します。

## Worktrees（分離作業ツリー）：並行作業の衝突を防ぐ

複数のエージェントを動かした瞬間から、ファイルは衝突し始めます。二つのエージェントが同じファイルを編集すれば、二人のエンジニアが調整なしに同じ行を変更するのと同じ問題が起きます。git worktree は、この機械的な問題を解決します。リポジトリの履歴を共有しながら、独自のブランチを持つ独立した作業ディレクトリなので、一方のエージェントによる編集が他方の作業コピーに触れることはありません。

Codex は worktree 対応をアプリに直接組み込んでおり、複数のスレッドが同じリポジトリを扱っても、互いの作業コピーを変更しません。Claude Code も、`git worktree`、独自の作業コピーでセッションを開く `--worktree` オプション、各補助エージェントに新しい作業コピーを与え、終了後に削除する `isolation: worktree` 設定によって同じ分離を提供します。私は [the orchestration tax](https://addyosmani.com/blog/orchestration-tax/) で人間側の問題を書きました。worktree は機械的な衝突を取り除きますが、上限を決めるのは依然として**あなた**です。実際に同時運用できるエージェント数は、ツールではなく、あなたが処理できるレビュー量によって決まります。

## Skills（スキル）：プロジェクトを毎回説明しないために

スキルがあれば、セッションのたびに同じプロジェクトコンテキストを説明し直す必要がありません。どちらのツールも同じ形式を使います。指示とメタデータを記述した `SKILL.md` を含むフォルダーに、必要に応じてスクリプト、参考資料、素材を加えます。Codex は `$` または `/skills` で呼び出されたときにスキルを実行し、タスクがスキルの説明と一致した場合には自動的に起動します。だからこそ、気の利いた説明よりも、簡潔で文字どおりの説明のほうが優れています。Claude Code も同様に動作します。このパターンは [agent skills](https://addyosmani.com/blog/agent-skills/) で説明しました。

スキルは、意図を繰り返し説明するコストをなくす場所でもあります。[intent debt](https://addyosmani.com/blog/intent-debt/) で論じたように、エージェントはセッションごとに何も知らない状態から始まり、あなたの意図にある空白を自信に満ちた推測で埋めます。スキルは、その意図を外部化します。規約、ビルド手順、「以前の問題を踏まえて、この方法は使わない」といった情報を一度書けば、エージェントは実行のたびに読みます。スキルがなければ、ループはサイクルごとにプロジェクトをゼロから推測し直します。スキルがあれば、プロジェクト知識をサイクルをまたいで蓄積できます。

重要な区別が一つあります。スキルは作成形式であり、プラグインは配布方法です。リポジトリをまたいでスキルを共有したり、複数をまとめたりするときは、プラグインとしてパッケージ化します。これは Codex でも Claude Code でも同じです。

## Plugins / connectors（プラグインと接続機能）：ループを実際のツールへ接続する

ファイルシステムしか見えないループは、できることが非常に限られます。MCP を基盤とするコネクターにより、エージェントは課題管理ツールの読み取り、データベースの検索、検証環境の API 呼び出し、Slack へのメッセージ送信が可能になります。Codex と Claude Code はどちらも MCP をサポートするため、一方のために作ったコネクターは通常もう一方でも動きます。プラグインはコネクターとスキルをまとめるので、チームメイトは記憶を頼りに構成を再現せず、必要な一式をまとめて導入できます。

これは、「修正案はこれです」と答えるエージェントと、PR を開き、Linear のチケットを関連づけ、CI が成功したらチャンネルへ知らせるループとの違いです。コネクターによってループは、アクセスできた場合の行動を説明するだけでなく、実際の環境内で行動できます。

## Sub-agents（補助エージェント）：作る役割と確かめる役割を分ける

ループで最も有効な構造上の手法は、書くエージェントと確認するエージェントを分けることです。モデルは自分の仕事を採点するとき、評価が甘くなりすぎます。異なる指示、場合によっては異なるモデルを使う 2 番目のエージェントなら、最初のエージェントが自分を納得させて見逃した点を発見できます。

Codex は要求に応じて補助エージェントを生成し、並行実行した結果を一つの回答へ統合します。独自エージェントは `.codex/agents/` 内の TOML ファイルとして定義し、それぞれに名前、説明、指示、任意のモデルと推論強度を設定します。セキュリティのレビュアーには推論強度の高い強力なモデルを使い、探索役には高速な読み取り専用モデルを使う、といった構成が可能です。Claude Code は `.claude/agents/` の subagents（補助エージェント）と、作業を受け渡す agent teams（エージェントチーム）で同じことを行います。両方のツールでよく使われる分担は、探索、実装、仕様に照らした検証をそれぞれ別のエージェントに任せる形です。

私はこの考えを、[the code agent orchestra](https://addyosmani.com/blog/code-agent-orchestra/) と [agentic code review](https://addyosmani.com/blog/agentic-code-review/) の二度にわたって論じてきました。ループ内では、あなたが見ていない間にも処理が動くため、特に重要です。本当に信頼できる検証役がいれば、無人実行の結果をより信頼できるものにできます。補助エージェントはそれぞれ独自にモデル処理とツール操作を行うため、より多くのトークンを消費します。セカンドオピニオンに費用をかける価値がある箇所へ使ってください。Claude Code の `/goal` も同じ構造です。作業したモデルではなく新しいモデルがループの完了を判定します。作成役と検証役の分離を、停止条件そのものへ適用しているのです。

## ループの実際の姿

これらの要素を組み合わせると、一つのスレッドが小さな操作盤になります。私が繰り返し使っているパターンを紹介します。

自動化処理が毎朝リポジトリに対して実行されます。そのプロンプトは振り分け用スキルを呼び出し、前日の CI 失敗、未解決の課題、直近のコミットを読み、結果を Markdown ファイルまたは Linear のボードに書き込みます。対応する価値がある検出事項ごとに、スレッドは分離した worktree を開き、一つの補助エージェントに修正案を作らせ、別の補助エージェントにプロジェクト用スキルと既存テストに照らしてその案をレビューさせます。

コネクターによって、ループは PR を開き、チケットを更新できます。ループが処理できないものは、すべて私の振り分け用受信箱に入ります。状態ファイルはシステムの背骨です。何を試し、何が通り、何が未完了かを記憶するので、翌朝の実行は今日止まった場所から続けられます。

実際に自分が何をしたか見てください。プロセスを一度設計しただけで、個々のステップにプロンプトを書いてはいません。これが Steinberger の主張を具体化した姿です。構成要素が同じなので、Codex でも Claude Code でも同じループになります。

## ループが今も代行できないこと

ループは仕事の形を変えますが、あなたを仕事から取り除くわけではありません。ループが改善されるほど、次の三つの問題は簡単になるのではなく、むしろ鮮明になります。

**検証は今もあなたの責任です。** 無人で動くループは、無人のまま誤りを生む可能性もあります。検証を担当する補助エージェントと作成役を分けるのは、ループが主張する「完了」により大きな重みを持たせるためです。それでも完了は主張であって、証明ではありません。[code review in the age of AI](https://addyosmani.com/blog/code-review-ai/) で使った言葉を繰り返します。あなたの仕事は、動作を自分で確認したコードを出荷することです。

**放置すれば、あなたの理解は今も劣化します。** ループが自分の書いていないコードを速く出荷するほど、実際に存在するシステムと、自分が理解しているシステムの差は広がります。これが [理解の負債（comprehension debt）](https://addyosmani.com/blog/comprehension-debt/) です。コードが増える一方で理解が追いつかず、その差が負債として蓄積します。ループの生成物を注意深く読まなければ、円滑なループはこの負債をさらに速く増やすだけです。

**心地よい姿勢には危険があります。** ループが自律的に動くと、自分で判断することをやめ、返ってきたものをそのまま受け入れたくなります。私はこれを [認知的な判断放棄（cognitive surrender）](https://addyosmani.com/blog/cognitive-surrender/) と呼びます。思考と判断をシステムへ明け渡す状態です。判断力を組み込んでループを設計すれば対策になりますが、考えることを避けるために設計すれば問題を加速させます。動作は同じように見えても、結果は正反対です。

## ループを構築し、エンジニアであり続ける

これは、私たちの仕事がどう変化していくかを示す予告編だと思います。一方で、自分でコードをレビューするのをやめたり、修正を自動ループだけに任せたりすれば、製品品質は低下するでしょう。おそらく悪循環に陥り、穴を深く掘り続けることになります。

だからループを作って構いません。ただし、エージェントへ直接プロンプトを与える方法も今なお有効です。目標は、適切なバランスを見つけることです。

ループは使う人によって、まったく異なる結果を生みます。二人が同じループを作っても、結果は正反対になり得ます。一人は、深く理解している仕事を速く進めるために使います。もう一人は、仕事を理解しないために使います。ループには、その違いがわかりません。あなたにはわかります。

だからループの設計はプロンプト・エンジニアリングより簡単なのではなく、難しいのです。Cherny の主張は、仕事が簡単になったという意味ではありません。力を加える場所が移ったのです。

ループを作ってください。ただし、開始ボタンを押すだけの人ではなく、エンジニアであり続ける人として設計してください。
