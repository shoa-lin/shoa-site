## はじめに

エンジニアの間では、キャッシュがシステムを左右すると言われることがあります。長時間動作するエージェントにも、同じことが当てはまります。

Claude Code のような製品が経済的に成立するのは、プロンプトキャッシュによって後続のリクエストが過去のターンの計算を再利用できるからです。会話が長くなるほど、この再利用はレイテンシーとコストの削減に大きく効いてきます。

## Claude Code のキャッシュアーキテクチャ

Claude Code はプロンプトキャッシュを中心に設計されています。プロンプトキャッシュのヒット率が高ければ運用コストが下がり、サブスクリプションでより余裕のあるレート制限を提供できます。チームはこの指標を継続的に監視し、深刻な低下をインシデントとして扱うこともあります。

以下では、大規模にプロンプトキャッシュを最適化する中で得られた本番運用の知見を整理します。

![プロンプトキャッシュのアーキテクチャ図](https://pbs.twimg.com/media/HBipHa1boAAXD_A?format=jpg&name=large)

## プロンプトキャッシュの仕組み

### プレフィックスマッチング

プロンプトキャッシュは**プレフィックスマッチング**によって機能します。リクエストの先頭からキャッシュブレークポイントまでの内容が変わらなければ、API はそのプレフィックスに対する計算を再利用できます。

したがって、並び順が重要です。より多くのリクエストが同じ先頭部分を共有するほど、再利用できるキャッシュも増えます。

### Claude Code のキャッシュに適した順序

Claude Code は、**安定した内容を先に、動的な内容を後に置く**というキャッシュに適した構成を採用しています。

リクエストは、おおむね次の順序で構成されます。

1. **安定したシステムプロンプトとツール**（広く共有される）
2. **プロジェクトコンテキスト**（同じプロジェクト内で共有される）
3. **セッションコンテキスト**（同じセッション内で共有される）
4. **会話メッセージ**

この配置により、リクエストやセッション間で再利用可能なプレフィックスを共有しやすくなります。

### なぜこの順序は壊れやすいのか

一見無害に見える変更でも、プレフィックスは崩れます。たとえば次のような変更です。

- 正確なタイムスタンプを安定したシステムプロンプトに入れる
- ツール定義を非決定的な順序で出力する
- エージェントツールが呼び出せるエージェントの範囲など、ツールのパラメーターを変更する

## キャッシュを有効に保つ

### 古くなった情報を更新する

プロンプト内の情報には、自然に古くなるものがあります。日付が変わる、ファイルが編集される、実行時の状態が更新される、といった場合です。

前方のシステムプロンプトを書き換えるのは整然とした方法に見えますが、プレフィックスが変わるため、それ以降の内容はすべてキャッシュミスになります。

Claude Code では、更新情報を後続のメッセージとして送ります。たとえば、次のユーザーメッセージやツール結果に `<system-reminder>` を含め、現在は水曜日だと伝えます。こうすれば、古いプレフィックスを再利用しながら、モデルには最新情報を渡せます。

## モデル切り替えの落とし穴

### キャッシュはモデルごとに分かれる

プロンプトキャッシュはモデル固有であるため、コスト計算は一見したほど単純ではありません。

Opus との会話にすでに 100k トークンのキャッシュがある場合、簡単な質問をもう一度 Opus に答えさせるほうが、Haiku に切り替えるより安いことがあります。Haiku では、その会話履歴に対するプロンプトキャッシュを新たに構築する必要があるからです。

### モデル間の引き継ぎにはサブエージェントを使う

別のモデルが適している場合、Claude Code は既存の会話のモデルを変更するのではなく、**サブエージェント**への引き継ぎを優先します。Opus が別のモデル向けに、簡潔なタスク説明を作成できます。

Explore エージェントは代表的な例です。親会話に蓄積されたモデル固有のキャッシュを捨てることなく、Haiku を利用できます。

## ツール変更のコストが高い理由

会話の途中でツールセットを変更することも、プロンプトキャッシュの再利用を損なう典型的な原因です。

その時点で必要なツールだけを公開するほうが効率的に見えるかもしれません。しかし、ツール定義はキャッシュされるプレフィックスの一部です。ツールを追加または削除すると、それ以降の会話プレフィックスが無効になります。

### Plan Mode：ツールを変えずに状態を表現する

Plan Mode は、Claude Code がこの制約を踏まえて機能を設計している例です。

素朴に実装するなら、ユーザーが Plan Mode に入った時点で通常のツールセットを読み取り専用ツールに置き換えるでしょう。しかし、このツールスキーマの変更はキャッシュを壊します。

そこで Claude Code はツールを安定したまま維持し、`EnterPlanMode` と `ExitPlanMode` も通常のツールとして含めます。その後に追加されるシステムメッセージが、現在は Plan Mode であること、コードベースを調査してファイルを編集しないこと、計画が完成したら `ExitPlanMode` を呼ぶことをエージェントに伝えます。ツール定義自体は変わりません。

`EnterPlanMode` 自体がツールなので、問題に詳細な計画が必要だとモデルが判断した場合にも、キャッシュされたプレフィックスを無効にせず自ら Plan Mode に入れます。

### Tool Search：ツールを削除せず遅延読み込みする

Tool Search にも同じ原則が当てはまります。Claude Code では数十個の MCP ツールが利用可能な場合がありますが、毎回すべての完全なツールスキーマを送るのは高コストです。一方、会話の途中でツールを削除しても、やはりキャッシュは壊れます。

解決策が `defer_loading` です。Claude Code は、`defer_loading: true` を付けた安定した軽量なツールスタブを送ります。必要になった時点で、モデルが `ToolSearch` を使って完全なツールスキーマを読み込みます。同じスタブが同じ順序で保たれるため、プレフィックスも維持されます。

API でも `ToolSearch` が提供されており、アプリケーションから同じパターンを利用できます。

## コンパクションとキャッシュ

![コンパクションとキャッシュの図](https://pbs.twimg.com/media/HBitEdRbUAMVSnM?format=jpg&name=large)

コンパクションは、会話がコンテキストウィンドウの上限に近づいたときに行われます。システムが要約を生成し、その小さな表現を使って会話を継続します。

この処理には、プロンプトキャッシュに関するいくつかの境界ケースがあります。

### 問題

要約を作るには、モデルが会話履歴を参照する必要があります。単純な実装では、異なるシステムプロンプトを使い、ツールを含めない別リクエストを送るかもしれません。するとメイン会話のプレフィックスと一致しなくなり、入力トークン全体を通常料金で処理し直す必要があります。

### 解決策：キャッシュを維持する分岐

Claude Code はコンパクションを、キャッシュを維持する分岐として扱います。コンパクションのリクエストは、親リクエストと同じシステムプロンプト、ユーザーおよびシステムのコンテキスト、ツール定義、会話履歴を使用し、最後にコンパクションの指示を新しいユーザーメッセージとして追加します。

入力プレフィックスの処理という観点では、キャッシュされたプレフィックスを再利用できるため、主に再処理が必要なのは新しく追加したコンパクション指示の入力です。ただし、要約の出力は引き続き生成する必要があり、その計算と出力トークンには料金が発生します。

また、コンパクション用の余裕も必要です。コンテキストウィンドウには、追加するコンパクション指示とモデルが生成する要約を収めるだけの空き容量を残しておかなければなりません。

## 5 つの教訓

コンパクションの扱いは難しいものの、ここから得られる教訓はプロンプトキャッシュを利用するあらゆるエージェントに当てはまります。

<div class="info-box">

**1. プロンプトキャッシュはプレフィックスマッチングで決まる**

プレフィックス内の変更は、それ以降をすべて無効にします。最初から安定した順序を前提にリクエストを設計してください。

</div>

<div class="tip-box">

**2. 更新情報はメッセージとして送る**

日付、実行時の状態、モードが変わったときは、前方のシステムプロンプトを書き換えず、メッセージを追加します。

</div>

<div class="warning-box">

**3. 会話の途中でモデルやツールを変更しない**

モデル変更には引き継ぎ、状態遷移にはツール、大規模なツールカタログには遅延読み込みを使います。

</div>

<div class="info-box">

**4. プロンプトキャッシュのヒット率を可用性と同じように監視する**

数パーセントの変化でもコストとレイテンシーに大きく影響するため、キャッシュの回帰には運用アラートを設定する価値があります。

</div>

<div class="tip-box">

**5. 分岐した処理でも親のプレフィックスを維持する**

コンパクション、要約、その他の副次的な処理では、可能な限り親リクエストのキャッシュを維持できる形を再利用します。

</div>

## まとめ

Claude Code は当初からプロンプトキャッシュを中心に構築されてきました。実践上の教訓は、すべてのエージェントが一つの決まった配置を模倣すべきだということではありません。キャッシュの安定性を、アーキテクチャ上の重要な制約として扱うべきだということです。
