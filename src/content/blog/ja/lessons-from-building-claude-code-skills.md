スキルは、Claude Code で最もよく使われる拡張ポイントの一つになりました。柔軟で、作りやすく、配布も簡単です。

その柔軟さゆえに、何が最も効果的なのかを見極めるのは簡単ではありません。どのようなスキルを作る価値があるのでしょうか。良いスキルを書く秘訣は何でしょうか。いつ他の人と共有すべきでしょうか。

Anthropic では Claude Code のスキルを幅広く活用しており、現在も数百のスキルが日常的に使われています。ここでは、スキルによって開発を加速する中で得た教訓を紹介します。

---

## スキルとは何か

スキルを初めて使う場合は、まず[ドキュメントを読む](https://code.claude.com/docs/en/skills)か、最新の [Agent Skills に関する Skilljar コース](https://anthropic.skilljar.com/introduction-to-agent-skills)を受講してください。この記事は、スキルについてある程度知っていることを前提としています。

よくある誤解は、スキルが「単なる Markdown ファイル」だというものです。興味深いのは、スキルがただのテキストファイルではない点です。スクリプト、アセット、データなど、エージェントが発見、探索、操作できるさまざまなリソースを含むフォルダです。

Claude Code のスキルには、動的フックを含む[多様な設定オプション](https://code.claude.com/docs/en/skills#frontmatter-reference)もあります。

特に興味深いスキルの中には、こうした設定オプションとフォルダ構造を創造的に活用しているものがあります。

---

## スキルの種類

社内のスキルを分類してみると、いくつかのパターンに集約されることが分かりました。優れたスキルは一つのカテゴリにきれいに収まり、分かりにくいスキルは複数のカテゴリにまたがる傾向があります。これは決定版の一覧ではありませんが、組織内で何が不足しているかを考えるうえで役立ちます。

![一般的なスキルのカテゴリを示す図](https://pbs.twimg.com/media/HDlvMmubEAIzF-N?format=jpg&name=small)

---

### 1. ライブラリと API のリファレンス

ライブラリ、CLI、SDK を正しく使う方法を説明するスキルです。社内ライブラリのほか、Claude Code がときどき扱いに苦労する一般的なツールも対象になります。この種のスキルには、参考になるコードスニペットと、Claude がスクリプトを書く際に避けるべき「よくある落とし穴」の一覧が含まれることがよくあります。

**例：**

- **billing-lib** - 社内の課金ライブラリについて、エッジケース、落とし穴、その他の失敗しやすい詳細をまとめる
- **internal-platform-cli** - 社内 CLI ラッパーの全サブコマンドと、それぞれをいつ使うべきかを例付きで説明する
- **frontend-design** - Claude がデザインシステムをより適切に適用できるようにする

---

### 2. プロダクト検証

コードが正しく動作することをテスト、検証する方法を説明するスキルです。Playwright や tmux などの外部ツールと組み合わせることがよくあります。

検証用スキルは、Claude の出力が正しいことを確かめるうえで非常に有用です。エンジニアが一週間かけて完成度を高めるだけの価値がある場合もあります。

Claude が何をテストしたのかを正確に確認できるよう動画を記録したり、各ステップで状態に対するプログラム的なアサーションを必須にしたりする方法を検討してください。こうした機能は、多くの場合スキル内のスクリプトで実装されます。

**例：**

- **signup-flow-driver** - ヘッドレスブラウザでサインアップ -> メール認証 -> オンボーディングを実行し、各ステップの状態をフックで検証する
- **checkout-verifier** - Stripe のテストカードでチェックアウト UI を操作し、請求書が正しい状態に到達したことを検証する
- **tmux-cli-driver** - ワークフローに TTY が必要な対話型 CLI をテストする

---

### 3. データ取得と分析

データ基盤や監視スタックに接続するスキルです。認証済みデータを取得するライブラリ、特定のダッシュボード ID、一般的なワークフローやクエリの手順などを含められます。

**例：**

- **funnel-query** - サインアップ -> アクティベーション -> 有料化を分析するために結合すべきイベントと、正規の `user_id` を持つテーブルを示す
- **cohort-compare** - 二つのコホート間でリテンションやコンバージョンを比較し、統計的に有意な差を示し、セグメント定義へリンクする
- **grafana** - データソース UID、クラスタ名、問題からダッシュボードを引くための対応表を提供する

---

### 4. ビジネスプロセスとチーム自動化

繰り返し発生するワークフローを一つのコマンドにまとめるスキルです。手順そのものは単純でも、他のスキルや MCP に依存する場合があります。過去の結果をログファイルに保存すると、モデルが一貫性を保ち、以前の実行を振り返りやすくなります。

**例：**

- **standup-post** - チケット管理システム、GitHub の活動、過去の Slack 投稿を集約し、差分だけを含む定型のスタンドアップ投稿を作る
- **create-<ticket-system>-ticket** - 有効な列挙値や必須フィールドを持つスキーマを適用し、作成後にレビュアーへの通知や Slack でのチケット共有などのワークフローを実行する
- **weekly-recap** - マージ済み PR、クローズ済みチケット、デプロイを定型の週間まとめ投稿に変換する

---

### 5. コードのスキャフォールディングとテンプレート

コードベース内の特定機能に必要なフレームワークのボイラープレートを生成するスキルです。自然言語によるガイドと組み合わせ可能なスクリプトを併用できます。スキャフォールディング要件をコードだけでは完全に表現できない場合に、特に役立ちます。

**例：**

- **new-<framework>-workflow** - 独自のアノテーションを付けた新しいサービス、ワークフロー、ハンドラを作成する
- **new-migration** - マイグレーション用テンプレートと一般的な落とし穴を提供する
- **create-app** - 認証、ログ、デプロイ設定まで接続済みの社内アプリを作成する

---

### 6. コード品質とレビュー

組織内のコード品質を担保し、コードレビューを支援するスキルです。堅牢性を高めるために決定論的なスクリプトやツールを含めたり、フックや GitHub Actions を通じて自動実行したりできます。

**例：**

- **adversarial-review** - 先入観のないサブエージェントを起動して成果物を批評させ、修正とレビューを繰り返し、指摘が細かなものだけになるまで改善する
- **code-style** - Claude がデフォルトではうまく守れないコードスタイルを強制する
- **testing-practices** - テストの書き方と、何をテストすべきかを説明する

---

### 7. CI/CD とデプロイ

コードの取得、プッシュ、デプロイを支援するスキルです。データ収集のために他のスキルを呼び出す場合もあります。

**例：**

- **babysit-pr** - PR を監視 -> 不安定な CI を再実行 -> マージコンフリクトを解消 -> 自動マージを有効化する
- **deploy-<service>** - ビルド -> スモークテスト -> エラー率を比較しながらトラフィックを段階的に移行 -> 回帰があれば自動ロールバックする
- **cherry-pick-prod** - 分離されたワークツリーを作成 -> cherry-pick -> コンフリクトを解消 -> 正しいテンプレートで PR を作成する

---

### 8. 運用手順

Slack thread、アラート、エラーシグネチャなどの症状を入力として、複数のツールを使った調査を進め、構造化されたレポートを作成するスキルです。

**例：**

- **<service>-debugging** - 高トラフィックなサービスについて、症状 -> ツール -> クエリパターンの対応を示す
- **oncall-runner** - アラートを取得 -> よくある原因を確認 -> 調査結果を整形する
- **log-correlator** - request ID を受け取り、そのリクエストを処理した可能性のある全システムから一致するログを取得する

---

### 9. インフラストラクチャ運用

日常的な保守や運用手順を実行するスキルです。破壊的な操作を含むものもあり、強いガードレールが有効です。重要な運用時に、エンジニアがベストプラクティスに従いやすくなります。

**例：**

- **<resource>-orphans** - 孤立した Pod やボリュームを検出 -> Slack に投稿 -> 安定化待機期間が過ぎるまで待機 -> ユーザーに確認 -> 連鎖的なクリーンアップを実行する
- **dependency-management** - 組織の依存関係承認ワークフローを実行する
- **cost-investigation** - 関連する buckets とクエリパターンを使い、ストレージ費用や egress 費用が急増した理由を調査する

---

## スキルを作るためのヒント

![スキルを作るためのヒントをまとめた図](https://pbs.twimg.com/media/HDoKg58bEAAL1bw?format=jpg&name=small)

作るスキルを決めたら、どのように書けばよいのでしょうか。ここでは、私たちの実践で特に効果のあった方法とテクニックを紹介します。

最近、Claude Code でスキルをより簡単に作れるようにする [Skill Creator](https://claude.com/blog/improving-skill-creator-test-measure-and-refine-agent-skills) もリリースしました。

---

### 当たり前のことを書かない

Claude Code はすでにコードベースについて多くを理解しており、Claude 自身もプログラミングに関する幅広い知識と、多くのデフォルトの見解を持っています。スキルが主に知識を提供するものなら、Claude を普段の思考パターンの外へ導く情報に集中してください。

[frontend design skill](https://github.com/anthropics/skills/blob/main/skills/frontend-design/SKILL.md) はその好例です。Anthropic のエンジニアは顧客との反復を通じてこのスキルを作り、Claude のデザインセンスを高め、Inter フォントや紫のグラデーションといった見慣れたデフォルトを避けられるようにしました。

---

### 「よくある落とし穴」セクションを作る

![「よくある落とし穴」セクションの例](https://pbs.twimg.com/media/HDlwEG1bEAUdmcV?format=jpg&name=small)

どのスキルでも、最も情報密度が高いのは「よくある落とし穴」セクションであることが少なくありません。Claude がそのスキルを使う際につまずきやすい点を集めて作り、新しい落とし穴が見つかるたびに更新してください。

---

### ファイルシステムと段階的開示を使う

![段階的開示に使うスキルのフォルダ構造](https://pbs.twimg.com/media/HDlwhSjbEAIJSc9?format=jpg&name=small)

スキルは単なる Markdown ファイルではなく、フォルダです。ファイルシステム全体を、コンテキストエンジニアリングと段階的開示の仕組みとして扱ってください。スキル内にどのファイルがあるかを Claude に伝えれば、必要になった時点で読み込めます。

最も単純な段階的開示は、別の Markdown ファイルを参照させる方法です。たとえば、詳細な関数シグネチャや使用例は `references/api.md` に置けます。

最終的な出力が Markdown ドキュメントなら、Claude がコピーして使えるテンプレートを `assets/` に含めることもできます。

リファレンス、スクリプト、例などのリソースをフォルダに分けることで、Claude はより効率的に作業できます。

---

### Claude の進め方を固定しすぎない

Claude は通常、指示に忠実に従おうとします。スキルは幅広い場面で再利用されるため、指示を細かく限定しすぎると壊れやすくなります。必要な情報を与えつつ、状況に合わせて適応できる余地を残してください。

![柔軟なガイドと制限しすぎた指示の比較例](https://pbs.twimg.com/media/HDlwurvbEAM5ZNu?format=jpg&name=small)

---

### セットアップをよく考える

![スキルのセットアップ設定例](https://pbs.twimg.com/media/HDlw1mYbEAY-Bul?format=jpg&name=small)

スキルによっては、セットアップ時にユーザーからコンテキストを受け取る必要があります。たとえば、スタンドアップを Slack に投稿するスキルなら、どの Slack チャンネルに投稿するかを Claude が質問しなければならない場合があります。

セットアップ情報はスキルのディレクトリ内の `config.json` に保存するのがよいパターンです。設定がなければ、エージェントがユーザーに質問できます。

構造化された選択式の質問を提示するには、AskUserQuestion ツールを使うよう Claude に指示します。

---

### 説明欄はモデルのためにある

Claude Code はセッションの開始時に、利用可能なすべてのスキルと各スキルの説明欄の一覧を作ります。Claude はその一覧を見て、「このリクエストに使えるスキルはあるか」と判断します。したがって説明欄は要約ではなく、モデルがそのスキルをいつ起動すべきかを記述するものです。

![モデルが起動条件を判断できるように書かれたスキルの説明欄の例](https://pbs.twimg.com/media/HDlw5ULbEAQOqtJ?format=jpg&name=small)

---

### メモリとデータの保存

![スキルにメモリとデータを保存する例](https://pbs.twimg.com/media/HDoImh1bEAU-mMI?format=jpg&name=small)

スキルには、データを保存することでメモリを持たせられます。追記専用のテキストログや JSON ファイルのような単純な方法から、SQLite データベースのような複雑な方法まであります。

たとえば、`standup-post` スキルは、これまでに書いたすべての投稿を `standups.log` に保存できます。次回の実行時に Claude が履歴を読めば、昨日から何が変わったかを特定できます。

スキルのディレクトリ内に置いたデータは、スキルのアップグレード時に削除される可能性があります。永続化するデータは安定した場所に保存してください。現時点では、`${CLAUDE_PLUGIN_DATA}` がプラグインごとに一つの安定したフォルダを提供します。

---

### スクリプトを保存し、コードを生成する

Claude に渡せるものの中で、コードは特に強力です。スクリプトやライブラリがあれば、Claude はボイラープレートを毎回作り直すのではなく、既存の機能を組み合わせ、次に何をするかを判断することにターンを使えます。

たとえば、データサイエンス用スキルに、イベントソースからデータを取得する関数を含められます。Claude に一連のヘルパー関数を渡せば、それらを組み合わせてより複雑な分析を作れます。

![スキル内にあるヘルパー関数の例](https://pbs.twimg.com/media/HDlxbtkbkAAOse7?format=jpg&name=small)

Claude はその場でスクリプトを生成し、それらの関数を組み合わせて「火曜日に何が起きたか」といったプロンプトに答えられます。

![Claude がヘルパー関数から生成したスクリプトの例](https://pbs.twimg.com/media/HDlxfEIb0AA2E7l?format=jpg&name=small)

---

### オンデマンドフック

スキルには、そのスキルが呼び出されたときだけ有効になり、セッション中は動作し続けるフックを定義できます。常時動かすと邪魔になる一方、特定の状況では価値の高い、強い保護策に適しています。

例：

- **/careful** - Bash に対する PreToolUse マッチャーを使い、`rm -rf`、`DROP TABLE`、強制プッシュ、`kubectl delete` をブロックします。本番環境に触れるときだけ有効にします。常時オンでは煩わしすぎます。
- **/freeze** - 指定したディレクトリ以外への Edit/Write をすべてブロックします。デバッグ時に、無関係なコードをうっかり「修正」せず、ログだけを追加したい場合に役立ちます。

## スキルを配布する

スキルの大きな利点の一つは、チームの他のメンバーと共有できることです。

一般的な配布方法は二つあります。

- スキルをリポジトリの `./.claude/skills` 配下にチェックインする
- プラグインと Claude Code のプラグイン・マーケットプレイスを作り、ユーザーがインストールできるようにする。詳しくは[プラグイン・マーケットプレイスのドキュメント](https://code.claude.com/docs/en/plugin-marketplaces)を参照してください

比較的少数のリポジトリを扱う小規模なチームなら、各リポジトリにスキルをチェックインする方法が適しています。ただし、チェックインされたスキルは一つずつモデルのコンテキストを少し増やします。規模が大きくなったら、社内プラグイン・マーケットプレイスを使うことで、組織としてスキルを配布しつつ、各チームがインストールするものを選べます。

---

### マーケットプレイスを管理する

どのスキルをマーケットプレイスに入れるか、チームはどう判断すべきでしょうか。また、メンバーはどのように提出すればよいのでしょうか。

Anthropic では、中央のチームがすべてを決めているわけではありません。有用なスキルは自然に生まれます。オーナーは GitHub の sandbox フォルダにスキルをアップロードし、Slack などで共有して他の人に試してもらえます。

十分に利用が広がったとオーナーが判断したら、マーケットプレイスに移す PR を作成できます。

質の低いスキルや重複するスキルは簡単に作れてしまうため、公開前には何らかのキュレーションが重要です。

---

### スキルを組み合わせる

スキル同士が依存することもあります。file-upload スキルがファイルをアップロードし、CSV-generation スキルが CSV を作ってから upload スキルを呼び出す、といった構成です。マーケットプレイスとスキルには、まだ標準の依存関係管理がありません。それでも、あるスキルから別のスキルを名前で参照すれば、インストール済みである限りモデルが呼び出します。

---

### スキルを計測する

スキルの利用状況を把握するため、私たちは PreToolUse フックを使って社内でのスキル使用を記録しています。[サンプルコード](https://gist.github.com/ThariqS/24defad423d701746e23dc19aace4de5)に実装方法が示されています。これにより、どのスキルがよく使われているか、どのスキルの起動頻度が想定より低いかが分かります。

---

## まとめ

スキルはエージェントのための強力で柔軟なツールですが、この分野はまだ若く、誰もが効果的な使い方を学んでいる途中です。

ここで紹介した教訓は、決定版のガイドではなく、役に立つテクニックの詰め合わせとして捉えてください。スキルを理解する最良の方法は、まず作り、試し、何が機能するかを観察することです。私たちのスキルの多くも、最初は数行の説明と一つの落とし穴だけでした。Claude が新たなエッジケースに出会うたびに学びを追加し、少しずつ改善されてきました。

この記事が役立てば幸いです。質問があれば、ぜひお知らせください。
