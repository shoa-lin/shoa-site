很多 Agent 文章把两件事揉在一起讲：**怎么让模型完成任务**，以及**怎么把系统写得可靠**。结果是名词记住了一串，真正开始设计时，却不知道该先解决哪一个问题。

这篇先只回答第一件事：一个 Agent 如何判断、行动、协作，并在真实环境里保持可控。你可以把它当作一张任务地图：从单个 Agent 的推理回路开始，逐步进入多人协作、记忆与知识，最后补上生产运行需要的安全和容错。

阅读时不必强记模式名称。每一节真正想回答的都是：**这类任务里，哪一种不确定性需要被消除？** 有时是下一步怎么做，有时是谁来做，有时是如何记住，有时则是如何避免造成不可逆的后果。

## 第一层：单 Agent 核心模式

### 01 ReAct — 推理与行动交织

**论文**：Yao et al., 2022 — _ReAct: Synergizing Reasoning and Acting in Language Models_

💡 核心思想

不要先想完再做，要**边想边做**。LLM 交替生成 **Thought（推理）** 和 **Action（行动）**，观察结果后继续推理。

> Thought: 用户想知道北京今天的天气。我需要搜索。
> Action: search("北京天气 2026年4月9日")
> Observation: 晴，18°C，北风3级
> Thought: 已经拿到天气数据了，可以回答了。
> Answer: 北京今天晴天，气温18°C，有北风3级。

**为什么有效**：

-   **推理指导行动**：每一步 Action 都有 Thought 解释"为什么要这样做"，减少盲目尝试
-   **行动反哺推理**：Observation 提供真实世界信息，让后续推理更准确
-   **可解释性**：Thought 链路本身就是天然的"决策日志"

**适用场景**：需要多步推理 + 工具调用的任务（问答、分析、调试）

**局限**：长链条容易"跑偏"，后续的 Observation 可能偏离原始目标。需要配合 Reflection 或 Planning 模式约束方向。

### 02 Plan-and-Execute — 先想清楚再动手

💡 核心思想

生成一个完整的执行计划，然后**按步骤执行**，而不是走一步看一步。与 ReAct 的"边想边做"不同，把**规划**和**执行**分离成两个明确阶段。

> [规划阶段]
> 用户：帮我分析竞品 HireGo 的最新动态
> Plan:
>   1. 搜索 HireGo 官网获取产品信息
>   2. 搜索 Google Play / App Store 评价
>   3. 搜索科技媒体最近 30 天的相关报道
>   4. 搜索 LinkedIn 上的招聘动态（判断团队扩张方向）
>   5. 综合以上信息，生成竞品情报简报
>
> [执行阶段]
> 执行步骤 1 → 观察 → 执行步骤 2 → 观察 → ... → 最终汇报

**关键设计**：

-   **计划可以动态调整**：执行步骤 2 时发现新线索，可以回过头补充步骤 1 的搜索
-   **与 ReAct 互补**：Plan-and-Execute 提供"全局视野"，ReAct 提供"局部灵活性"
-   **实际工程中经常混合使用**：先生成粗粒度计划，每一步内用 ReAct 模式执行

**适用场景**：多步骤、有明确流程的任务（研究报告、数据处理管线、代码重构）

**局限**：初始计划可能不完美；如果环境变化剧烈，静态计划会过时。

### 03 Reflection — 做完了回头看

💡 核心思想

让 Agent **评估自己的输出**，发现不足后自我修正。这可能是最"人类"的模式——做完一道题，检查一遍发现算错了，然后改正。

> [第一轮输出]
> "英伟达 2026 Q1 收入 280 亿美元，同比增长 78%"
>
> [自我反思]
> Wait, let me verify: 280 亿 vs 去年 Q1 的 260 亿...
> 280/260 - 1 = 7.7%，不是 78%。78% 是同比 vs 2024 Q1。
> 我应该明确标注对比基准。
>
> [修正输出]
> "英伟达 2026 Q1 收入 280 亿美元，同比增长 7.7%（环比），同比增长 78%（同比去年同期）"

**Andrew Ng 的经典案例**：让 LLM 写代码 → 运行测试 → 把错误反馈给 LLM → LLM 修正代码 → 再测试。这个循环可以重复多次，直到通过。

**实现要点**：

-   反馈信号可以是**外部**的（测试结果、用户反馈、工具返回值）也可以是**内部**的（LLM 自我评估）
-   反思频率要控制：每次都反思会浪费时间，只在关键节点反思（代码写完、报告生成后）
-   需要明确的"完成标准"，否则 Agent 会无限循环地"反思"

**适用场景**：代码生成、写作、数据分析——任何"质量很重要"的任务

### 04 Tool Use — 让 Agent 长出手脚

💡 核心思想

LLM 本身只是一个文本生成器，通过 **Function Calling / Tool Use**，让它能操作真实世界的工具。这是所有 Agent 系统的**基础设施**——没有 Tool Use，就没有 Agent。

> 用户: "帮我创建一个定时提醒，明天早上 9 点提醒我开会"
>
> LLM 内部决策:
>   需要调用 → cron_schedule(
>     message: "提醒：9点开会",
>     time: "2026-04-10T09:00:00+08:00"
>   )
>
> → 调用成功 → "已设置提醒：明天 9:00 — 提醒你开会"

**工具的分类**：

| 类别 | 示例 | 特征 |
| --- | --- | --- |
| **信息获取** | web\_search, web\_fetch, read\_file | 只读，无副作用 |
| **行动执行** | send\_message, write\_file, exec | 有副作用，需谨慎 |
| **交互工具** | browser, message, canvas | 双向通信 |
| **计算工具** | calculator, code\_interpreter | 确定性输出 |

**工程实践**：

-   **工具描述质量决定调用质量**：LLM 能否正确选择工具，90% 取决于工具的 description 写得好不好
-   **最小权限原则**：Agent 只需要它当前任务所需的工具，不要一股脑全给
-   **错误处理是核心**：工具调用失败是常态，Agent 必须能优雅地处理失败（重试、fallback、汇报）

### 05 Chain-of-Thought & Tree-of-Thought

**Chain-of-Thought**：让 LLM "把思考过程写出来"，而不是直接给答案。

> 差的做法: "答案是 42"（黑箱）
> 好的做法: "设 x 为...，代入公式得...，所以答案是 42"（过程可见）

这不是一个独立的 Agent 模式，而是**所有模式的底层基础设施**——ReAct 中的 Thought、Plan-and-Execute 中的 Plan、Reflection 中的评估，都依赖 CoT。

**Tree-of-Thought**：CoT 的升级版——不只一条思考链，而是**探索多条路径，选最优的**。

> 问题: "如何把搜索 API 的延迟从 2s 降到 200ms？"
>
> 路径 A: 换更快的 API 提供商 → 成本上升 3x → 不推荐
> 路径 B: 加缓存层 → 命中率未知 → 需要评估
> 路径 C: 并发请求多个 API，取最快返回 → 复杂度增加 → 值得尝试
>
> 评估: 路径 C 在成本可控的情况下可能达到目标 → 选择路径 C

**适用场景**：需要探索多个方案的复杂决策（架构设计、方案选型）

## 第二层：多 Agent 协作模式

当单个 Agent 能力不够时，就需要**多个专业 Agent 协作**。这就进入了编排（Orchestration）的领域。

### 06 Supervisor — 主管分配任务

一个"主管 Agent"负责理解用户意图，将任务拆解后**分发给专业 Agent 执行**。

> 用户: "帮我做一份 AI 行业的周报"
>
> Supervisor Agent:
>   → 信息搜集 Agent: "搜集本周 AI 行业重大新闻"
>   → 财务 Agent: "搜集本周 AI 相关公司股价动态"
>   → 研究 Agent: "搜集本周重要论文"
>   ← 汇总所有 Agent 的输出 → 整合为周报 → 发送给用户

💡 **一个常见实践**是由主管 Agent 协调信息搜集、财务和研究等专业 Agent。

**优点**：职责清晰，每个 Agent 专注自己的领域
**缺点**：Supervisor 是单点瓶颈；如果任务分解错误，整个流水线会跑偏

### 07 Hierarchical — 分层管理

Supervisor 的升级版——**多层级的指挥链**。

```text
协调 Agent
├── 产品负责人 Agent — 负责产品相关任务
│   ├── 竞品分析 Agent — 数据搜集
│   └── 用户研究 Agent — 数据搜集
├── 技术负责人 Agent — 负责技术任务
│   ├── 前端开发 Agent
│   └── 后端开发 Agent
└── 运营负责人 Agent — 负责运营任务
    ├── 数据分析 Agent
    └── 内容创作 Agent
```

**与 Supervisor 的区别**：Supervisor 是"扁平"的——一个老板直接管所有人。Hierarchical 是"树形"的——有中间管理层，可以处理更大规模的任务。

### 08 Swarm — 去中心化蜂群

没有中央控制，Agent 之间通过**手牌传递（Handoff）**完成协作。

> 用户: "我要订一张北京到上海的机票"
>
> 路由 Agent:
>   分析意图 → 识别为"机票预订"
>   → 手牌给 机票 Agent
>
> 机票 Agent:
>   搜索航班 → 发现需要登录
>   → 手牌给 认证 Agent
>
> 认证 Agent:
>   引导用户登录 → 登录成功
>   → 手牌回 机票 Agent
>
> 机票 Agent:
>   继续预订流程 → 完成

**Handoff（手牌）**：Agent A 完成自己的部分后，把上下文传递给 Agent B 继续。没有全局管理者，每个 Agent 只负责自己的职责。

**优点**：灵活、可扩展、没有单点瓶颈
**缺点**：流程难以追踪；如果 handoff 逻辑设计不当，任务会在 Agent 之间"乒乓"循环

**典型实现**：OpenAI Swarm 框架就是基于这个模式设计的。

### 09 Blackboard — 共享黑板

多个 Agent 围绕一块**共享的黑板**工作，各自往上面写信息，也可以读取别人的信息。

```text
[共享黑板]
┌─────────────────────────────────┐
│ 用户需求: "做一个天气预报 App"    │
│                                 │
│ [搜索 Agent 写入]               │
│ 天气API: OpenWeatherMap 免费    │
│ 地理API: Nominatim              │
│                                 │
│ [设计 Agent 写入]               │
│ 技术栈: React + Node.js        │
│ 架构: 三层（UI/API/数据）        │
│                                 │
│ [代码 Agent 读取黑板后开始编码]  │
│ ...                            │
└─────────────────────────────────┘
```

**与 Supervisor 的区别**：Supervisor 是"指令驱动"——主管告诉谁做什么。Blackboard 是"数据驱动"——谁看到自己能处理的信息就主动处理。

### 10 Pipeline / DAG — 流水线

任务被分解为**固定的阶段**，每个阶段由一个 Agent 处理，数据按顺序流过。

> 输入 → [搜集 Agent] → [分析 Agent] → [写作 Agent] → [审核 Agent] → 输出
>          原始数据        结构化分析       生成报告        质量检查

-   每个阶段可以并行处理（如果阶段间无依赖）
-   阶段间通过明确的接口传递数据
-   容易监控和调试（每个阶段的输入输出都是确定的）

**局限**：灵活性差——流程是固定的，不能根据中间结果动态调整。

## 第三层：记忆与知识模式

Agent 没有"真正的记忆"，每次对话都是全新开始。记忆模式解决的就是**跨会话的信息保持**问题。

### 11 短期记忆 — 当前对话上下文

**本质**：就是 LLM 的上下文窗口（Context Window）。

> 当前对话:
> 用户: "帮我分析一下 NVIDIA"
> Agent: [调用搜索工具，获取数据]
> 用户: "那 AMD 呢？"  ← Agent 通过短期记忆知道"分析"指的是财务分析

**工程挑战**：

-   **上下文窗口有限**：长对话会"遗忘"前面的内容（GPT-4 128K、Gemini 1M、GLM 128K）
-   **噪声累积**：对话越长，无关信息越多，影响推理质量
-   **成本**：Token 数 = 钱，长上下文 = 高成本

**工程实践**：对话压缩（定期摘要化）、滑动窗口（只保留最近 N 轮）、检索增强（按需检索相关片段）

### 12 长期记忆 — 跨会话的知识保持

让 Agent 在**会话之间**保持信息——记住你是谁、你做过什么、你的偏好。

| 方式 | 原理 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **文件记忆** | 读写 MEMORY.md / .learnings/ | 简单、透明、可审查 | 粗粒度、手动维护 |
| **向量记忆** | 信息 embedding 后存入向量数据库 | 语义检索、自动关联 | 需要额外基础设施 |
| **结构化记忆** | 知识图谱、关系数据库 | 精确查询、推理能力强 | 构建成本高 |

💡 **一种常见的文件记忆实践**：用长期记忆文档保存稳定事实，以按日期归档的记录保存原始上下文，再用学习日志沉淀错误和改进项。

### 13 RAG — 检索增强生成

不让 LLM "记住"所有知识，而是在需要时**从外部知识库检索**相关信息，注入到提示词中。

> 用户: "我们在 2026-04-02 讨论过什么？"
>
> → 向量数据库检索: query="2026-04-02 讨论"
> → 检索到: memory/2026-04-02.md 的相关片段
> → 注入到 prompt: "根据以下上下文回答用户问题: [检索结果]"
> → LLM 生成答案

💡 **RAG 是记忆模式的"搜索引擎"**——它解决的不是"怎么存"，而是"怎么快速找到需要的信息"。

## 第四层：生产级工程模式

学术原型和可用的生产系统之间，隔着工程实践。

### 14 Guardrails — 安全护栏

给 Agent 的行为设置**边界约束**，防止它做不该做的事。

```text
[输入护栏]                          [输出护栏]
用户输入 → ┌──────────┐ → LLM → ┌──────────┐ → 最终输出
          │ 过滤敏感词  │         │ 验证事实   │
          │ 检测注入    │         │ 检查格式   │
          │ 限制话题    │         │ 过滤有害内容│
          └──────────┘         └──────────┘
```

| 层面 | 示例 |
| --- | --- |
| **输入验证** | 检测 prompt injection、过滤敏感指令 |
| **工具权限** | exec 命令白名单、文件写操作需确认 |
| **输出审核** | 事实核查、格式验证、敏感信息过滤 |
| **行为约束** | 限制 Agent 只能访问特定数据源、不能发送外部消息 |

### 15 Human-in-the-Loop — 人在回路

Agent 不是完全自主的，**关键决策需要人类确认**。

> Agent: "我准备删除 /tmp/old-data/ 目录，共 342 个文件。确认删除？"
> 人类: [确认] / [拒绝] / [修改: 只删除 .log 文件]

| 决策类型 | 自动化程度 | 原因 |
| --- | --- | --- |
| 读文件、搜索 | 全自动 | 无副作用 |
| 写文件到指定位置 | 半自动 | 可能覆盖已有内容 |
| 发送消息给他人 | 需确认 | 有外部影响 |
| 删除数据 | 需确认 | 不可逆操作 |
| 执行 shell 命令 | 按风险分级 | 可能影响系统安全 |

**工程实现**：审批门（approval gate）是 Human-in-the-Loop 的典型实现。

### 16 Fallback & Retry — 容错与降级

Agent 的每一步都可能失败，系统必须能**优雅地处理失败**。

```text
搜索请求 → web_search(Gemini)
              ↓ 429 限流
           unified-search(Tavily)
              ↓ 无结果
           unified-search(Exa)
              ↓ 全部失败
           返回提示: "搜索服务暂时不可用，请稍后重试"
```

💡 **一个稳健的搜索层**可以在主搜索源没有结果时自动切换到备用来源，并向用户提供有意义的降级结果。

**容错设计原则**：

1.  **快速失败（Fail Fast）**：不要在一个注定失败的操作上等太久
2.  **有意义的降级**：fallback 方案不能是"什么都做不了"，要提供部分功能
3.  **重试有上限**：无限重试 = 无限循环，必须设置最大重试次数
4.  **记录失败原因**：把失败写入结构化错误队列，供后续分析和改进

### 17 Self-Improvement — 自我进化

Agent 能从自己的错误中学习，持续改进。

```text
执行任务 → 出错 → 记录错误 → 分析模式 → 提炼规则 → 下次避免
                                                    ↑
                                              注入到行为中
```

**一种典型实现**：

-   在工具调用后自动检测错误，并写入结构化错误队列
-   在构建提示词前注入待处理的学习提醒
-   通过定期审查归档错误、检测重复模式，并将规则提升为稳定配置

💡 **关键洞察**：自我进化的核心不是"学习"本身——LLM 天生就会从上下文中学习。核心是**让学习持续化、系统化、自动化**，而不是依赖人工提醒。

## 模式全景图

```text
[生产级工程层]
├ Guardrails
├ Human-in-Loop
├ Fallback
└ Self-Improvement

[多 Agent 编排层]
├ Supervisor
├ Hierarchical
├ Swarm
├ Blackboard
└ Pipeline

[记忆与知识层]
├ 短期记忆
├ 长期记忆
└ RAG

[单 Agent 核心层]
├ ReAct
├ Plan-Execute
├ Reflection
└ Tool Use · CoT / ToT
```

一个完整的 Agent 系统通常会同时使用：**ReAct**（边想边调用工具）、**Tool Use**、**Supervisor**（协调专业 Agent）、**文件记忆**、**Guardrails**、**Human-in-the-Loop**、**Fallback** 与 **Self-Improvement**。这些模式共同构成可运行、可治理的系统。

## 选型指南

| 任务特征 | 推荐模式 | 理由 |
| --- | --- | --- |
| 简单问答 | **ReAct + Tool Use** | 一两步就能解决，不需要复杂规划 |
| 多步研究 | **Plan-and-Execute + ReAct** | 需要全局视野，但每步需要灵活性 |
| 代码生成 | **Tool Use + Reflection** | 需要实际运行代码并修正 |
| 多领域协作 | **Supervisor / Hierarchical** | 需要专业分工 |
| 高并发处理 | **Swarm / Pipeline** | 无需中央协调，可并行 |
| 知识密集型 | **RAG + 长期记忆** | 需要从大量知识中检索 |
| 安全敏感 | **Guardrails + Human-in-Loop** | 必须有约束和确认 |
| 长期运行 | **Self-Improvement + Fallback** | 需要从错误中学习，需要容错 |

## 小结：先选任务模式，再谈代码结构

到这里，你应该能先判断一个 Agent **该怎样工作**：它需要边想边做，还是先规划再执行；需要一个主管协调，还是让专业 Agent 自主交接；需要记忆什么，又要在哪些关键动作前停下来让人确认。

这些选择决定了系统的行为边界。下一步才是把这些能力组织成可替换、可观测、可恢复的代码结构。
