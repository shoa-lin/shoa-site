## 结论先说

**当 AI 只负责生成文字时，Retry 是重新采样一个答案；当 AI 开始操作真实世界时，Retry 就可能变成重复执行一次任务。**

这也是 Codex 这类 Agent 产品不再突出传统“重新生成”按钮的重要背景：

- Chatbot 的 Retry 通常只替换一段文字；
- Agent 的一次运行可能已经修改文件、运行命令或调用外部工具；
- 第二次运行面对的，已经不是第一次运行之前的世界；
- 因此更可靠的交互不是笼统地 Retry，而是继续反馈、重跑失败步骤或从明确状态 Fork。

这里需要说明：目前没有官方资料明确表示，Codex 是“因为状态复杂而取消了 Retry”。本文是在官方公开的 Thread、Turn、Item 和 Fork 机制基础上，对 Agent 产品交互做出的架构分析。

![Retry 从重新回答变成保留历史的分支](/assets/blog/ai-agent-retry-state/retry-becomes-fork.png)

过去使用 Chatbot 时，我很习惯一个功能：**Retry，或者叫重新生成。**

用户问一个问题，AI 给出一个答案。如果答案不满意，不需要重新组织问题，点一下 Retry，AI 就会再回答一次。

这个功能很自然，以至于当我开始使用 Codex 时，反而产生了一个疑问：**为什么在 Codex 里，我们很少看到那个熟悉的“重新生成”按钮？**

它是被遗漏了吗？还是 Agent 产品不再重视这种交互？

我现在更倾向于一个解释：

> 当 AI 只负责生成文字时，Retry 是重新采样一个答案；当 AI 开始操作真实世界时，Retry 就可能变成重复执行一次任务。

这两件事看起来只差一个按钮，背后却是两种完全不同的系统。

## Chatbot 时代，Retry 为什么这么简单

传统 Chatbot 的核心交互大致是：

```text
用户问题 → 模型生成 → 回答 A
                   └→ Retry → 回答 B
```

第一次回答不满意，系统可以保留相同的对话上下文，再让模型生成一次。旧答案即使被丢弃，通常也不会产生什么严重后果。

因为在大多数情况下，Chatbot 改变的只是屏幕上的文字：

- 没有修改本地文件；
- 没有运行一段命令；
- 没有改变 Git 分支；
- 没有向外部系统写入数据；
- 没有发送一封无法收回的邮件。

从产品体验上看，这很像让一个人把同一道题换一种方式再回答一遍。第一次答案不好，就把纸揉掉，重新写一张。

所以 Chatbot 的 Retry 往往可以被理解为：**保留输入，放弃输出，再生成一个候选结果。**

## 但 Codex 返回的不只是“一段回答”

Codex 这类 Agent 的不同之处，是它在给出最终回答之前，可能已经做了很多事情。

OpenAI 对 Codex App Server 的定义里，一个 Thread 由多个 Turn 组成，而一个 Turn 又包含多个 Item。这里的 Item 不只是用户消息和 AI 消息，也包括命令执行、文件修改和工具调用。

换句话说，我们在界面上看到的最后一段文字，可能只是整次工作的总结。

```text
用户任务
   │
   ▼
理解仓库与上下文
   │
   ▼
读取文件 → 运行命令 → 调用工具 → 修改代码 → 执行测试
   │
   ▼
最终回复：“已经修改了这些内容……”
```

这时候，如果用户对最后的回复不满意，再点一次 Retry，系统到底应该重试哪一部分？

只重新写最后的总结？重新进行一次推理？重新运行所有命令？还是先撤销已经发生的修改，再从头执行？

问题从这里开始变得不再简单。

## 同一句任务，第二次执行时，世界已经不一样了

假设我对 Codex 说：

> 修复这个 Bug，运行测试，然后创建一个 Pull Request。

第一次执行时，Codex 可能已经：

1. 读取代码并定位问题；
2. 修改了三个文件；
3. 运行测试；
4. 创建了新分支；
5. 提交并推送代码；
6. 创建了一个 Pull Request。

如果这时点击 Retry，会发生什么？

- 如果只重新生成文字，新回复可能和已经发生的操作对不上；
- 如果重新推理，第二次看到的是一份已经被修改过的代码；
- 如果重新执行工具，可能产生重复提交、冲突分支或者第二个 PR；
- 如果要求先回滚，外部操作又未必都能撤销。

例如文件可以恢复，邮件却可能已经发出；本地分支可以删除，外部系统的审批、消息和交易未必能够无痕回退。

所以真正的问题不是笼统的“中间信息太复杂”，而是：

> 一次 Agent 运行已经形成了真实的因果链。第二次运行面对的，不再是第一次运行之前的世界。

![文字可以重写，但 Agent 已经改变了真实状态](/assets/blog/ai-agent-retry-state/text-vs-world-state.png)

## Agent 的 Retry，至少涉及四种状态

如果从系统角度拆开，一次 Agent 工作通常同时改变四类状态。

### 1. 对话状态

包括用户要求、历史消息、已经确认的约束、Agent 的推理线索，以及工具返回的结果。

重新生成时，哪些历史应该保留，哪些应该丢弃，本身就是一个选择。

### 2. 执行状态

包括本地文件、Git 工作区、运行中的进程、测试产物、临时文件和安装的依赖。

这些状态可能已经被第一次运行改变。

### 3. 外部状态

包括 GitHub PR、数据库记录、发送出去的消息、云端任务、表单和第三方系统。

这一类状态最危险，因为它们经常无法完全回滚。

### 4. 权限与审计状态

包括用户批准过哪些动作、调用过哪些工具、什么时候发生了什么，以及系统应该如何追踪责任。

如果 Retry 静默重放了一串操作，系统必须回答：第二次执行是新的授权，还是沿用第一次授权？两次行为在审计记录里是什么关系？

这四类状态叠加在一起之后，Retry 就不再是一个简单按钮，而更像一套关于回滚、避免重复执行和分支管理的机制。

## Retry 到底可能代表什么

在 Agent 产品里，用户口中的“再试一次”，至少可能有四种不同含义：

```text
再写一次    → 保留已完成的工作，只重写最终回答
重新规划    → 保留当前环境，但换一条推理和执行路线
重新执行    → 再运行一次失败的命令或工具
回到分叉点  → 保留原始历史，从某个状态创建一条新分支
```

这四个动作的风险完全不同，却很难被压缩进同一个 Retry 按钮。

这也是为什么我认为，Agent 产品未来不是完全不需要 Retry，而是需要把 Retry **拆成语义更明确的操作**。

例如：

- “只重新生成回答”；
- “根据当前状态继续修改”；
- “重新运行失败步骤”；
- “从这里创建新分支”；
- “恢复到某个检查点后重跑”。

只有用户和系统都知道究竟重试什么，这个动作才是可靠的。

## Codex 没有简单 Retry，但提供了 Fork

能从官方资料确认的是，Codex 把工作建模为 Thread、Turn 和 Item，并提供了 `fork` 机制。Codex CLI 的 `/fork` 可以把当前任务复制成一个新任务；App Server 也提供 `thread/fork`，用于在保留原始历史的情况下创建新的 Thread。

Fork 和 Retry 看起来相似，但它们表达的是两种不同态度：

- Retry 像是在说：“刚才那次不算，再来一次。”
- Fork 则是在说：“刚才那次真实发生过，现在从这里探索另一条路线。”

对于只能输出文字的 Chatbot，第一种说法通常没有问题。

但对于会修改文件、运行命令和调用外部工具的 Agent，第二种说法更诚实，也更容易追踪。

它承认历史已经发生，新的尝试应该拥有自己的分支和身份，而不是悄悄覆盖掉旧过程。

## 很多时候，反馈比 Retry 更有效

当我们不满意 Agent 的结果时，真正需要的往往也不是“全部再来一次”，而是告诉它哪里不对。

比如：

- 保留前面的调研，但把结论写得更直接；
- 不要改 API，只调整内部实现；
- 测试结果可以保留，重新检查根因；
- 当前代码不要回滚，换一种 UI 方案；
- 先不要继续执行，把已有改动展示给我。

这种反馈可以复用已经付出的成本，也让 Agent 明确知道上一轮哪里没有满足要求。

传统 Retry 依赖随机性，希望下一次“运气更好”；而高质量的 Agent 协作，更像人与同事一起工作：指出偏差、保留正确部分、从当前状态继续收敛。

## 一个小按钮背后的产品分水岭

所以，Codex 里不再突出传统 Retry，并不一定意味着少了一个功能。

它更像一个产品分水岭：AI 正在从“回答机器”变成“行动系统”。

当 AI 只生成文字时，历史是可以被覆盖的；当 AI 已经开始做事，历史就成为系统状态的一部分。

这时候，真正重要的不再是“能不能再回答一次”，而是：

- 能不能知道上一轮究竟做了什么；
- 能不能保留正确结果，只修改错误部分；
- 能不能从明确的状态创建另一条路线；
- 能不能避免工具和外部操作被重复执行；
- 能不能让每次尝试都有可追踪的因果关系。

> Retry 按钮的淡出，不只是少了一个熟悉的交互，而是提醒我们：当 AI 开始改变真实世界，“再试一次”就必须回答一个更严肃的问题——究竟要从哪里，再做什么？
