The first article covered how an agent **works**: how it reasons, calls tools, coordinates work, and where memory and safety boundaries belong. Once implementation begins, the difficult question changes from "what should happen next?" to **how should these capabilities be organized into a system that remains maintainable over time?**

When it comes to implementation, the questions are often very specific:

- There are three search tools: Tavily, Exa, and Gemini. How can the Agent not care which one to use?
- Different Agents (finance, research, operations) need to be searched, but the parameters are different. How to **unify management tool registration**?
- Agent needs to do logging, current limiting, and permission checking before and after calling the tool. Why not mix business code and cross-cutting logic?
- The content generated by Agent needs to be reviewed, formatted, and translated. How can it be put into a processing pipeline?

These are not abstract “AI problems”, but new forms of classic software engineering problems in Agent systems. This article does not require you to memorize 12 patterns; the goal is to first see clearly what changes and what risks each pattern isolates for you.

## An Engineering Map

You can first divide the following patterns into three categories according to the problems they solve:

- **Replacement and Combination**: Strategy, Factory, and Adapter allow models, tools, and Agent configurations to be replaced instead of being scattered into a large number of conditional judgments.
- **Cross-cutting control**: Chain of responsibility, Observer, and Decorator extract permissions, logs, current limiting, caching, and auditing from the core business.
- **State and Boundary**: Sandbox, Checkpoint, Prompt Composition, and State Machine handle the Agent-specific problems of isolation, recovery, context, and life cycle.

With this map, when looking at each pattern, the focus is not "what is it called?" but "what kind of change or loss of control is my system facing?"

## I. Classic GoF Patterns in Agent Systems

### 01 Strategy — The soul of tool selection

**Classic definition**: Define a family of algorithms and encapsulate each one so that they can be replaced with each other.

**Agent scenario**: There are multiple implementations behind the same "search" operation - Tavily, Exa, Gemini. The caller should not know which one to use.

```python
# ❌ 没有策略模式 — 到处都是 if-else
def search(query: str):
    if use_tavily:
        return tavily_client.search(query)
    elif use_exa:
        return exa_client.search(query)
    else:
        return gemini_search(query)

# ✅ 策略模式 — 搜索器是可替换的
from abc import ABC, abstractmethod

class SearchProvider(ABC):
    @abstractmethod
    async def search(self, query: str, n: int = 5) -> list[SearchResult]:
        ...

class TavilySearch(SearchProvider):
    async def search(self, query, n=5):
        resp = await http_post("https://api.tavily.com/search", ...)
        return [SearchResult(title=r["title"], ...) for r in resp["results"]]

class ExaSearch(SearchProvider):
    async def search(self, query, n=5):
        resp = await http_post("https://api.exa.ai/search", ...)
        return [SearchResult(title=r["title"], ...) for r in resp["results"]]

# 统一搜索服务 — FallbackChain 本身也是一种策略
class UnifiedSearch(SearchProvider):
    def __init__(self, providers: list[SearchProvider]):
        self.providers = providers  # [TavilySearch(), ExaSearch()]

    async def search(self, query, n=5):
        for provider in self.providers:
            try:
                results = await provider.search(query, n)
                if results:
                    return results
            except Exception:
                continue
        raise AllProvidersFailedError()
```

💡 **This is also the core design of the unified search service**. Strategy mode makes the expansion of fallback links natural - to add a new search source, just implement the `SearchProvider` interface and add it to the list.

**Framework practice**: **DeerFlow** Lead Agent selects sub-Agents, which is essentially strategy selection; **AgentScope** `Toolkit.register_tool_function()` registers tools - tools are strategies.

### 02 Factory — Unified creation of Agent

**Classic definition**: Encapsulate the creation logic of the object, and the caller does not need to know the specific class.

**Agent scenario**: There are multiple Agents in the system, each with a different model, system prompt, and tool set. Creation logic shouldn't be scattered everywhere.

```python
# ❌ 没有工厂 — 每次都要记住每个 Agent 的配置
financial_advisor = Agent(model="model-a", system_prompt="你是专业财务顾问", tools=[search, message])
researcher = Agent(model="model-b", system_prompt="你是信息搜集专家", tools=[search, web_fetch])

# ✅ 工厂模式 — 配置驱动，一处定义
AGENT_REGISTRY = {
    "financial-advisor": AgentConfig(
        name="Financial Advisor", model="model-a",
        system_prompt="你是专业财务顾问。",
        tools=["search", "message"],
    ),
    "researcher": AgentConfig(
        name="Researcher", model="model-b",
        system_prompt="你是信息搜集专家。",
        tools=["search", "web-fetch", "browser"],
    ),
}

class AgentFactory:
    @staticmethod
    def create(agent_id: str) -> Agent:
        config = AGENT_REGISTRY[agent_id]
        tools = [ToolRegistry.get(tool) for tool in config.tools]
        return Agent(model=config.model, prompt=config.system_prompt, tools=tools)

agent = AgentFactory.create("financial-advisor")
result = agent.run("分析最新财报")
```

**Framework practice**: **DeepAgents** `create_deep_agent()` hides LangGraph graph compilation details; **AgentScope** `ReActAgent()` hides memory and formatter initialization; runtime configuration can also declare an Agent list and create corresponding instances according to identifiers.

### 03 Chain of Responsibility — Tool calling pipeline

**Classic Definition**: A request is passed along a chain of handlers, with each handler deciding to process or pass it on to the next.

**Agent scenario**: When an Agent calls a tool, it needs to go through multiple layers of checks - permissions → current limiting → log → execution → result processing.

```python
class ToolHandler(ABC):
    def __init__(self, next_handler: "ToolHandler | None" = None):
        self.next = next_handler

    def handle(self, ctx: ToolContext) -> ToolResult:
        if self.next:
            return self.next.handle(ctx)
        return ctx.execute()

class PermissionCheck(ToolHandler):
    def handle(self, ctx):
        if ctx.tool_name == "exec" and ctx.command in DANGEROUS_COMMANDS:
            if not ctx.has_approval():
                return ToolResult(approval_required=True)
        return super().handle(ctx)

class RateLimiter(ToolHandler):
    def handle(self, ctx):
        if self.limiter.is_limited(ctx.tool_name):
            return ToolResult(error="Rate limited, retry later")
        return super().handle(ctx)

class AuditLogger(ToolHandler):
    def handle(self, ctx):
        logger.info(f"[TOOL_CALL] {ctx.tool_name}({ctx.params})")
        result = super().handle(ctx)
        logger.info(f"[TOOL_RESULT] status={result.status}")
        return result

# 组装管线
pipeline = PermissionCheck(RateLimiter(AuditLogger()))
result = pipeline.handle(ToolContext(tool_name="exec", command="rm -rf /"))
# → PermissionCheck 拦截 → 返回 approval_required
```

💡 **The design philosophy of plug-in Hook** can form an implicit chain of responsibility: insert cross-cutting logic before building prompt words and after tool calls.

### 04 Observer — event-driven loose coupling

**Classic definition**: When the state of an object changes, all objects that depend on it are automatically notified.

**Agent scenario**: After the tool call is completed, multiple systems need to know - the error queue needs to be recorded, the monitoring needs to be reported, and the learning engine needs to analyze. They should not be coupled to each other.

```python
from typing import Protocol

class ToolEventObserver(Protocol):
    def on_tool_call(self, tool: str, params: dict, result: ToolResult):
        ...

class ErrorQueueObserver:
    """自动将错误记录到队列"""
    def on_tool_call(self, tool, params, result):
        if result.exit_code != 0:
            error_queue.append({
                "tool": tool, "error": result.stderr,
                "timestamp": datetime.now().isoformat()
            })

class MetricsObserver:
    """记录工具调用耗时和成功率"""
    def on_tool_call(self, tool, params, result):
        metrics.record(tool, duration=result.duration_ms, success=result.ok)

class LearningReminderObserver:
    """检测到重复错误时注入学习提醒"""
    def on_tool_call(self, tool, params, result):
        if is_repeated_error(tool, result.stderr):
            inject_reminder(f"你之前犯过同样的错误：{result.stderr[:100]}")

class ToolEventBus:
    def __init__(self):
        self._observers: list[ToolEventObserver] = []

    def register(self, observer: ToolEventObserver):
        self._observers.append(observer)

    def emit(self, tool, params, result):
        for obs in self._observers:
            obs.on_tool_call(tool, params, result)

# 注册观察者（松耦合，互不知道对方存在）
bus = ToolEventBus()
bus.register(ErrorQueueObserver())
bus.register(MetricsObserver())
bus.register(LearningReminderObserver())
```

💡 **The learning module can adopt the observer architecture**: the observer after the tool is called listens for events. When adding capabilities, you only need to register a new observer.

### 05 Decorator — Non-intrusive feature enhancements

**Classic definition**: Dynamically add additional responsibilities to an object without changing its interface.

**Agent scenario**: Add retries, caching, logging, and current limiting to the Agent - without modifying the code of the Agent itself.

```python
def with_retry(max_retries=3, delay=1.0):
    """工具调用自动重试装饰器"""
    def decorator(func):
        @functools.wraps(func)
        async def wrapper(*args, **kwargs):
            for attempt in range(max_retries):
                try:
                    return await func(*args, **kwargs)
                except RateLimitError:
                    if attempt < max_retries - 1:
                        await asyncio.sleep(delay * (attempt + 1))
                    else:
                        raise
        return wrapper
    return decorator

def with_cache(ttl_seconds=300):
    """搜索结果缓存装饰器"""
    cache: dict[str, tuple[float, Any]] = {}

    def decorator(func):
        @functools.wraps(func)
        async def wrapper(query: str, *args, **kwargs):
            key = f"{func.__name__}:{query}"
            if key in cache:
                ts, result = cache[key]
                if time.time() - ts < ttl_seconds:
                    return result  # 缓存命中
            result = await func(query, *args, **kwargs)
            cache[key] = (time.time(), result)
            return result
        return wrapper
    return decorator

# 组合使用 — 搜索工具自动缓存 + 重试
@with_cache(ttl_seconds=300)
@with_retry(max_retries=3, delay=2.0)
async def search_web(query: str, n: int = 5) -> list[SearchResult]:
    return await unified_search.search(query, n)
```

**Difference from Chain of Responsibility**: The chain of responsibility is "pipelined" - the request passes through each processor in turn. Decorators are "wrapped" - each layer of decorators wraps around core functionality. The effect is similar, but the decorator is lighter.

### 06 Adapter — Unified interface for heterogeneous tools

**Classic definition**: Convert the interface of a class into another interface expected by the client.

**Agent scenario**: Different LLM providers (OpenAI, Google, Wisdom) have different API formats, and the Agent code should not care about these differences.

```python
class LLMAdapter(ABC):
    @abstractmethod
    async def complete(self, messages: list[dict], tools: list = None) -> str:
        ...

class OpenAIAdapter(LLMAdapter):
    async def complete(self, messages, tools=None):
        resp = await openai_client.chat.completions.create(
            model="gpt-4o", messages=messages, tools=tools
        )
        return resp.choices[0].message.content

class GeminiAdapter(LLMAdapter):
    async def complete(self, messages, tools=None):
        resp = await gemini_client.generate_content(
            contents=messages, tools=tools
        )
        return resp.text

class GLMAdapter(LLMAdapter):
    async def complete(self, messages, tools=None):
        resp = await glm_client.chat.completions.create(
            model="glm-5", messages=messages
        )
        return resp.choices[0].message.content

# Agent 代码完全不关心底层用的是哪个模型
class Agent:
    def __init__(self, llm: LLMAdapter):
        self.llm = llm

    async def run(self, task: str):
        response = await self.llm.complete([
            {"role": "system", "content": self.system_prompt},
            {"role": "user", "content": task}
        ])
        return response
```

**Framework practice**: **DeepAgents** `init_chat_model("openai:gpt-4o")` One function adapts to all providers; **OpenClaw** `zai/glm-5`, `google/gemini-2.5-flash` unified `provider/model` format.

### 07 Template Method — The skeleton of the Agent execution process

**Classic definition**: Define the algorithm skeleton in the base class and defer certain steps to the subclass implementation.

**Agent scenario**: The execution process of all Agents is "receive tasks → think → call tools → process results → return", but the specific implementation of each step is different.

```python
class BaseAgent(ABC):
    """所有 Agent 的执行骨架"""

    async def run(self, task: str) -> str:
        context = await self.before_run(task)
        result = await self.think_and_act(context)
        return await self.after_run(result)

    async def before_run(self, task):
        logger.info(f"[{self.name}] 开始任务: {task[:50]}...")
        return TaskContext(task=task, memory=self.load_memory())

    @abstractmethod
    async def think_and_act(self, ctx: TaskContext) -> str:
        ...

    async def after_run(self, result):
        self.save_memory(result)
        return result

class ReActAgent(BaseAgent):
    async def think_and_act(self, ctx):
        while not ctx.is_complete():
            thought = await self.llm.complete(ctx.to_messages())
            action = self.parse_action(thought)
            observation = await self.execute_tool(action)
            ctx.add_step(thought, action, observation)
        return ctx.final_answer()

class PlanExecuteAgent(BaseAgent):
    async def think_and_act(self, ctx):
        plan = await self.generate_plan(ctx.task)
        for step in plan.steps:
            result = await self.execute_step(step)
            plan.mark_done(step, result)
        return plan.compile_results()
```

**Framework Practice**: **AgentScope** `ReActAgent` overrides `think_and_act`; **DeepAgents** `create_deep_agent()` returns the compiled LangGraph StateGraph.

## II. Agent-Native Engineering Patterns

The above is the mapping of the classic GoF mode in Agent. The following patterns are more "Agent-native" - ​​not common in traditional software, but almost essential in Agent development.

### 08 Sandbox — Safely execute untrusted code

The code/commands generated by the Agent are executed in an **isolated environment** and do not affect the host system.

```text
Agent 生成的代码
      │
      ▼
┌─────────────┐
│  Docker 容器  │  ← 隔离的文件系统、网络、进程
│  或 E2B 沙箱  │
└─────────────┘
      │
      ▼
  执行结果（ stdout / 产出文件 ）
```

⚠️ **This is not a GoF mode, but it is the bottom line of the Agent system** - allowing the Agent to execute arbitrary shell commands without sandboxing is equivalent to handing over system control to a black box.

**Framework Practice**: **DeerFlow** Each task runs in an independent Docker container; **DeepAgents** supports remote sandbox (E2B); **OpenClaw** exec tool has an allowlist mechanism.

### 09 Skill — Ability module loaded on demand

Agent's capabilities are not fixed, but modular skills loaded on demand. Just like game characters equipped with different skill books.

```python
class Skill(ABC):
    name: str
    description: str  # Agent 读取这段描述决定是否使用

    @abstractmethod
    def get_tools(self) -> list[Tool]: ...

    @abstractmethod
    def get_instructions(self) -> str: ...

class Agent:
    def equip_skill(self, skill: Skill):
        self.skills[skill.name] = skill
        for tool in skill.get_tools():
            self.register_tool(tool)
        self.append_system_prompt(skill.get_instructions())
```

💡 **Key Insight**: Skill is not just a "collection of tools" - it also contains **usage guidelines** (instructions) that tell the Agent \*when to use it and how to use it\*. Therefore, `SKILL.md` should not only write the API address, but also the usage policy.

**Framework Practice**: **DeerFlow** Skill is a Markdown file; **OpenClaw** Skill = SKILL.md + optional script/hook.

### 10 Checkpoint — persistence and recovery of long tasks

Long-running tasks may be interrupted (timeout, crash, API current limit), and need to save state at key nodes to support recovery from breakpoints.

```python
class CheckpointManager:
    def save(self, task_id: str, state: TaskState):
        db.put(f"checkpoint:{task_id}", state.to_json())

    def restore(self, task_id: str) -> TaskState | None:
        data = db.get(f"checkpoint:{task_id}")
        return TaskState.from_json(data) if data else None

# 第一次运行：执行到第 3 步超时
agent.run(task_id="report-001", task="生成 AI 周报")
# → 保存 checkpoint：{completed_steps: [1,2,3], current: 4}

# 恢复运行：从第 4 步继续
agent.run(task_id="report-001", task="生成 AI 周报")
# → 加载 checkpoint → 跳过步骤 1-3 → 从步骤 4 继续
```

**Why Agent is particularly needed**: The retry of traditional programs is to "execute it again". The Agent cannot retry from scratch - LLM has already spent 30 seconds thinking about the plan and the first 5 steps have been executed. Re-execution is a huge waste.

### 11 Prompt Composition — Layered construction System Prompt

The Agent's system prompt is not a piece of hard-coded text, but the product of **multi-layered dynamic combination**.

> Final System Prompt =
> Basic Code of Conduct
> + role definition
> + User Preferences
> + Current task context
> + Usage Guide for Equipped Skills
> + Pending study reminders
> + tool description list

```python
class PromptComposer:
    def build(self, agent: Agent, context: RunContext) -> str:
        parts = []
        # 1. 核心身份（固定）
        parts.append(read_file(agent.soul_path))
        parts.append(read_file(agent.identity_path))
        # 2. 动态注入
        if context.pending_learnings:
            parts.append(f"## 待处理学习\n{context.pending_learnings}")
        if context.equipped_skills:
            for skill in context.equipped_skills:
                parts.append(skill.get_instructions())
        # 3. 工具描述
        parts.append("## 可用工具\n")
        for tool in agent.tools:
            parts.append(f"- {tool.name}: {tool.description}")
        return "\n\n".join(parts)
```

**Framework Practice**: Many runtimes will dynamically inject content during the prompt word construction phase; **AgentScope** `sys_prompt + formatter` two layers; **DeerFlow** Skill + task + tool definition dynamic combination.

### 12 State Machine — Precise control of Agent life cycle

The execution process of the Agent is modeled as a finite state machine, and each state has clear transition rules.

```text
[PLANNING]
  │ plan_ready
  ▼
[EXECUTING]
  ├─ success
  │   └─ [REVIEW]
  │       └─ [COMPLETE]
  ├─ error
  │   └─ [FALLBACK]
  │       ├─ recovered
  │       │ └─ [REVIEW]
  │       └─ failed
  │         └─ [TIMEOUT]
  └─ timeout
      └─ [TIMEOUT]

[TIMEOUT]
  └─ retry
      └─ [EXECUTING]
```

**Why a state machine is needed**: Without a state machine, the Agent's state transitions are scattered in various if-else - "If the execution fails and the number of retries < 3, retry, otherwise downgrade, timeout if the downgrade fails" - this kind of logic quickly becomes spaghetti.

**Framework practice**: **DeepAgents** The bottom layer is LangGraph StateGraph; **DeerFlow** state chart + supervisor mode; **OpenClaw** Cron job status management (ok → error → retry → timeout).

## III. Comparing Four Frameworks

| Dimensions | OpenClaw | AgentScope | DeepAgents | DeerFlow |
| --- | --- | --- | --- | --- |
| **Positioning** | Personal AI Assistant Platform | Universal Agent Framework | Agent Harness (SDK) | Super Agent Harness |
| **Language** | TypeScript | Python | Python | Python (backend) |
| **Bottom layer** | Self-research (Pi Agent) | Self-research | LangGraph + LangChain | LangGraph + LangChain |
| **Core Mode** | Plugin + Hook + Skill | ReAct Agent + Workflow | StateGraph + Tool | Supervisor + Sandbox |
| **Sandbox** | exec allowlist | No built-in | E2B remote sandbox | Docker container |
| **Multi-Agent** | Supervisor | Workflow + MessageHub | SubAgent (task tool) | Supervisor (Lead→sub) |
| **Memory** | File System | InMemory + Database | Auto-summarization | Persistent + TIAMAT |
| **MCP** | mcporter | built-in | langchain-mcp-adapters | none |
| **Applicable scenarios** | Daily assistant, automation | Research, prototype verification | Coding Agent, SDK | Long tasks, content production |

#### OpenClaw TypeScript · Plugin + Hook

Not a traditional Python SDK, but a runtime platform. Core design patterns: Plugin extension, Hook chain of responsibility, Channel Adapter, Skill system.

- **Plugin mode**: Third-party extensions register hooks and tools through plugins and are loosely coupled with the core
- **Hook mode**: `before_prompt_build`, `after_tool_call`, `agent_end` form an implicit chain of responsibility
- **Channel abstract**: unified interface for channels such as Telegram, Discord, Feishu, etc. - Adapter mode
- **Skill System**: Markdown-driven skill loading - zero code threshold

💡 **Unique Advantages**: Skill is defined in pure Markdown, with zero coding threshold. Anyone can write a Skill, no Python is required.

#### AgentScope Python · Alibaba Damo Academy

Academic rigor + engineering practical route. Built-in ReAct mode, Toolkit abstraction, Workflow orchestration, Memory layering, and A2A protocol support.

- **ReAct built-in**: `ReActAgent` is a first class citizen and works out of the box
- **Toolkit Abstract**: `Toolkit.register_tool_function()` - Factory + Strategy
- **Workflow Orchestration**: Pipeline mode combines multiple Agents
- **Memory layering**: InMemory + Database + ReMe (enhanced long-term memory)
- **A2A Protocol**: Agent-to-Agent communication standard - decentralized collaboration

💡 **Unique Advantages**: A2A protocol support allows Agents to interoperate across systems and frameworks. The built-in model fine-tuning (Agentic RL) capability is also unique.

#### DeepAgents Python · LangChain Official

Design philosophy: Trust LLM and make constraints at the tool level. Agent Harness works right out of the box.

- **create\_deep\_agent factory**: Create a complete Agent with one line of code
- **Built-in Planning**: `write_todos` tool allows Agent to decompose tasks independently
- **Built-in Filesystem**: read/write/edit/ls/glob/grep
- **Sub-Agent dispatch**: `task` tool creates isolated sub-Agents
- **Auto-summarization**: Automatic summary when the conversation is too long

💡 **Design philosophy**: "Enforce boundaries at the tool/sandbox level, not by expecting the model to self-police."

#### DeerFlow Python · ByteDance

Core difference: Most frameworks are the "inference layer", DeerFlow is the "execution layer".

- **Supervisor + Sub-Agent**: Lead Agent decomposes tasks and dispatches sub-Agents for parallel execution
- **Docker Sandbox**: each task runs in an isolated container, with real file system and bash
- **Skill System**: Markdown defines skills and loads them on demand
- **Persistent Memory**: cross-session memory, asynchronous update
- **Stateful Pipeline**: Based on LangGraph checkpointing, long tasks can be interrupted and resumed

💡 **Unique Advantages**: Really give Agent "a computer" - not just generate text, but generate downloadable PPT, runnable code, and deployable web pages.

## IV. Quick Pattern-Selection Guide

| The problem you want to solve | Recommended mode | Corresponding to classic mode |
| --- | --- | --- |
| Multiple implementations of the same operation | **Strategy** | Strategy pattern |
| Unifiedly create Agents with different configurations | **Factory** | Factory mode |
| Multi-layer processing before and after tool invocation | **Chain of Responsibility** | Chain of responsibility model |
| Multiple listeners after tool execution | **Observer** | Observer pattern |
| Caching/retrying the tool without changing the code | **Decorator** | Decorator mode |
| Unified interface for different LLM providers | **Adapter** | Adapter mode |
| Define Agent execution process skeleton | **Template Method** | Template method pattern |
| Isolate execution of untrusted code | **Sandbox** | — |
| Load ability modules on demand | **Skill** | — |
| Long task interruption recovery | **Checkpoint** | — |
| Dynamic composition System Prompt | **Prompt Composition** | — |
| Precisely control Agent state transitions | **State Machine** | State machine mode |
