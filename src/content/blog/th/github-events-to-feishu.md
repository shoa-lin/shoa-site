ในการทำงานร่วมกันด้านวิศวกรรม มีงานเล็ก ๆ ที่รบกวนสมาธิอยู่เสมอ นั่นคือการเปิด GitHub ซ้ำ ๆ เพื่อตรวจว่ามี PR, Issue หรือ Review ใหม่หรือไม่ คำถามที่สำคัญจริง ๆ ไม่ใช่เพียง “เกิดเหตุการณ์หรือยัง” แต่คือ “สิ่งนี้มีความหมายอะไรต่อทีม”

ฉันจึงชอบทำให้มันเป็นสายงานที่ขับเคลื่อนด้วยเหตุการณ์ เมื่อ GitHub มีความเปลี่ยนแปลง Local Agent จะถูกปลุกขึ้น คัดเฉพาะข้อเท็จจริงที่จำเป็น แล้วส่งอัปเดตด้านวิศวกรรมสั้น ๆ หนึ่งข้อความไปยังกลุ่ม Feishu

![รวบรวมเหตุการณ์ GitHub เป็นอัปเดตด้านวิศวกรรม](/assets/blog/github-events-to-feishu/01-event-to-update.png)

## แนวคิดนั้นเรียบง่าย

สายงานทั้งหมดมีหน้าตาเช่นนี้:

```text
GitHub Webhook
→ Cloudflare Tunnel
→ Local Agent
→ กลุ่มวิศวกรรม Feishu
```

GitHub เป็นแหล่งข้อเท็จจริง เช่น มีการเปิด PR, Review ขอให้แก้ไข หรือ Issue ถูกปิด Cloudflare Tunnel นำคำขอ HTTPS จากสาธารณะไปยังบริการที่รับฟังเฉพาะบนเครื่องภายในอย่างปลอดภัย จากนั้น Local Agent จะสรุปเหตุการณ์เป็นภาษาจีนแบบสั้นที่คนอ่านได้รวดเร็ว แล้วส่งให้กลุ่มทีม

หัวใจคือการแบ่งหน้าที่ให้ชัดเจน Tunnel ทำหน้าที่ขนส่งเท่านั้น ไม่ตีความโค้ดและไม่เรียกโมเดล ส่วน Agent จัดการเฉพาะเหตุการณ์ที่ตรวจสอบแล้ว และไม่ทำงานเขียนบน GitHub แทนทีม

## ทำไมไม่คอยสำรวจ GitHub ตามเวลา

การเรียก GitHub API ทุกไม่กี่นาทีทำได้ แต่จะสร้างคำขอที่ไม่มีประโยชน์ ความหน่วง และภาระในการจำว่าอะไรถูกตรวจไปแล้ว Webhook เหมาะกับความต้องการจริงมากกว่า คือแจ้งเมื่อ repository เปลี่ยน และไม่ทำอะไรเมื่อไม่มีการเปลี่ยนแปลง

สำหรับ repository เดียวและกลุ่มวิศวกรรมเดียว เพียงเท่านี้ก็เพียงพอ ไม่จำเป็นต้องเริ่มจาก message queue, event platform หรือระบบกำกับหลาย repository ที่ซับซ้อน

## ขอบเขตสองอย่างที่ต้องรักษา

ข้อแรกคือความปลอดภัย การมี URL สาธารณะไม่ได้แปลว่าใครก็เรียก Agent ได้ ตัวรับควรตรวจสอบลายเซ็น Webhook จากเนื้อหาคำขอดิบก่อนแยกวิเคราะห์เหตุการณ์ และใช้ delivery ID ตัดรายการซ้ำเพื่อไม่ให้การส่งซ้ำของแพลตฟอร์มสร้างข้อความซ้ำในกลุ่ม

ข้อที่สองคือสิทธิ์ Agent นี้เหมาะจะเป็นผู้ช่วยสรุปข้อมูลแบบอ่านอย่างเดียว คืออ่านบริบทที่จำเป็น สรุปข้อเท็จจริง ชี้ความเสี่ยง และส่งการแจ้งเตือน โดยปริยายไม่ควร push โค้ด merge PR แก้ไข Issue หรือส่งต่อ payload ดิบและข้อมูลรับรอง

## ข้อความในกลุ่มควรมีหน้าตาอย่างไร

แทนที่จะโยน JSON ดิบเข้าไปในกลุ่ม อัปเดตด้านวิศวกรรมที่ดีควรตอบเพียงสามเรื่อง: เกิดอะไรขึ้น กระทบตรงไหน และต้องมีคนติดตามหรือไม่

```text
เปิด PR แล้ว

ข้อเท็จจริง: มีอะไรเพิ่มและสถานะปัจจุบันคืออะไร
จุดที่ควรดู: โมดูลหรือการเปลี่ยนแปลงที่ควรใส่ใจ
ลิงก์: กลับไปดูบริบทต้นฉบับบน GitHub
```

วินัยเล็ก ๆ ที่ช่วยได้มากคือแยกข้อเท็จจริงออกจากการตีความ “PR ถูก merge แล้ว” คือข้อเท็จจริง ส่วน “อาจกระทบความเข้ากันได้” คือการตีความที่ยังต้องตรวจสอบ วิธีนี้ทำให้ข้อความมีประโยชน์โดยไม่สรุปเกินจริง

## ชุดเริ่มต้นที่ใช้งานได้จริง

หากอยากลอง เริ่มจากชุดเล็ก ๆ นี้ได้เลย:

```text
GitHub Webhook
+ Cloudflare Tunnel
+ ตัวรับ Webhook ที่รับฟังเฉพาะในเครื่อง
+ การตรวจลายเซ็นและตัดรายการซ้ำ
+ สรุปเหตุการณ์แบบอ่านอย่างเดียว
+ Feishu Bot เฉพาะงาน
```

สมัครรับเฉพาะเหตุการณ์ที่สำคัญจริง เช่น Issue, Pull request และ Review ก่อนอื่นทำให้การแจ้งเตือนเชื่อถือได้ สั้น และตรวจสอบย้อนกลับได้ แล้วค่อยขยายเมื่อเจอปัญหาเรื่องข้อความรบกวน การจัดการหลาย repository หรือความต้องการด้าน audit และ retry จริง ๆ

Agent ที่ขับเคลื่อนด้วยเหตุการณ์ไม่ใช่เรื่องลึกลับ GitHub ให้ข้อเท็จจริง Tunnel ให้เส้นทาง Agent จัดระเบียบข้อมูล และ Feishu ส่งต่อเพื่อการทำงานร่วมกัน การลดการรีเฟรชหน้าเว็บซ้ำ ๆ ก็เป็น automation ที่มีค่าแล้ว

> ส่งลิงก์บทความนี้ให้ AI Agent ของคุณ ให้มันเข้าใจแนวคิดก่อน แล้วค่อยออกแบบเวอร์ชันเล็กที่สุดสำหรับทีมของคุณ
>
> ไม่ต้องคัดลอกบัญชี กุญแจ หรือการตั้งค่าภายใน เริ่มจากสายงานเล็ก ๆ ที่เชื่อถือได้ก็พอ
