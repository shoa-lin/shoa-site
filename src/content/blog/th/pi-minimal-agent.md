*เขียนเมื่อวันที่ 31 มกราคม 2026*

OpenClaw กลายเป็นกระแสอย่างรวดเร็วภายใต้หลายชื่อ รวมถึง ClawdBot และ MoltBot แก่นของมันคือ Agent ที่เชื่อมต่อกับช่องทางการสื่อสารและสามารถรันโค้ดได้

เบื้องหลัง OpenClaw ใช้ coding agent ขนาดเล็กชื่อ **Pi** Armin Ronacher อธิบายว่า Pi คือ coding agent ที่ปัจจุบันเขาใช้แทบทั้งหมด และชี้ให้เห็นว่าเหตุใดการออกแบบที่จงใจให้เล็กจึงน่าสนใจมาก

Pi สร้างโดย **Mario Zechner** แนวทางที่ยึดความเป็นจริงของ Mario แตกต่างจากแนว "ไซไฟปนความบ้าคลั่งเล็กน้อย" ของ Peter แต่ Pi และ OpenClaw ยืนอยู่บนสมมติฐานเดียวกัน: LLM เขียนและรันโค้ดได้ดีมาก ระบบจึงควรใช้ความสามารถนั้นอย่างเต็มที่

## Pi คืออะไร?

Pi เป็นหนึ่งใน coding agent จำนวนมาก แต่มีคุณลักษณะสองข้อที่ทำให้โดดเด่น Armin ยังกล่าวถึง **AMP** ว่าเป็นอีกผลิตภัณฑ์ที่เห็นได้ชัดว่าถูกสร้างโดยคนที่ทดลอง agentic programming อย่างจริงจัง ไม่ใช่เพียงห่อมันด้วยอินเทอร์เฟซที่ดูสวยงาม

Pi น่าสนใจด้วยเหตุผลหลักสองข้อ:

- **แกนมีขนาดเล็กมาก** พรอมป์ระบบสั้นผิดปกติ และแกนหลักเปิดให้ใช้เพียงสี่เครื่องมือ ได้แก่ Read, Write, Edit และ Bash
- **ระบบ extension ทรงพลัง** Extension สามารถเพิ่มพฤติกรรม และบันทึกสถานะของตัวเองลงใน session อย่างถาวรได้

ยังมีข้อดีในทางปฏิบัติด้วย: Pi ถูกเขียนขึ้นเหมือนซอฟต์แวร์ที่ผ่านการดูแลอย่างรอบคอบ มันเสถียร ใช้หน่วยความจำน้อย และไม่รบกวนผู้ใช้ด้วยหน้าจอกะพริบหรือความล้มเหลวแบบสุ่ม

Pi ยังเป็นชุดคอมโพเนนต์ขนาดเล็กสำหรับใช้สร้าง Agent อื่น ๆ โดย OpenClaw ก็พัฒนาขึ้นจากคอมโพเนนต์เหล่านี้ Armin นำไปใช้สร้าง Telegram bot ส่วน Mario ใช้สร้าง `mom` และเมื่อให้ Pi อ่านโค้ดของตัวเองพร้อมตัวอย่างอย่าง `mom` มันก็ช่วยนำคอมโพเนนต์เหล่านี้มาประกอบเป็น Agent ตัวใหม่ที่เชื่อมต่อกับระบบเป้าหมายได้

## สิ่งที่ไม่มีใน Pi

การเข้าใจ Pi ต้องเข้าใจสิ่งที่จงใจไม่รวมไว้ด้วย แกนหลัก **ไม่มีการรองรับ MCP ในตัว** แต่ไม่ได้หมายความว่าจะใช้ MCP ไม่ได้ Extension สามารถเพิ่มความสามารถนี้ หรือ Agent อาจใช้ **mcporter** ซึ่งเปิดให้เรียก MCP ผ่าน CLI หรือ TypeScript bindings

การละความสามารถนี้ออกสะท้อนปรัชญาของ Pi เมื่อ Agent ขาดความสามารถบางอย่าง แนวทางตั้งต้นไม่ใช่ค้นหา extension สำเร็จรูปจาก marketplace แต่เป็นการขอให้ Agent ขยายตัวเองด้วยการเขียนและรันโค้ด

Pi ยังคงรองรับการดาวน์โหลด extension ความแตกต่างอยู่ที่วัฒนธรรมการใช้งาน: extension ที่มีอยู่สามารถเป็นข้อมูลอ้างอิงให้ Agent นำมาดัดแปลงตามความต้องการในเครื่อง แทนที่จะมองว่าเป็น dependency ที่แก้ไขไม่ได้

## Agent ที่สร้างมาเพื่อ Agent ซึ่งกำลังสร้าง Agent

ซอฟต์แวร์ที่ต้องปรับรูปตัวเองจำเป็นต้องมีความสามารถพื้นฐานบางประการ

ข้อแรก AI SDK ของ Pi อนุญาตให้ session หนึ่งมีข้อความจากผู้ให้บริการโมเดลหลายรายได้ มันยอมรับว่า session ไม่สามารถย้ายข้ามระบบได้อย่างสมบูรณ์แบบ ขณะเดียวกันก็หลีกเลี่ยงการผูกกับฟีเจอร์เฉพาะของผู้ให้บริการโดยไม่จำเป็น

ข้อสอง ไฟล์ session สามารถมี custom messages อยู่ร่วมกับข้อความของโมเดล Extension ใช้ข้อความเหล่านี้เพื่อบันทึกสถานะได้ และระบบสามารถกำหนดให้ข้อมูลบางส่วนไม่ถูกส่งไปยังโมเดลเลย หรือส่งไปเพียงบางส่วน

ข้อสาม สถานะของ extension บันทึกลงดิสก์ได้ และ extension รองรับ hot reload Agent จึงเขียน extension โหลดใหม่ ทดสอบ และทำซ้ำต่อได้ ข้อสี่ Pi มาพร้อมเอกสารและตัวอย่างที่ Agent อ่านได้ระหว่างขยายตัวเอง ข้อห้า session มีโครงสร้างเป็น tree ผู้ใช้สามารถแตก branch ไปทำงานย่อย แก้เครื่องมือที่เสียโดยไม่ใช้บริบทของ branch หลัก แล้วกลับมาโดยให้ Pi สรุปว่าเกิดอะไรขึ้นในอีก branch

การตัดสินใจเหล่านี้สำคัญต่อเครื่องมือ ในผู้ให้บริการโมเดลหลายราย เครื่องมือ MCP และเครื่องมือ LLM อื่น ๆ จะถูกโหลดลงในบริบทระบบหรือส่วนเครื่องมือตั้งแต่เริ่ม session การแทนที่คำจำกัดความทั้งหมดในภายหลังอาจทำลายแคช หรือทำให้โมเดลมีหน่วยความจำที่ขัดแย้งกันว่าการเรียกเครื่องมือก่อนหน้านี้ทำงานอย่างไร

## ความสามารถที่อยู่นอกบริบทของโมเดล

Extension ของ Pi สามารถลงทะเบียนเครื่องมือที่ LLM เรียกใช้ได้ และบางครั้ง Armin ก็เลือกวิธีนี้ issue tracker ที่สร้างไว้ใช้ในเครื่องเป็นตัวอย่างหนึ่ง เนื่องจาก Agent ต้องจัดการสิ่งที่ต้องทำโดยตรง เขาจึงเปิดเครื่องมือเพิ่มอีกหนึ่งรายการแทนการใช้ CLI ปัจจุบันนี่เป็นเครื่องมือเสริมเพียงรายการเดียวที่เขาโหลดเข้าไปในบริบทของโมเดล

ความสามารถเสริมส่วนใหญ่ไม่จำเป็นต้องใช้พื้นที่ในบริบทของโมเดลในรูป tool schema พวกมันเป็น skills หรือ TUI extensions ที่ช่วยปรับปรุงเวิร์กโฟลว์ระหว่างมนุษย์กับระบบ Extension ของ Pi สามารถแสดง spinner แถบความคืบหน้า ตัวเลือกไฟล์ ตาราง และ preview pane ได้โดยตรงในเทอร์มินัล Mario ถึงกับสาธิตการรัน Doom ใน TUI แม้จะใช้จริงไม่ได้มากนัก แต่พิสูจน์ได้ดีว่าอินเทอร์เฟซนี้ยืดหยุ่นเพียงใด

Extension ต่อไปนี้เป็นเพียงตัวอย่าง ไม่ใช่แพ็กเกจตายตัว เวิร์กโฟลว์ที่ตั้งใจไว้คือชี้ Agent ไปยังตัวอย่างหนึ่ง แล้วขอให้มันดัดแปลงพฤติกรรมนั้น

### `/answer`

Armin ไม่ใช้ Plan Mode เขาชอบการโต้ตอบที่เดินหน้าไปมาอย่างเป็นธรรมชาติผ่านร้อยแก้วของ Agent โดยแทรกคำอธิบายและแผนภาพเป็นระยะ มากกว่าบทสนทนาคำถามแบบมีโครงสร้างตายตัว

คำถามที่แทรกอยู่ในข้อความอาจตอบให้เป็นระเบียบได้ยาก ดังนั้น `/answer` จะอ่านข้อความตอบล่าสุดของ Agent ดึงคำถามออกมา แล้วจัดรูปใหม่เป็นกล่องรับข้อมูลที่มีจุดโฟกัสชัดเจน

![Extension /answer แสดงกล่องคำถาม](https://lucumr.pocoo.org/static/pi-answer.png)

### `/todos`

แม้ Armin จะวิจารณ์วิธีสร้าง Beads แต่เขาเห็นว่ารายการสิ่งที่ต้องทำสำหรับ Agent มีประโยชน์ `/todos` จะเปิดรายการที่เก็บเป็นไฟล์ Markdown ใต้ `.pi/todos` ทั้งผู้ใช้และ Agent แก้ไขไฟล์เหล่านี้ได้ และ session สามารถรับงานหนึ่งไปทำเพื่อทำเครื่องหมายว่ากำลังดำเนินการ

### `/review`

เมื่อ Agent เขียนโค้ดมากขึ้น งานที่ยังไม่เสร็จสมบูรณ์ควรผ่านการ review โดย Agent ก่อนส่งให้มนุษย์ เพราะ session ของ Pi เป็น tree Armin จึงแตก branch ไปยังบริบท review ใหม่ รวบรวมข้อค้นพบ แล้วนำการแก้ไขกลับมายัง session หลักได้

![Extension /review แสดงตัวเลือก preset สำหรับการ review](https://lucumr.pocoo.org/static/pi-review.png)

อินเทอร์เฟซนี้ยึด Codex เป็นต้นแบบ และรองรับเป้าหมายการ review เช่น commit, diff, การเปลี่ยนแปลงที่ยังไม่ commit และ pull request บน remote พรอมป์สำหรับ review เน้น feedback ที่ Armin ให้ความสำคัญ รวมถึงการระบุ dependency ที่เพิ่มเข้ามาใหม่อย่างชัดเจน

### `/control`

นี่เป็น extension เชิงทดลอง ไม่ใช่ส่วนหนึ่งของเวิร์กโฟลว์ประจำวันของ Armin มันเปิดให้ Pi Agent ตัวหนึ่งส่งพรอมป์ไปยังอีกตัว จึงสร้างระบบหลาย Agent ขนาดเล็กได้โดยไม่ต้องมี orchestration layer ที่ซับซ้อน

### `/files`

Extension นี้แสดงรายการไฟล์ที่ถูกแก้ไขหรืออ้างถึงใน session ผู้ใช้สามารถแสดงไฟล์ใน Finder เปรียบเทียบ diff ใน VS Code เปิดด้วย Quick Look หรืออ้างถึงไฟล์ในพรอมป์ได้ `shift+ctrl+r` จะเปิด Quick Look ของไฟล์ล่าสุดที่ถูกกล่าวถึง ซึ่งสะดวกเมื่อ Agent สร้าง PDF

นักพัฒนาคนอื่นสร้าง extension เพิ่มเติมด้วย เช่น subagent extension ของ Nico และ `interactive-shell` ที่ช่วยให้ Pi รัน CLI แบบโต้ตอบได้เองภายใน TUI overlay ที่ผู้ใช้มองเห็น

## ซอฟต์แวร์สร้างซอฟต์แวร์

ประเด็นสำคัญคือ Armin ไม่ได้เขียน extension เหล่านี้ด้วยมือตัวเอง เขาอธิบายสิ่งที่ต้องการ แล้ว Pi เป็นผู้สร้าง แกนของ Pi ไม่มี MCP และไม่มี community skills ที่ติดตั้งมาให้ แต่ Agent สามารถสร้างและดูแลความสามารถที่ปรับให้เหมาะกับเจ้าของได้ ตัวอย่างหนึ่งคือแทนที่ CLI สำหรับ browser automation หรือ integration ผ่าน MCP ด้วย skill ที่สื่อสารกับ CDP โดยตรง

Agent ของเขามี skills จำนวนมาก แต่ทั้งหมดทิ้งและสร้างใหม่ได้ บางรายการอ่านข้อความใน session ของ Pi ที่วิศวกรคนอื่นแชร์มาเพื่อช่วย code review ส่วนรายการอื่นกำหนดรูปแบบ commit message พฤติกรรมการ commit หรือการอัปเดต changelog เขายังย้าย slash commands เดิมบางส่วนไปเป็น skills และใช้ skill ที่สนับสนุน `uv` ร่วมกับ extension ที่เปลี่ยนเส้นทางการเรียก `pip` และ `python` ไปยัง `uv`

นี่คือเสน่ห์ของ Agent แบบมินิมอลอย่าง Pi: มันทำให้ซอฟต์แวร์สร้างซอฟต์แวร์กลายเป็นวิธีทำงานตามปกติ OpenClaw ผลักแนวคิดนี้ไปอีกขั้นด้วยการนำ UI ในเครื่องออก แล้วเชื่อม Agent เข้ากับแชต ข้อสรุปของ Armin ไม่ใช่ว่ารายละเอียดทุกอย่างลงตัวแล้ว แต่คือทิศทางนี้ดูเหมือนจะเป็นส่วนหนึ่งของอนาคตซอฟต์แวร์มากขึ้นเรื่อย ๆ
