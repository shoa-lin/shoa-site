## สรุปก่อน

**เมื่อ AI มีหน้าที่เพียงสร้างข้อความ Retry คือการสร้างคำตอบอีกแบบหนึ่ง แต่เมื่อ AI เริ่มลงมือทำกับโลกจริง Retry อาจหมายถึงการรันงานทั้งงานซ้ำอีกครั้ง**

ความแตกต่างนี้ช่วยอธิบายว่าเหตุใดผลิตภัณฑ์ Agent อย่าง Codex จึงไม่ได้เน้นปุ่ม “สร้างคำตอบใหม่” แบบ Chatbot เดิม

- Retry ของ Chatbot มักเปลี่ยนเพียงข้อความหนึ่งชุด
- การทำงานหนึ่งรอบของ Agent อาจแก้ไฟล์ รันคำสั่ง หรือเรียกเครื่องมือภายนอกไปแล้ว
- เมื่อเริ่มรอบที่สอง โลกไม่ได้อยู่ในสถานะเดิมก่อนรอบแรกอีกต่อไป
- วิธีที่ปลอดภัยกว่าคือให้ feedback ต่อ รันเฉพาะขั้นตอนที่ล้มเหลว หรือ Fork จากสถานะที่ระบุชัดเจน

อย่างไรก็ตาม ไม่มีเอกสารทางการที่ระบุว่า Codex “ยกเลิก Retry เพราะสถานะซับซ้อนเกินไป” บทความนี้เป็นการวิเคราะห์เชิงสถาปัตยกรรมจากโมเดล Thread, Turn, Item และ Fork ที่เปิดเผยต่อสาธารณะ

![จาก Retry ที่สร้างคำตอบใหม่ ไปสู่การแตกแขนงที่เก็บประวัติเดิมไว้](/assets/blog/ai-agent-retry-state/retry-becomes-fork-th.png)

ตอนใช้ Chatbot รุ่นก่อน ผมคุ้นเคยกับฟังก์ชันเล็ก ๆ อย่าง **Retry** หรือ **Regenerate** มาก

ผู้ใช้ถามคำถาม AI ตอบกลับมา หากยังไม่พอใจ ก็ไม่ต้องเขียนคำถามใหม่ เพียงกด Retry ก็ได้คำตอบอีกแบบหนึ่ง

ความรู้สึกนี้เป็นธรรมชาติมาก จนเมื่อเริ่มใช้ Codex ผมกลับสงสัยว่า **ทำไมปุ่ม “สร้างคำตอบใหม่” ที่คุ้นเคยจึงไม่เด่นชัดใน Codex**

มันแค่ถูกละไว้หรือไม่ หรือผลิตภัณฑ์ Agent ต้องการรูปแบบการโต้ตอบที่ต่างออกไป?

คำอธิบายที่ผมเห็นว่าสมเหตุสมผลที่สุดในตอนนี้คือ:

> เมื่อ AI มีหน้าที่เพียงสร้างข้อความ Retry คือการสร้างคำตอบอีกแบบหนึ่ง แต่เมื่อ AI เริ่มลงมือทำกับโลกจริง Retry อาจหมายถึงการรันงานทั้งงานซ้ำอีกครั้ง

ดูเหมือนต่างกันแค่ปุ่มเดียว แต่เบื้องหลังคือระบบสองประเภทที่แตกต่างกันโดยสิ้นเชิง

## ทำไม Retry ในยุค Chatbot จึงเรียบง่าย

การโต้ตอบของ Chatbot แบบเดิมมีหน้าตาประมาณนี้:

```text
คำถามของผู้ใช้ → โมเดลสร้างคำตอบ → คำตอบ A
                                      └→ Retry → คำตอบ B
```

หากคำตอบแรกไม่ดี ระบบสามารถเก็บบริบทการสนทนาเดิมแล้วให้โมเดลสร้างคำตอบใหม่ การทิ้งคำตอบแรกมักไม่ก่อให้เกิดผลกระทบร้ายแรง

เพราะในกรณีส่วนใหญ่ สิ่งที่ Chatbot เปลี่ยนมีเพียงข้อความบนหน้าจอ:

- ไม่ได้แก้ไขไฟล์ในเครื่อง
- ไม่ได้รันคำสั่ง
- ไม่ได้เปลี่ยน Git branch
- ไม่ได้เขียนข้อมูลลงระบบภายนอก
- ไม่ได้ส่งอีเมลที่เรียกคืนไม่ได้

ประสบการณ์นี้คล้ายการขอให้ใครสักคนตอบคำถามเดิมด้วยวิธีใหม่ หากกระดาษแผ่นแรกไม่ดี ก็ขยำทิ้งแล้วเขียนใหม่

ดังนั้น Retry ของ Chatbot จึงหมายถึง **เก็บ input ไว้ ทิ้ง output แล้วสร้างคำตอบตัวเลือกใหม่**

## สิ่งที่ Codex ส่งกลับมาไม่ใช่แค่ “คำตอบ”

Agent อย่าง Codex อาจทำงานไปมากแล้วก่อนจะเขียนคำตอบสุดท้าย

ในโมเดลของ Codex App Server หนึ่ง Thread ประกอบด้วยหลาย Turn และแต่ละ Turn มีหลาย Item โดย Item ไม่ได้มีเพียงข้อความของผู้ใช้กับ AI แต่รวมถึงการรันคำสั่ง การเปลี่ยนไฟล์ และการเรียกเครื่องมือ

ข้อความสุดท้ายที่เห็นในหน้าจอจึงอาจเป็นเพียงสรุปของงานทั้งหมด:

```text
งานของผู้ใช้
   │
   ▼
ทำความเข้าใจ repository และ context
   │
   ▼
อ่านไฟล์ → รันคำสั่ง → เรียกใช้เครื่องมือ → แก้ไขโค้ด → รันการทดสอบ
   │
   ▼
คำตอบสุดท้าย: “ได้แก้ไขเนื้อหาต่อไปนี้แล้ว…”
```

หากผู้ใช้ไม่พอใจกับคำตอบสุดท้าย Retry ควรทำส่วนใดซ้ำ?

เขียนเฉพาะสรุปใหม่? คิดเหตุผลใหม่? รันทุกคำสั่งอีกครั้ง? หรือย้อนการเปลี่ยนแปลงที่เกิดขึ้นแล้วจึงเริ่มตั้งแต่ต้น?

ตั้งแต่จุดนี้ Retry ก็ไม่ใช่การทำงานที่เรียบง่ายอีกต่อไป

## ในรอบที่สอง โลกได้เปลี่ยนไปแล้ว

สมมติว่าผมบอก Codex ว่า:

> แก้ bug นี้ รันการทดสอบ แล้วเปิด Pull Request

ในรอบแรก Codex อาจทำสิ่งเหล่านี้เสร็จแล้ว:

1. อ่านโค้ดและหาสาเหตุ
2. แก้ไขไฟล์สามไฟล์
3. รันชุดทดสอบ
4. สร้าง branch ใหม่
5. commit และ push การเปลี่ยนแปลง
6. เปิด Pull Request

หากกด Retry ตอนนี้ จะเกิดอะไรขึ้น?

- หากสร้างเฉพาะข้อความใหม่ รายงานอาจไม่ตรงกับสิ่งที่เกิดขึ้นจริง
- หากคิดใหม่ รอบที่สองจะเริ่มจาก repository ที่ถูกแก้ไขไปแล้ว
- หากรันเครื่องมือซ้ำ อาจเกิด commit ซ้ำ branch ขัดแย้ง หรือ PR ที่สอง
- หากพยายาม rollback ก่อน บางการกระทำในระบบภายนอกอาจย้อนกลับไม่ได้

ไฟล์อาจกู้คืนได้ แต่อีเมลอาจถูกส่งไปแล้ว branch ในเครื่องอาจลบได้ แต่การอนุมัติ ข้อความ หรือธุรกรรมในระบบภายนอกอาจไม่สามารถย้อนกลับโดยไร้ร่องรอย

ดังนั้นปัญหาจริงไม่ใช่แค่ “ข้อมูลระหว่างทางซับซ้อนเกินไป” แต่คือ:

> การทำงานหนึ่งรอบของ Agent สร้างห่วงโซ่เหตุและผลที่เกิดขึ้นจริง รอบที่สองไม่ได้เริ่มจากโลกก่อนรอบแรก

![ข้อความเขียนใหม่ได้ แต่สถานะของโลกที่ Agent เปลี่ยนไปแล้วไม่ได้หายไป](/assets/blog/ai-agent-retry-state/text-vs-world-state-th.png)

## Retry ของ Agent เกี่ยวข้องกับสถานะอย่างน้อยสี่ประเภท

จากมุมมองของระบบ งานหนึ่งงานของ Agent สามารถเปลี่ยนสถานะได้สี่ประเภท

### 1. สถานะการสนทนา

รวมถึงคำขอของผู้ใช้ ข้อความก่อนหน้า ข้อจำกัดที่ยืนยันแล้ว บริบทการให้เหตุผล และผลลัพธ์จากเครื่องมือ

การ Retry ต้องตัดสินใจว่าประวัติส่วนใดยังใช้ได้ และส่วนใดควรถูกทิ้ง

### 2. สถานะการทำงาน

รวมถึงไฟล์ในเครื่อง Git working tree โปรเซสที่กำลังรัน ผลลัพธ์การทดสอบ ไฟล์ชั่วคราว และ dependency ที่ติดตั้งแล้ว

สิ่งเหล่านี้อาจเปลี่ยนไปตั้งแต่รอบแรก

### 3. สถานะภายนอก

รวมถึง GitHub Pull Request, record ในฐานข้อมูล, ข้อความที่ส่งออกไป, cloud job, แบบฟอร์ม และระบบของบุคคลที่สาม

สถานะประเภทนี้อันตรายที่สุด เพราะหลายการกระทำไม่สามารถ rollback ได้ทั้งหมด

### 4. สถานะสิทธิ์และการตรวจสอบ

รวมถึงการกระทำที่ผู้ใช้อนุมัติ เครื่องมือที่ถูกเรียก เวลาเกิดเหตุการณ์ และข้อมูลที่ใช้ติดตามความรับผิดชอบ

หาก Retry เล่นชุดการกระทำซ้ำโดยเงียบ ๆ รอบที่สองยังอยู่ภายใต้การอนุมัติเดิมหรือไม่ หรือต้องขออนุมัติใหม่? ใน audit log ควรเชื่อมโยงสองรอบนี้อย่างไร?

เมื่อสถานะทั้งสี่ทับซ้อนกัน Retry จึงไม่ใช่ปุ่มธรรมดา แต่เป็นกลไกสำหรับ rollback ป้องกันการทำซ้ำ และจัดการการแตกแขนง

## Retry อาจหมายถึงอะไรกันแน่

ในผลิตภัณฑ์ Agent คำว่า “ลองอีกครั้ง” อาจหมายถึงการทำงานอย่างน้อยสี่แบบ:

```text
เขียนใหม่          → เก็บงานที่เสร็จแล้วไว้ และเขียนเฉพาะคำตอบสุดท้ายใหม่
วางแผนใหม่         → เก็บ environment ปัจจุบันไว้ แต่เปลี่ยนเส้นทางการให้เหตุผลและการทำงาน
รันใหม่             → รันคำสั่งหรือการเรียกเครื่องมือที่ล้มเหลวอีกครั้ง
กลับไปยังจุด Fork  → เก็บประวัติเดิมไว้ และสร้างแขนงใหม่จากสถานะหนึ่ง
```

ความเสี่ยงของการทำงานเหล่านี้แตกต่างกันมาก แต่ปุ่ม Retry เพียงปุ่มเดียวซ่อนความแตกต่างไว้

ผลิตภัณฑ์ Agent ยังต้องการการลองใหม่ เพียงแต่ควรแยกเป็นคำสั่งที่มีความหมายชัดเจน:

- สร้างเฉพาะคำตอบสุดท้ายใหม่
- แก้ไขต่อจากสถานะปัจจุบัน
- รันเฉพาะขั้นตอนที่ล้มเหลว
- สร้างแขนงใหม่จากจุดนี้
- กู้คืน checkpoint แล้วรันใหม่จากตรงนั้น

Retry จะเชื่อถือได้ก็ต่อเมื่อทั้งผู้ใช้และระบบรู้ชัดว่ากำลังทำอะไรซ้ำ

## Codex ใช้ Fork แทน Retry แบบครอบจักรวาล

สิ่งที่ยืนยันได้จากเอกสารทางการคือ Codex จัดรูปงานเป็น Thread, Turn และ Item พร้อมมีกลไก `fork` ใน Codex CLI คำสั่ง `/fork` จะคัดลอกงานปัจจุบันไปเป็นงานใหม่ ส่วน App Server มี `thread/fork` เพื่อสร้าง Thread ใหม่โดยเก็บประวัติเดิมไว้

Fork และ Retry ดูคล้ายกัน แต่สื่อความหมายต่างกัน:

- Retry: “ถือว่ารอบเมื่อกี้ไม่นับ แล้วทำใหม่”
- Fork: “รอบเมื่อกี้เกิดขึ้นจริง จากตรงนี้ลองเส้นทางอื่น”

สำหรับ Chatbot ที่สร้างเพียงข้อความ ความหมายแรกมักเพียงพอ

แต่สำหรับ Agent ที่แก้ไฟล์ รันคำสั่ง และเรียกเครื่องมือภายนอก ความหมายที่สองซื่อตรงกว่าและติดตามได้ง่ายกว่า

Fork ยอมรับว่าประวัติศาสตร์เกิดขึ้นแล้ว การลองใหม่มีแขนงและตัวตนของตัวเอง แทนที่จะเขียนทับกระบวนการเดิม

## หลายครั้ง feedback มีประโยชน์กว่า Retry

เมื่อไม่พอใจกับผลลัพธ์ของ Agent เราไม่ได้ต้องการให้ทำทุกอย่างใหม่เสมอไป ส่วนใหญ่เราต้องการบอกว่าจุดใดไม่ถูกต้อง

- เก็บงานวิจัยไว้ แต่เขียนข้อสรุปให้ตรงขึ้น
- อย่าเปลี่ยน API ให้แก้เฉพาะ implementation ภายใน
- เก็บผลการทดสอบไว้ แต่ตรวจสอบ root cause ใหม่
- อย่า rollback โค้ดปัจจุบัน ให้ลองแนวทาง UI อีกแบบ
- หยุดการทำงานก่อน แล้วแสดงการเปลี่ยนแปลงที่มีอยู่

feedback แบบนี้รักษางานที่มีประโยชน์ไว้ และบอก Agent อย่างชัดเจนว่ารอบก่อนพลาดเป้าตรงไหน

Retry แบบเดิมอาศัยความสุ่มและหวังว่าคำตอบรอบหน้าจะดีขึ้น การร่วมงานกับ Agent ที่ดีคล้ายการทำงานกับเพื่อนร่วมทีม: ชี้ความคลาดเคลื่อน เก็บส่วนที่ถูกต้อง และค่อย ๆ ปรับจากสถานะปัจจุบัน

## ปุ่มเล็ก ๆ ที่บอกเส้นแบ่งของผลิตภัณฑ์

การที่ Retry แบบเดิมไม่เด่นใน Codex ไม่ได้แปลว่าความสามารถหนึ่งถูกตัดออก

มันบอกเส้นแบ่งที่ AI กำลังเปลี่ยนจาก **เครื่องตอบคำถาม** ไปเป็น **ระบบที่ลงมือทำ**

เมื่อ AI สร้างเพียงข้อความ ประวัติอาจถูกแทนที่ได้ แต่เมื่อ AI เริ่มลงมือทำ ประวัติจะกลายเป็นส่วนหนึ่งของสถานะระบบ

คำถามสำคัญจึงไม่ใช่แค่ “ตอบใหม่ได้ไหม” แต่เป็น:

- เราเห็นได้หรือไม่ว่ารอบก่อนทำอะไรไปแล้ว?
- เก็บส่วนที่ถูกต้องและแก้เฉพาะส่วนที่ผิดได้หรือไม่?
- แตกแขนงจากสถานะที่ระบุชัดเจนได้หรือไม่?
- ป้องกันเครื่องมือและการกระทำภายนอกจากการทำงานซ้ำได้หรือไม่?
- ติดตามความสัมพันธ์เชิงเหตุและผลของแต่ละรอบได้หรือไม่?

> การเลือนหายของปุ่ม Retry ไม่ใช่เพียงการหายไปของการโต้ตอบที่คุ้นเคย เมื่อ AI เปลี่ยนโลกจริงได้ คำว่า “ลองอีกครั้ง” ต้องตอบคำถามที่จริงจังกว่านั้น: จะเริ่มจากสถานะใด และจะทำการกระทำใดซ้ำ?
