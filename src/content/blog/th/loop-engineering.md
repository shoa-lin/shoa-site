**Loop engineering คือการเลิกทำหน้าที่เป็นคนเขียนพรอมป์ให้ AI Agent ด้วยตัวเอง แล้วออกแบบระบบให้ทำหน้าที่นั้นแทน** ลูปในที่นี้มองได้ว่าเป็นเป้าหมายแบบวนซ้ำ: คุณกำหนดจุดประสงค์ แล้ว AI ปรับแก้ไปเรื่อย ๆ จนเสร็จ ผมคิดว่านี่อาจเป็นอนาคตของวิธีทำงานร่วมกับ AI Agent สำหรับเขียนโค้ด แต่ยังอยู่ในช่วงเริ่มต้น ผมยังคงตั้งข้อสงสัย และคุณ**ต้อง**[ระมัดระวัง](https://x.com/weswinder/status/2063700289710964906)เรื่องต้นทุนโทเคนอย่างยิ่ง เพราะรูปแบบการใช้งานเปลี่ยนไปมากตามงบโทเคนที่มีอยู่ ดังนั้นผมจึงอยากแจกแจงว่า loop engineering คืออะไรและมีความหมายอย่างไร

Peter Steinberger เพิ่ง[กล่าวไว้](https://x.com/steipete/status/2063697162748260627)ว่า: "คุณไม่ควรเขียนพรอมป์ให้ AI Agent สำหรับเขียนโค้ดด้วยตัวเองอีกต่อไป แต่ควรออกแบบลูปที่เขียนพรอมป์ให้ AI Agent ของคุณ" ในทำนองเดียวกัน Boris Cherny หัวหน้าทีม Claude Code ที่ Anthropic [กล่าวว่า](https://x.com/rohanpaul_ai/status/2063289804708835412): "ผมไม่เขียนพรอมป์ให้ Claude แล้ว ผมมีลูปที่ทำงานอยู่ คอยเขียนพรอมป์ให้ Claude และหาว่าควรทำอะไร งานของผมคือเขียนลูป"

แล้วทั้งหมดนี้หมายความว่าอย่างไรกันแน่?

ตลอดประมาณสองปีที่ผ่านมา การได้ผลลัพธ์ที่มีประโยชน์จาก AI Agent สำหรับเขียนโค้ดหมายถึงการเขียนพรอมป์ที่ดีและให้บริบทมากพอ คุณพิมพ์บางอย่าง อ่านสิ่งที่ตอบกลับมา แล้วพิมพ์สิ่งถัดไป AI Agent เป็นเครื่องมือที่คุณถือควบคุมตลอดกระบวนการ ทีละรอบต่อเนื่องกัน ช่วงเวลานั้นโดยพื้นฐานอาจจบลงแล้ว หรืออย่างน้อยบางคนก็คิดเช่นนั้น

ตอนนี้คุณสร้างระบบขนาดเล็กที่ค้นหางาน มอบหมาย ตรวจสอบ บันทึกสิ่งที่เสร็จแล้ว และตัดสินว่าขั้นต่อไปคืออะไร จากนั้นปล่อยให้ระบบขับเคลื่อน AI Agent แทนคุณ ผมเคยเขียนถึงแนวคิดใกล้เคียงสองเรื่อง: [agent harness engineering](https://addyosmani.com/blog/agent-harness-engineering/) ซึ่งออกแบบสภาพแวดล้อมที่ AI Agent หนึ่งตัวทำงานอยู่ภายใน และ [factory model](https://addyosmani.com/blog/factory-model/) ซึ่งเป็นระบบที่สร้างซอฟต์แวร์ Loop engineering อยู่เหนือ harness ขึ้นไปหนึ่งชั้น มันคือ harness ที่ทำงานตามเวลา สร้างผู้ช่วย และป้อนผลลัพธ์กลับเข้าไปในรอบถัดไป

สิ่งที่ทำให้ผมประหลาดใจคือ เรื่องนี้ไม่ใช่ประเด็นระดับเครื่องมืออีกต่อไป หนึ่งปีก่อน หากคุณต้องการลูป คุณต้องเขียนสคริปต์ bash กองใหญ่ ดูแลมันไปเรื่อย ๆ และครอบครองระบบเฉพาะกิจที่ไม่มีใครอื่นใช้ แต่ตอนนี้ชิ้นส่วนเหล่านี้มากับตัวผลิตภัณฑ์ รายการของ Steinberger ตรงกับ Codex app แทบทั้งหมด และตรงกับ Claude Code เกือบเท่า ๆ กัน เมื่อเห็นว่าโครงสร้างเหมือนกัน คุณจะเลิกถกเถียงเรื่องเครื่องมือ แล้วหันมาออกแบบลูปที่ทำงานได้ไม่ว่าจะอยู่ในผลิตภัณฑ์ใด

## องค์ประกอบ 5 ส่วน และข้อสังเกตเรื่องสถานะ

[ลูป](https://x.com/reach_vb/status/2063713960495558940) ต้องมีองค์ประกอบ 5 อย่างและที่เก็บสถานะ ก่อนจะเทียบรายละเอียดของแต่ละเครื่องมือ มาดูรายการทั้งหมดก่อน

1. **Automations** ที่ทำงานตามตาราง และจัดการการค้นหาและคัดแยกงานได้เอง
2. **Worktrees** เพื่อให้ AI Agent สองตัวที่ทำงานขนานกันไม่แก้ไฟล์ทับกัน
3. **Skills** ที่บันทึกความรู้ของโปรเจกต์ซึ่ง AI Agent จะต้องเดาเองหากไม่มีข้อมูลนี้
4. **Plugins และ connectors** ที่เชื่อม AI Agent เข้ากับเครื่องมือที่คุณใช้อยู่
5. **Sub-agents** เพื่อให้ตัวหนึ่งเสนอวิธีแก้ และอีกตัวตรวจสอบ

จากนั้นยังมีองค์ประกอบที่หก: หน่วยความจำ อาจเป็นไฟล์ Markdown, บอร์ด Linear หรือสิ่งอื่นใดที่อยู่ได้นานกว่าบทสนทนาหนึ่งครั้ง และบันทึกว่างานใดเสร็จแล้วกับควรทำอะไรต่อ ฟังดูเรียบง่ายจนเหมือนไม่สำคัญ แต่นี่คือเทคนิคเดียวกับที่ AI Agent ระยะยาวทุกตัวพึ่งพา ดังที่ผมอธิบายไว้ใน [long-running agents](https://addyosmani.com/blog/long-running-agents/): โมเดลจะลืมทุกอย่างระหว่างการรัน หน่วยความจำจึงต้องอยู่บนดิสก์ ไม่ใช่อยู่ในบริบท AI Agent ลืม แต่ repo ไม่ลืม

ปัจจุบันทั้งสองผลิตภัณฑ์มีองค์ประกอบครบทั้ง 5 ส่วน

| Primitive | หน้าที่ในลูป | Codex app | Claude Code |
| --- | --- | --- | --- |
| **Automations** | ค้นหา + คัดแยกตามตาราง | [Automations tab](https://developers.openai.com/codex/app/automations): เลือกโปรเจกต์ พรอมป์ ความถี่ และสภาพแวดล้อม; ผลลัพธ์เข้าสู่ Triage inbox; `/goal` ทำงานต่อจนเสร็จ | Scheduled tasks และ cron, `/loop`, `/goal`, hooks, GitHub Actions |
| **Worktrees** | แยก feature ที่ทำขนานกัน | worktree ในตัวสำหรับแต่ละ thread | `git worktree`, `--worktree` และ `isolation: worktree` บน subagent |
| **Skills** | จัดเก็บความรู้โปรเจกต์ให้เป็นระบบ | [Agent Skills](https://developers.openai.com/codex/skills) (`SKILL.md`) เรียกด้วย `$name` หรือเริ่มโดยอัตโนมัติเมื่อเงื่อนไขตรง | [Agent Skills](https://addyosmani.com/blog/agent-skills/) (`SKILL.md`) |
| **Plugins / connectors** | เชื่อมต่อเครื่องมือของคุณ | Connectors (MCP) รวมกับ plugins สำหรับการแจกจ่าย | MCP servers รวมกับ plugins |
| **Sub-agents** | พัฒนาและตรวจสอบวิธีแก้ | [Subagents](https://developers.openai.com/codex/subagents) กำหนดเป็น TOML ใน `.codex/agents/` | Subagents ใน `.claude/agents/` รวมถึง agent teams |
| **State** | ติดตามงานที่เสร็จแล้ว | Markdown หรือ Linear ผ่าน connector | Markdown (`AGENTS.md`, progress files) หรือ Linear ผ่าน MCP |

ชื่อเรียกแตกต่างกันเล็กน้อย แต่ความสามารถเหมือนกัน รายละเอียดเหล่านี้สำคัญ เพราะเป็นตัวกำหนดว่าลูปจะทำงานประสานกันได้ดี หรือค่อย ๆ รั่วไหลจนควบคุมไม่ได้

## Automations: จังหวะหัวใจของลูป

Automations คือสิ่งที่ทำให้ลูปเป็นลูปจริง แทนที่จะเป็นการรันครั้งเดียว ใน Codex app คุณสร้าง automation จาก Automations tab เลือกโปรเจกต์ พรอมป์ ความถี่ และว่าจะรันกับ checkout ในเครื่องหรือ background worktree การรันที่พบสิ่งต้องจัดการจะเข้าสู่ Triage inbox ส่วนการรันที่ไม่พบอะไรจะ archive ตัวเอง OpenAI ใช้ automations ภายในกับงานประจำ เช่น คัดแยก issue รายวัน สรุป CI ที่ล้มเหลว เขียนสรุป commit และตามหา bug ที่เกิดขึ้นในสัปดาห์ก่อน Automation ยังเรียก skill ได้ ทำให้พฤติกรรมที่นำกลับมาใช้ซ้ำดูแลรักษาได้ง่าย: เรียก `$skill-name` แทนการวางคำสั่งยาวเป็นกำแพงไว้ในตารางเวลาที่ไม่มีใครปรับปรุง

Claude Code ไปถึงจุดหมายเดียวกันผ่านการตั้งเวลาและ hooks คุณใช้ `/loop` รันพรอมป์หรือคำสั่งตามช่วงเวลา ตั้ง cron tasks ใช้ hooks เรียก shell command ในจังหวะเฉพาะของวงจรชีวิต AI Agent หรือย้ายทั้งกระบวนการไป GitHub Actions เพื่อให้ทำงานต่อหลังปิดแล็ปท็อปได้ แนวคิดเหมือนกัน: กำหนดงานอัตโนมัติ กำหนดความถี่ แล้วปล่อยให้สิ่งที่ค้นพบมาหาคุณ แทนที่คุณจะตรวจทุกระบบด้วยตัวเอง

ยังมี primitive ภายใน session ที่เข้าใกล้แก่นของบทความนี้มากขึ้น `/loop` ทำงานซ้ำตามจังหวะเวลา ส่วน `/goal` ทำงานต่อจนเงื่อนไขที่คุณเขียนเป็นจริง หลังแต่ละรอบ โมเดลขนาดเล็กอีกตัวจะตรวจว่าเงื่อนไขสำเร็จหรือยัง ดังนั้น AI Agent ที่เขียนโค้ดจึงไม่ได้ให้คะแนนตัวเอง กำหนดเงื่อนไขอย่าง "การทดสอบทั้งหมดใน test/auth ผ่านและ lint สะอาด" แล้วเดินออกไป Codex มี primitive เดียวกันและเรียกว่า `/goal` เช่นกัน มันทำงานต่อข้ามหลายรอบจนกว่าเงื่อนไขหยุดที่ตรวจสอบได้จะสำเร็จ พร้อมรองรับ pause, resume และ clear primitive เดียวกันมีอยู่ในเครื่องมือทั้งสอง ซึ่งเป็นรูปแบบที่พบตลอดบทความนี้

นั่นคือส่วนที่นำงานขึ้นมาให้เห็น ส่วนที่เหลือของลูปจะลงมือจัดการงานนั้น

## Worktrees ป้องกันไม่ให้งานขนานกลายเป็นความวุ่นวาย

ทันทีที่คุณรัน AI Agent มากกว่าหนึ่งตัว ไฟล์ก็เริ่มชนกัน AI Agent สองตัวที่แก้ไฟล์เดียวกันสร้างปัญหาแบบเดียวกับวิศวกรสองคนแก้บรรทัดเดียวกันโดยไม่ประสานงาน git worktree แก้ปัญหาเชิงกลนี้: มันคือ working directory อิสระบน branch ของตัวเองที่ใช้ประวัติ repository ร่วมกัน ดังนั้นการแก้ไขของ AI Agent ตัวหนึ่งจะไม่แตะ checkout ของอีกตัว

Codex มี worktree อยู่ในตัว app ทำให้หลาย thread ทำงานบน repository เดียวกันได้โดยไม่แก้ checkout ของกันและกัน Claude Code ให้การแยกแบบเดียวกันผ่าน `git worktree`, flag `--worktree` ที่เปิด session ใน checkout ของตัวเอง และการตั้งค่า `isolation: worktree` สำหรับ subagent ซึ่งมอบ checkout ใหม่ให้ผู้ช่วยแต่ละตัวและล้างเมื่อใช้งานเสร็จ ผมเคยเขียนเรื่องด้านมนุษย์ไว้ใน [the orchestration tax](https://addyosmani.com/blog/orchestration-tax/): worktrees ขจัดการชนกันเชิงกล แต่**คุณ**ยังคงเป็นเพดาน แบนด์วิดท์ในการ review ของคุณ ไม่ใช่เครื่องมือ เป็นตัวกำหนดว่าคุณรัน AI Agent ได้จริงกี่ตัว

## Skills ช่วยให้ไม่ต้องอธิบายโปรเจกต์ซ้ำทุกครั้ง

Skill ช่วยให้คุณไม่ต้องอธิบายบริบทโปรเจกต์เดิมใหม่ในทุก session เครื่องมือทั้งสองใช้รูปแบบเดียวกัน: โฟลเดอร์ที่มี `SKILL.md` สำหรับคำสั่งและ metadata พร้อมสคริปต์ เอกสารอ้างอิง และ asset ที่เพิ่มได้ Codex รัน skill เมื่อคุณเรียกด้วย `$` หรือ `/skills` หรือเริ่มอัตโนมัติเมื่องานตรงกับคำอธิบายของ skill นี่คือเหตุผลที่คำอธิบายสั้น กระชับ และตรงความหมาย ดีกว่าคำอธิบายที่พยายามเล่นคำ Claude Code ทำงานแบบเดียวกัน ดังที่ผมอธิบายใน [agent skills](https://addyosmani.com/blog/agent-skills/)

Skills ยังช่วยลดต้นทุนจากการต้องอธิบายเจตนาเดิมซ้ำ ๆ ใน [intent debt](https://addyosmani.com/blog/intent-debt/) ผมเสนอว่า AI Agent เริ่มทุก session โดยไม่มีบริบท และเติมช่องว่างในเจตนาของคุณด้วยการคาดเดาอย่างมั่นใจ Skill ทำให้เจตนานั้นอยู่ภายนอกโมเดลในรูปของข้อตกลง ขั้นตอน build และหมายเหตุอย่าง "เราไม่ทำแบบนี้เพราะเหตุการณ์ครั้งนั้น" เขียนครั้งเดียว แล้ว AI Agent อ่านทุกครั้งที่รัน หากไม่มี skills ลูปต้องอนุมานโปรเจกต์ใหม่จากศูนย์ในทุก cycle แต่เมื่อมี skills ความรู้โปรเจกต์จะสะสมข้าม cycle ได้

มีข้อแตกต่างสำคัญหนึ่งอย่าง: skill คือรูปแบบสำหรับเขียนเนื้อหา ส่วน plugin คือวิธีแจกจ่าย เมื่อคุณต้องการแชร์ skill ข้าม repository หรือรวมหลาย skill ไว้ด้วยกัน คุณจะจัดแพ็กเกจเป็น plugin ทั้ง Codex และ Claude Code เป็นเช่นนี้

## Plugins และ connectors เชื่อมลูปเข้ากับเครื่องมือจริง

ลูปที่เห็นได้เฉพาะระบบไฟล์เป็นลูปที่มีขอบเขตเล็กมาก Connectors ซึ่งสร้างบน MCP ช่วยให้ AI Agent อ่าน issue tracker, query ฐานข้อมูล, เรียก staging API หรือส่งข้อความใน Slack ได้ Codex และ Claude Code รองรับ MCP ทั้งคู่ ดังนั้น connector ที่เขียนให้ระบบหนึ่งมักใช้กับอีกระบบได้ Plugins รวม connectors กับ skills เข้าด้วยกัน ทำให้เพื่อนร่วมทีมติดตั้งชุดตั้งค่าทั้งหมดในครั้งเดียว แทนการประกอบใหม่จากความจำ

นี่คือความแตกต่างระหว่าง AI Agent ที่พูดว่า "นี่คือวิธีแก้" กับลูปที่เปิด PR เชื่อม Linear ticket และแจ้งใน channel เมื่อ CI เป็นสีเขียว Connectors ทำให้ลูปลงมือในสภาพแวดล้อมจริงของคุณ แทนที่จะเพียงอธิบายว่าจะทำอะไรหากเข้าถึงระบบได้

## Sub-agents แยกผู้สร้างออกจากผู้ตรวจ

เทคนิคเชิงโครงสร้างที่มีประโยชน์ที่สุดในลูป คือแยก AI Agent ที่เขียนออกจาก AI Agent ที่ตรวจ โมเดลมักใจดีกับผลงานตัวเองมากเกินไป AI Agent ตัวที่สองซึ่งมีคำสั่งต่างกัน และบางครั้งใช้โมเดลต่างกัน จะจับสิ่งที่ตัวแรกโน้มน้าวตัวเองให้ยอมรับได้

Codex สร้าง subagents เมื่อคุณสั่ง รันพร้อมกัน แล้วรวมผลกลับเป็นคำตอบเดียว คุณกำหนด custom agents เป็นไฟล์ TOML ใน `.codex/agents/` โดยแต่ละตัวมีชื่อ คำอธิบาย คำสั่ง และโมเดลกับระดับ reasoning effort ที่เลือกได้ นั่นหมายความว่า security reviewer ใช้โมเดลที่แข็งแรงและ effort สูงได้ ขณะที่ explorer ใช้ตัวที่เร็วและอ่านอย่างเดียว Claude Code ทำงานแบบเดียวกันผ่าน subagents ใน `.claude/agents/` และ agent teams ที่ส่งต่องานระหว่างกัน การแบ่งหน้าที่ทั่วไปในเครื่องมือทั้งสองคือ ให้ตัวหนึ่งสำรวจ ตัวหนึ่งลงมือ และอีกตัวตรวจสอบตามข้อกำหนด

ผมเสนอแนวคิดนี้มาแล้วสองครั้ง: ครั้งแรกใน [the code agent orchestra](https://addyosmani.com/blog/code-agent-orchestra/) และอีกครั้งใน [agentic code review](https://addyosmani.com/blog/agentic-code-review/) เรื่องนี้สำคัญเป็นพิเศษในลูป เพราะลูปทำงานตอนที่คุณไม่ได้เฝ้าดู ผู้ตรวจที่คุณเชื่อถือได้จริงคือสิ่งที่ทำให้คุณเดินออกไปได้ Subagents ใช้โทเคนมากขึ้น เพราะแต่ละตัวทำงานกับโมเดลและเครื่องมือของตัวเอง จึงควรใช้โทเคนเหล่านี้ในจุดที่ความเห็นที่สองคุ้มค่า นี่คือโครงสร้างเบื้องหลัง `/goal` ของ Claude Code เช่นกัน: โมเดลใหม่เป็นผู้ตัดสินว่าลูปเสร็จหรือยัง แทนโมเดลที่ทำงาน กล่าวคือใช้การแยกผู้สร้างและผู้ตรวจแม้กระทั่งกับเงื่อนไขหยุด

## ลูปมีหน้าตาอย่างไร

เมื่อนำชิ้นส่วนมารวมกัน thread เดียวจะกลายเป็นแผงควบคุมขนาดเล็ก นี่คือรูปแบบหนึ่งที่ผมใช้เป็นประจำ

Automation ทำงานกับ repository ทุกเช้า พรอมป์ของมันเรียก triage skill ที่อ่าน CI failures จากเมื่อวาน, issue ที่ยังเปิดอยู่ และ commit ล่าสุด แล้วเขียนสิ่งที่พบลงไฟล์ Markdown หรือบอร์ด Linear สำหรับทุกประเด็นที่ควรดำเนินการ thread จะเปิด worktree แยก ส่ง subagent ตัวหนึ่งร่างวิธีแก้ และส่งอีกตัว review ร่างนั้นเทียบกับ skill ของโปรเจกต์และการทดสอบที่มีอยู่

Connectors ช่วยให้ลูปเปิด PR และอัปเดต ticket สิ่งที่ลูปจัดการไม่ได้จะเข้าสู่ triage inbox ของผม ไฟล์สถานะคือแกนหลักของระบบ: มันจำว่าได้ลองอะไร อะไรผ่าน และอะไรยังค้างอยู่ เพื่อให้การรันเช้าวันถัดไปทำต่อจากจุดที่วันนี้หยุด

ลองดูว่าคุณทำอะไรจริง ๆ คุณออกแบบกระบวนการเพียงครั้งเดียว ไม่ได้เขียนพรอมป์ให้แต่ละขั้นตอน นั่นคือประเด็นของ Steinberger ที่กลายเป็นรูปธรรม และเป็นลูปเดียวกันทั้งใน Codex และ Claude Code เพราะองค์ประกอบเหมือนกัน

## สิ่งที่ลูปยังทำแทนคุณไม่ได้

ลูปเปลี่ยนรูปแบบงาน แต่ไม่ได้ตัดคุณออกจากงาน ปัญหา 3 ข้อต่อไปนี้ยิ่งชัดขึ้นเมื่อลูปดีขึ้น ไม่ได้ง่ายลง

**การตรวจสอบยังคงเป็นความรับผิดชอบของคุณ** ลูปที่ทำงานโดยไม่มีคนเฝ้าก็ทำผิดโดยไม่มีคนเห็นได้ เหตุผลที่แยก subagent ผู้ตรวจออกจากผู้สร้าง คือเพิ่มน้ำหนักให้คำกล่าวว่า "เสร็จแล้ว" ของลูป ถึงอย่างนั้น คำว่าเสร็จก็ยังเป็นคำกล่าว ไม่ใช่หลักฐาน ผมย้ำประโยคเดิมจาก [code review in the age of AI](https://addyosmani.com/blog/code-review-ai/) อยู่เสมอ: งานของคุณคือส่งมอบโค้ดที่คุณยืนยันแล้วว่าใช้งานได้

**ความเข้าใจของคุณยังเสื่อมลงได้หากปล่อยไว้** ยิ่งลูปส่งโค้ดที่คุณไม่ได้เขียนได้เร็ว ช่องว่างระหว่างระบบที่มีอยู่จริงกับระบบที่คุณเข้าใจก็ยิ่งกว้าง นี่คือ [comprehension debt](https://addyosmani.com/blog/comprehension-debt/) หรือหนี้ด้านความเข้าใจ หากคุณไม่อ่านสิ่งที่ลูปสร้างอย่างรอบคอบ ลูปที่ราบรื่นจะเพียงทำให้หนี้ก้อนนี้โตเร็วขึ้น

**ท่าทีที่สบายเกินไปเป็นอันตราย** เมื่อลูปทำงานด้วยตัวเอง เราอาจหยุดใช้วิจารณญาณและยอมรับทุกอย่างที่ส่งกลับมา ผมเรียกสิ่งนี้ว่า [cognitive surrender](https://addyosmani.com/blog/cognitive-surrender/) หรือการยอมสละวิจารณญาณ การออกแบบลูปโดยใช้ดุลยพินิจอาจเป็นทางแก้ แต่การออกแบบเพื่อหลีกเลี่ยงการคิดจะเร่งปัญหา การกระทำดูเหมือนกัน แต่ผลลัพธ์ตรงข้ามกัน

## สร้างลูป และยังคงเป็นวิศวกร

ผมคิดว่านี่คือภาพล่วงหน้าของวิวัฒนาการในการทำงานของเรา อย่างไรก็ตาม หากผมหยุด review โค้ดด้วยตัวเอง หรือพึ่งลูปอัตโนมัติทั้งหมดในการแก้ปัญหา คุณภาพผลิตภัณฑ์ของผมจะลดลง และน่าจะลงเอยในวงจรถดถอยที่ขุดหลุมให้ลึกขึ้นเรื่อย ๆ

ดังนั้นสร้างลูปต่อไปได้ แต่อย่าลืมว่าการเขียนพรอมป์ให้ AI Agent โดยตรงยังคงได้ผล เป้าหมายคือหาสมดุลที่เหมาะสม

ลูปเดียวกันอาจให้ผลลัพธ์ต่างกันมากตามผู้ใช้ คนสองคนสร้างลูปเหมือนกันทุกประการแต่ได้ผลตรงข้าม คนหนึ่งใช้มันเพื่อเดินหน้าเร็วขึ้นในงานที่ตัวเองเข้าใจลึกซึ้ง อีกคนใช้มันเพื่อหลีกเลี่ยงการเข้าใจงาน ลูปไม่รู้ความแตกต่าง แต่คุณรู้

นี่คือเหตุผลที่การออกแบบลูปยากกว่าวิศวกรรมพรอมป์ ไม่ได้ง่ายกว่า ประเด็นของ Cherny ไม่ใช่ว่างานง่ายลง แต่คือจุดที่สร้างแรงทวีคูณย้ายไป

สร้างลูป แต่สร้างในฐานะคนที่ตั้งใจจะยังคงเป็นวิศวกร ไม่ใช่คนที่เพียงกดเริ่มทำงาน
