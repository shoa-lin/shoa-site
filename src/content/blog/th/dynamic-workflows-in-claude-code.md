เมื่อสัปดาห์ที่แล้ว เราเปิดตัว [เวิร์กโฟลว์แบบไดนามิก](https://code.claude.com/docs/en/workflows) ใน Claude Code ตอนนี้ Claude สามารถเขียน [harness](https://code.claude.com/docs/en/glossary#agentic-harness) ของตัวเองได้ทันที โดยปรับให้เหมาะกับงานตรงหน้าโดยเฉพาะ

> **Harness** หมายถึงชั้นควบคุมที่อยู่รอบโมเดล AI ซึ่งครอบคลุมการประกอบพรอมป์ การประสานเครื่องมือ การจัดการบริบท และการกู้คืนจากข้อผิดพลาด เราสามารถมอง Claude Code เป็น **Model + Harness** ได้ บทความนี้จะใช้คำว่า "harness" ในภาษาอังกฤษต่อไป

Harness เริ่มต้นของ Claude Code สร้างขึ้นสำหรับงานเขียนโค้ด แต่ก็มีประโยชน์กับงานอื่นอีกมาก เพราะเมื่อพิจารณาให้ลึกลงไป งานจำนวนมากมีลักษณะคล้ายงานเขียนโค้ด อย่างไรก็ตาม งานบางประเภทจำเป็นต้องมี harness เฉพาะทางซ้อนอยู่บน Claude Code เพื่อให้ได้ประสิทธิภาพสูงสุด เช่น [Research](https://support.claude.com/en/articles/11088861-using-research-on-claude), [การวิเคราะห์ความปลอดภัย](https://support.claude.com/en/articles/11932705-automated-security-reviews-in-claude-code), [agent teams](https://code.claude.com/docs/en/agent-teams) และ [Code Review](https://code.claude.com/docs/en/code-review)

เวิร์กโฟลว์ช่วยให้ Claude สร้าง harness เฉพาะงานเหล่านี้บน Claude Code แบบไดนามิก และยังสามารถบันทึก แชร์ และนำกลับมาใช้ซ้ำได้

บทความนี้สรุปประสบการณ์และบทเรียนช่วงแรกจากการใช้เวิร์กโฟลว์ แนวทางปฏิบัติที่ดียังอยู่ระหว่างการพัฒนา โดยเวิร์กโฟลว์แบบไดนามิกมักใช้โทเคนมากกว่า และเหมาะที่สุดกับงานซับซ้อนที่มีมูลค่าสูง

## ตัวอย่างพรอมป์

ก่อนลงรายละเอียดทางเทคนิค ต่อไปนี้คือตัวอย่างพรอมป์ที่แสดงให้เห็นขอบเขตการใช้งานที่เป็นไปได้:

"การทดสอบนี้ล้มเหลวประมาณ 1 ครั้งในทุก 50 รอบ ช่วยสร้างเวิร์กโฟลว์เพื่อทำให้เกิดซ้ำ ตั้งสมมติฐานที่แข่งขันกันเกี่ยวกับภาวะแข่งขัน (race condition) และอย่าหยุดจนกว่าจะเหลือสมมติฐานเดียวที่ผ่านการพิสูจน์ด้วยหลักฐาน"

"ใช้เวิร์กโฟลว์ตรวจสอบ 50 sessions ล่าสุดของฉัน หาสิ่งที่ฉันแก้ซ้ำ ๆ แล้วเปลี่ยนประเด็นที่เกิดซ้ำให้เป็นกฎใน `CLAUDE.md`"

"ใช้เวิร์กโฟลว์สำรวจ #incidents ใน Slack ย้อนหลังหกเดือน และหารากสาเหตุที่เกิดซ้ำแต่ยังไม่มีใครเปิด ticket"

"นำแผนธุรกิจของฉันไปรันเวิร์กโฟลว์ โดยให้ agents แต่ละตัววิจารณ์อย่างเข้มข้นจากมุมมองของนักลงทุน ลูกค้า และคู่แข่ง"

"นี่คือโฟลเดอร์ที่มีเรซูเม่ 80 ฉบับ ใช้เวิร์กโฟลว์จัดอันดับสำหรับตำแหน่ง backend และตรวจสอบสิบอันดับแรกซ้ำ สัมภาษณ์ฉันด้วยเครื่องมือ AskUserQuestion เพื่อกำหนด rubric"

"ฉันต้องการชื่อสำหรับเครื่องมือ CLI นี้ ใช้เวิร์กโฟลว์ระดมตัวเลือกให้หลากหลาย แล้วรัน tournament เพื่อเลือกสามอันดับแรก"

"ใช้เวิร์กโฟลว์เปลี่ยนชื่อ User model เป็น Account ในทุกจุด"

"ตรวจร่างบทความบล็อกของฉัน แล้วใช้เวิร์กโฟลว์เทียบกับ codebase เพื่อยืนยันข้ออ้างทางเทคนิคทุกข้อ ฉันไม่ต้องการเผยแพร่ข้อมูลที่ผิด"

## เวิร์กโฟลว์แบบไดนามิกทำงานอย่างไร

เวิร์กโฟลว์แบบไดนามิกจะรันไฟล์ JavaScript ที่มีฟังก์ชันพิเศษบางอย่างสำหรับสร้างและประสานงาน [subagents](https://code.claude.com/docs/en/sub-agents):

![แผนภาพแสดงเวิร์กโฟลว์แบบไดนามิกที่สร้างและประสานงาน subagents](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f1684f559cc83ff4b465b_image1.png)

นอกจากนี้ยังมีออบเจ็กต์ JavaScript มาตรฐาน เช่น JSON, Math และ Array สำหรับประมวลผลข้อมูล

เวิร์กโฟลว์แบบไดนามิกสามารถเลือกได้ว่า agent จะใช้โมเดลใด และ subagent จะทำงานใน worktree แยกของตัวเองหรือไม่ ทำให้ Claude เลือกระดับความสามารถและการแยกสภาพแวดล้อมที่เหมาะกับแต่ละขั้นตอนได้

หากเวิร์กโฟลว์ถูกขัดจังหวะจากการกระทำของผู้ใช้หรือการออกจากเทอร์มินัล เมื่อกลับมาใช้งาน session ต่อ ก็สามารถทำงานต่อจากจุดที่หยุดไว้ได้

## เหตุผลที่ควรใช้เวิร์กโฟลว์แบบไดนามิก

Harness เริ่มต้นของ Claude Code ต้องวางแผนและลงมือทำงานภายใน context window เดียว วิธีนี้ทำงานได้ดีมากกับงานเขียนโค้ดหลายประเภท แต่อาจเริ่มมีข้อจำกัดเมื่อต้องรับมือกับงานที่ใช้เวลานาน ทำงานขนานจำนวนมาก มีโครงสร้างซับซ้อนสูง หรือต้องตรวจสอบแบบเป็นปฏิปักษ์

ยิ่ง Claude ทำงานซับซ้อนใน context window เดียวนานเท่าไร ก็ยิ่งเสี่ยงต่อรูปแบบความล้มเหลวหลายประการ:

- **Agentic laziness**: Claude หยุดก่อนทำงานซับซ้อนที่มีหลายส่วนเสร็จ และประกาศว่าสำเร็จหลังทำได้เพียงบางส่วน เช่น จัดการข้อค้นพบจาก security review เพียง 35 ข้อจาก 50 ข้อ
- **Self-preferential bias**: Claude มีแนวโน้มให้ความสำคัญกับผลลัพธ์ของตัวเอง โดยเฉพาะเมื่อถูกขอให้ตรวจสอบหรือตัดสินผลลัพธ์นั้นตาม rubric
- **Goal drift**: ความยึดมั่นต่อวัตถุประสงค์เดิมค่อย ๆ ลดลงตลอดหลายรอบสนทนา โดยเฉพาะหลังการ compaction การสรุปแต่ละครั้งทำให้ข้อมูลสูญหาย ดังนั้นข้อกำหนดเฉพาะกรณีและข้อจำกัดอย่าง "อย่าทำ X" อาจหายไปได้

เวิร์กโฟลว์รับมือกับปัญหาเหล่านี้ด้วยการประสาน Claude subagents แยกกัน แต่ละตัวมี context window ของตัวเองและมีเป้าหมายที่เฉพาะเจาะจงและแยกจากกัน

## เวิร์กโฟลว์แบบไดนามิกเทียบกับเวิร์กโฟลว์แบบคงที่

คุณอาจเคยสร้างเวิร์กโฟลว์แบบคงที่ด้วย Claude Agent SDK หรือ `claude -p` เพื่อประสาน Claude Code หลาย instances มาแล้ว

เนื่องจากเวิร์กโฟลว์แบบคงที่ต้องคำนึงถึง edge cases ทุกแบบล่วงหน้า จึงมักออกมาเป็นระบบที่ค่อนข้างทั่วไป แต่เมื่อใช้ [Claude Opus 4.8](https://www.anthropic.com/news/claude-opus-4-8) ร่วมกับเวิร์กโฟลว์แบบไดนามิก Claude สามารถเขียน harness ที่ปรับเฉพาะให้ตรงกับกรณีใช้งานปัจจุบันได้

## รูปแบบที่มีประโยชน์สำหรับเวิร์กโฟลว์แบบไดนามิก

คุณเริ่มต้นได้ง่าย ๆ เพียงขอให้ Claude สร้างเวิร์กโฟลว์แบบไดนามิก หรือใช้คำกระตุ้น `ultracode` เพื่อระบุเจตนาให้ชัดเจน

การมีกรอบความคิดเกี่ยวกับรูปแบบที่พบบ่อยจะช่วยให้คุณมองออกว่าเวิร์กโฟลว์มีประโยชน์เมื่อใด และควรชี้นำ Claude ผ่านพรอมป์อย่างไร

Claude สามารถใช้และผสมผสานรูปแบบต่อไปนี้:

![ภาพรวมรูปแบบเวิร์กโฟลว์แบบไดนามิกที่พบบ่อย](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f16d86247e586b929a407_image10.png)

### จำแนกแล้วลงมือทำ

ใช้ classifier agent ระบุประเภทของงาน แล้วส่งต่อไปยัง agent หรือพฤติกรรมที่แตกต่างกัน นอกจากนี้ยังรัน classifier ในตอนท้ายเพื่อกำหนดวิธีจัดการผลลัพธ์ได้ด้วย

### กระจายงานแล้วสังเคราะห์

แบ่งงานเป็นขั้นตอนย่อย ให้ agent ทำงานในแต่ละขั้น แล้วสังเคราะห์ผลลัพธ์ วิธีนี้มีประโยชน์อย่างยิ่งเมื่องานมีหลายขั้น หรือแต่ละขั้นได้ประโยชน์จาก context window ที่สะอาดเพื่อป้องกันการรบกวนและการปะปนข้ามงาน ขั้นตอนสังเคราะห์ทำหน้าที่เป็นจุดกั้น โดยรอให้ fan-out agents ทุกตัวทำงานเสร็จ แล้วจึงรวมผลลัพธ์แบบมีโครงสร้างเป็นผลลัพธ์เดียว

### การตรวจสอบแบบเป็นปฏิปักษ์

สำหรับ agent ทุกตัวที่สร้างผลลัพธ์ ให้สร้าง agent อิสระอีกตัวเพื่อโต้แย้งและตรวจสอบผลลัพธ์นั้นตาม rubric หรือเกณฑ์ที่ระบุไว้อย่างชัดเจน

### สร้างแล้วคัดกรอง

สร้างแนวคิดจำนวนมาก คัดกรองด้วย rubric หรือขั้นตอนตรวจสอบ ตัดรายการซ้ำ และส่งกลับเฉพาะตัวเลือกที่แข็งแรงที่สุดซึ่งผ่านการทดสอบแล้ว

### Tournament

แทนที่จะแบ่งงาน ให้ agents แข่งขันกัน สร้าง agents จำนวน N ตัวเพื่อทดลองทำงานเดียวกันด้วยแนวทางที่แตกต่างกัน แล้วใช้ judge agent หรือโมเดลเปรียบเทียบผลลัพธ์แบบจับคู่จนเหลือผู้ชนะเพียงหนึ่งเดียว

### วนซ้ำจนกว่าจะเสร็จ

สำหรับงานที่ไม่ทราบปริมาณล่วงหน้า ให้สร้าง agents ต่อไปจนกว่าจะตรงตามเงื่อนไขหยุด เช่น ไม่มีข้อค้นพบใหม่หรือไม่มีข้อผิดพลาดเหลืออยู่ใน logs แทนการกำหนดจำนวนรอบตายตัว

## กรณีใช้งาน

ลองคิดอย่างสร้างสรรค์ว่าเมื่อใดควรขอให้ Claude Code สร้างเวิร์กโฟลว์แบบไดนามิกสำหรับงานบางประเภท เวิร์กโฟลว์อาจมีประโยชน์กับงานที่ไม่ใช่ด้านเทคนิคมากกว่างานเขียนโค้ดเสียอีก

### การย้ายระบบและการ refactor

[Bun](https://bun.com/) ถูกเขียนใหม่จาก Zig เป็น Rust ด้วยเวิร์กโฟลว์ โดย [เธรด X ของ Jarred](https://x.com/jarredsumner/status/2060050578026189172) อธิบายวิธีที่ทีมใช้ดำเนินงานนี้

หัวใจสำคัญคือแบ่งการย้ายระบบออกเป็นหน่วยที่ชัดเจน เช่น call sites, การทดสอบที่ล้มเหลว หรือ modules สร้าง subagent ใน worktree สำหรับแต่ละจุดที่ต้องแก้ ให้อีก agent ทำ adversarial review แล้วจึงรวมการเปลี่ยนแปลง เมื่อจำเป็น ให้กำชับ agents หลีกเลี่ยงคำสั่งที่ใช้ทรัพยากรหนัก เพื่อให้เครื่องรองรับงานขนานได้มากขึ้น

### การวิจัยเชิงลึก

Claude Code มี skill สำหรับการวิจัยเชิงลึกชื่อ `/deep-research` ซึ่งสร้างขึ้นด้วยเวิร์กโฟลว์แบบไดนามิก โดยจะกระจายการค้นหาเว็บ ดึงแหล่งข้อมูล ตรวจสอบข้ออ้างแบบเป็นปฏิปักษ์ และสังเคราะห์เป็นรายงานพร้อมแหล่งอ้างอิง

รูปแบบเดียวกันนี้ใช้ได้มากกว่าการค้นหาเว็บ Claude สามารถรวบรวมรายงานสถานะจากบริบทใน Slack หรือสำรวจ codebase อย่างละเอียดเพื่อศึกษาว่าฟีเจอร์หนึ่งทำงานอย่างไร

สมมติว่าคุณต้องจัดอันดับรายการตามเกณฑ์เชิงคุณภาพที่ Claude Code ประเมินได้ เช่น เรียง support tickets ตามความรุนแรงของ bug หากพยายามจัดเรียงข้อมูลมากกว่า 1,000 แถวในพรอมป์เดียว คุณภาพจะลดลงและเกินขอบเขตบริบทที่มีประโยชน์ ควรใช้ tournament, pipeline ของ agents ที่เปรียบเทียบเป็นคู่ หรือจัดอันดับแบบแบ่ง bucket ขนานกันแล้วรวมผล การตัดสินเชิงเปรียบเทียบเชื่อถือได้มากกว่าการให้คะแนนแบบสัมบูรณ์ และแต่ละการเปรียบเทียบจะมี context window อิสระ

### หน่วยความจำและการปฏิบัติตามกฎ

![เวิร์กโฟลว์สำหรับหน่วยความจำและการปฏิบัติตามกฎ](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f17517076bb59050d90bb_image8.png)

หาก Claude พลาดกฎชุดเดิมซ้ำ ๆ แม้จะเขียนไว้ใน `CLAUDE.md` แล้ว ให้สร้างเวิร์กโฟลว์ที่มี verifier หนึ่งตัวต่อหนึ่งกฎ และให้ skeptic subagent ตรวจทานตัวกฎเองเพื่อลด false positives

แนวทางย้อนกลับก็ใช้ได้เช่นกัน: ค้นหาการแก้ไขที่เกิดซ้ำจาก sessions ล่าสุดและความคิดเห็นใน code review จัดกลุ่มด้วย agents ที่ทำงานขนานกัน ทดสอบแบบเป็นปฏิปักษ์ว่ากฎแต่ละข้อที่เสนอจะป้องกันความผิดพลาดจริงได้หรือไม่ แล้วกลั่นกฎที่ผ่านการทดสอบกลับเข้าไปใน `CLAUDE.md`

### การสืบหารากสาเหตุ

การ debug ได้ผลดีที่สุดเมื่อสร้างสมมติฐานอิสระหลายข้อแล้วทดสอบแต่ละข้อ การใช้ context window เดียวทำให้เกิด self-preferential bias ได้ง่ายขึ้น

เวิร์กโฟลว์ป้องกันปัญหานี้ในเชิงโครงสร้างได้ด้วยการมอบหลักฐานที่ไม่ทับซ้อนกัน เช่น บันทึก ไฟล์ และข้อมูล ให้ agents แยกกันวิเคราะห์ จากนั้นให้แต่ละสมมติฐานผ่านการตรวจสอบจาก verifier และ refuter ที่เป็นอิสระ

รูปแบบนี้ไม่ได้จำกัดอยู่แค่โค้ด แต่ใช้ได้กับการวิเคราะห์งานขาย ความล้มเหลวด้านวิศวกรรมข้อมูล และการสืบสวนหลังเหตุการณ์ทุกประเภท

### การคัดแยกในวงกว้าง

![เวิร์กโฟลว์สำหรับการคัดแยกในวงกว้าง](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f1778dc00d34cca70819d_image6.png)

ทุกทีมมีคิวสนับสนุน รายงาน bug หรือ backlog รูปแบบอื่นที่มนุษย์ไม่สามารถจัดการได้ครบทั้งหมด

เวิร์กโฟลว์คัดแยกจะจำแนกแต่ละรายการ ตรวจหารายการซ้ำกับสิ่งที่ติดตามอยู่แล้ว และดำเนินการที่เหมาะสม ไม่ว่าจะเป็นการลองแก้ไขหรือส่งต่อให้บุคคลรับผิดชอบ

Quarantine เป็นรูปแบบที่มีประโยชน์ในกรณีนี้: agents ที่อ่านเนื้อหาสาธารณะจากแหล่งที่ไม่เชื่อถือจะถูกห้ามไม่ให้ดำเนินการที่ต้องใช้สิทธิ์สูง ส่วน agents แยกอีกชุดจะทำงานต่อจากข้อมูลที่ได้รับ

ใช้เวิร์กโฟลว์คัดแยกร่วมกับ [`/loop`](https://claude.com/blog/getting-started-with-loops) เพื่อให้ Claude รันงานเหล่านี้อย่างต่อเนื่อง

### การสำรวจและรสนิยม

เวิร์กโฟลว์มีประโยชน์สำหรับการสำรวจหลายแนวทาง เมื่อการตัดสินใจสุดท้ายขึ้นอยู่กับรสนิยม เช่น งานออกแบบหรือการตั้งชื่อ และสามารถถ่ายทอดรสนิยมนั้นออกมาเป็น rubric ได้

ให้ Claude สำรวจตัวเลือกจำนวนมาก แล้วมอบ rubric ของคำตอบที่ดีให้ review agent งานจะสิ้นสุดเมื่อ reviewer พบว่าเกณฑ์ทั้งหมดได้รับการตอบสนองแล้ว นอกจากนี้ยังใช้ tournament เพื่อจัดอันดับหรือเลือกตัวเลือกตาม rubric ได้

### Evals

รัน evals แบบเบาโดยสร้าง agents อิสระใน worktrees และใช้ comparison agents ให้คะแนนผลลัพธ์ตาม rubric วิธีนี้ใช้ประเมินและปรับปรุง skill ตามเกณฑ์เฉพาะได้

### การกำหนดเส้นทางให้โมเดลและระดับความสามารถ

สร้าง classifier agent ที่ปรับให้เข้ากับงาน แล้วให้ agent นั้นเลือกโมเดล วิธีนี้มีประโยชน์เมื่องานวิจัยเบื้องต้นและการเรียกใช้เครื่องมือช่วยให้ทราบว่าการลงมือทำจริงต้องใช้ความสามารถระดับใด

ตัวอย่างเช่น โมเดลที่เหมาะกับคำสั่ง "อธิบายว่า auth module ทำงานอย่างไร" ขึ้นอยู่กับขนาดและโครงสร้างของ module นั้น Classifier สามารถตรวจสอบก่อน แล้วส่งต่องานไปยัง Sonnet หรือ Opus ตามความซับซ้อนที่คาดไว้

## เมื่อใดไม่ควรใช้เวิร์กโฟลว์แบบไดนามิก

เวิร์กโฟลว์ยังเป็นความสามารถใหม่ แม้จะสร้างผลลัพธ์ที่โดดเด่นมากได้ แต่ก็ไม่จำเป็นสำหรับทุกงาน และอาจใช้โทเคนเพิ่มขึ้นอย่างมาก

ควรใช้เมื่อประโยชน์จากการทำงานขนาน ความเชี่ยวชาญเฉพาะทาง หรือการตรวจสอบแบบเป็นปฏิปักษ์คุ้มกับต้นทุนในการประสานงาน งานเขียนโค้ดทั่วไปส่วนใหญ่ไม่ต้องการคณะ reviewer ห้าคน หลักตัดสินเดียวกันนี้ใช้ในระดับสถาปัตยกรรมเมื่อต้องเลือกระหว่าง[ระบบ multi-agent กับระบบ single-agent](https://claude.com/blog/building-multi-agent-systems-when-and-how-to-use-them)

## เคล็ดลับสำหรับการสร้างเวิร์กโฟลว์แบบไดนามิก

### การเขียนพรอมป์

พรอมป์ที่มีรายละเอียดและระบุรูปแบบเวิร์กโฟลว์ที่เกี่ยวข้องอย่างชัดเจนมักให้ผลลัพธ์ดีที่สุด

เวิร์กโฟลว์ไม่ได้จำกัดเฉพาะงานขนาดใหญ่ คุณสามารถขอ "เวิร์กโฟลว์แบบรวดเร็ว" เช่น การทำ adversarial review สั้น ๆ กับสมมติฐานหนึ่งข้อได้

### ใช้ร่วมกับ `/goal` และ `/loop`

สำหรับเวิร์กโฟลว์ที่ทำซ้ำได้ เช่น การคัดแยก การวิจัย หรือการตรวจสอบ ให้ใช้ [`/loop`](https://claude.com/blog/getting-started-with-loops) ร่วมกับ [`/goal`](https://code.claude.com/docs/en/workflows) เพื่อรันตามช่วงเวลาที่กำหนดและตั้งเงื่อนไขการเสร็จสิ้นที่ชัดเจน

### งบประมาณการใช้โทเคน

คุณกำหนดงบประมาณโทเคนสำหรับเวิร์กโฟลว์แบบไดนามิกได้โดยตรง พรอมป์อย่าง "use 10k tokens" จะตั้งเพดานการใช้โทเคนของงานไว้ที่ 10k

### การบันทึกและแชร์เวิร์กโฟลว์แบบไดนามิก

กด `s` ในเมนูเวิร์กโฟลว์เพื่อบันทึกเวิร์กโฟลว์ คุณสามารถ check in ไว้ใน `~/.claude/workflows` หรือเผยแพร่ผ่าน skill

![การบันทึกเวิร์กโฟลว์จากเมนูเวิร์กโฟลว์](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f17b1ca20533e666c867c_image4.png)

หากต้องการแชร์ผ่าน skill ให้วางไฟล์ JavaScript ของเวิร์กโฟลว์ไว้ในโฟลเดอร์ skill แล้วอ้างอิงจาก `SKILL.md` เพื่อให้ยืดหยุ่นมากขึ้น ให้บอก Claude ว่าควรใช้เวิร์กโฟลว์เป็น template ไม่ใช่ script ที่ต้องรันตามแบบคำต่อคำ

![การแชร์เวิร์กโฟลว์ผ่าน skill](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f17cb835cf4f9fd5da921_image7.png)

## จุดเริ่มต้นใหม่สำหรับการค้นพบ

เวิร์กโฟลว์เป็นวิธีใหม่ที่มีประโยชน์ในการต่อยอด Claude Code ควรมองเป็นจุดเริ่มต้นสำหรับค้นพบวิธีใหม่ ๆ ที่ Claude จะช่วยงานของคุณได้ เพราะยังมีอีกมากให้เรียนรู้เกี่ยวกับการใช้งานให้เกิดประโยชน์สูงสุด

หากต้องการแนวทางว่า harness ควรมีองค์ประกอบใดบ้าง โปรดดู[รูปแบบการออกแบบ harness ทั้งสามแบบของ Anthropic](https://claude.com/blog/harnessing-claudes-intelligence)
