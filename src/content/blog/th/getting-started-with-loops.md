ขณะนี้มีการพูดถึงแนวคิด "ออกแบบลูปแทนที่จะเพียงเขียนพรอมป์ให้ AI Agent สำหรับเขียนโค้ด" กันมาก หากลองใช้เวลาบน X เพื่อหาคำตอบว่าลูปคืออะไรกันแน่ คุณจะพบคำอธิบายหลายแบบ

สำหรับทีม Claude Code ลูปคือการที่ AI Agent ทำงานเป็นรอบซ้ำไปเรื่อย ๆ จนกว่าเงื่อนไขหยุดจะสำเร็จ ทีมแยกประเภทของลูปตามมิติหลักดังนี้:

1. ลูปเริ่มทำงานอย่างไร
2. ลูปหยุดอย่างไร
3. ใช้ primitive ใดของ Claude Code
4. เหมาะกับงานประเภทใดที่สุด

บทความนี้ครอบคลุมลูปประเภทหลัก ช่วงเวลาที่ควรใช้แต่ละแบบ และวิธีรักษาคุณภาพโค้ดพร้อมควบคุมการใช้โทเคน ไม่ใช่ทุกงานจะต้องใช้ลูปซับซ้อน ควรเริ่มจากวิธีที่ง่ายที่สุด แล้วเลือกใช้รูปแบบเหล่านี้เฉพาะจุดที่เหมาะสม

## ลูป 4 ประเภท

บทความต้นฉบับแบ่งลูปเป็น 4 ประเภท ได้แก่ ลูปตามรอบสั่งงาน ลูปตามเป้าหมาย ลูปตามเวลา และลูปเชิงรุก ความแตกต่างไม่ได้อยู่แค่ระดับการทำงานอัตโนมัติ แต่รวมถึงตัวกระตุ้น เงื่อนไขหยุด และขอบเขตงานด้วย

### ลูปตามรอบสั่งงาน

![แผนภาพลูปตามรอบสั่งงาน](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a43eb603762e725a739d98c_8ace2295.png)

- **เริ่มโดย**: พรอมป์จากผู้ใช้
- **เงื่อนไขหยุด**: Claude ตัดสินว่างานเสร็จแล้ว หรือต้องการบริบทเพิ่มเติม
- **เหมาะที่สุดสำหรับ**: งานสั้น ๆ แบบครั้งเดียว ซึ่งไม่ได้เป็นส่วนหนึ่งของกระบวนการประจำหรือตารางเวลา
- **วิธีควบคุมการใช้งาน**: เขียนพรอมป์ให้เฉพาะเจาะจง และปรับปรุงการตรวจสอบด้วย skills เพื่อลดจำนวนรอบโต้ตอบ

ทุกพรอมป์ที่คุณส่งจะเริ่มลูปแบบควบคุมด้วยตนเอง โดยคุณเป็นผู้กำกับแต่ละรอบ Claude จะรวบรวมบริบท ลงมือทำ ตรวจสอบผลงาน ทำซ้ำเมื่อจำเป็น แล้วจึงตอบกลับ นี่คือ agentic loop ที่บทความต้นฉบับกล่าวถึง

ตัวอย่างเช่น คุณขอให้ Claude สร้างปุ่ม Like มันจะอ่านโค้ด แก้ไขไฟล์ รันการทดสอบ แล้วส่งคืนผลลัพธ์ที่เชื่อว่าใช้งานได้ จากนั้นคุณตรวจดูผลและเขียนพรอมป์ถัดไป

คุณปรับปรุงขั้นตอนตรวจสอบนี้ได้ด้วยการเขียนสิ่งที่เคยตรวจด้วยตนเองลงใน `SKILL.md` เพื่อให้ Claude ตรวจสอบงานของตัวเองแบบครบวงจรได้มากขึ้น หากต้องเลือกระหว่าง skills, hooks และ subagents สำหรับระบบอัตโนมัติลักษณะนี้ โปรดดูคู่มือ [steering Claude Code](https://claude.com/blog/steering-claude-code-skills-hooks-rules-subagents-and-more) skill ควรมอบเครื่องมือหรือ connector ที่ช่วยให้ Claude มองเห็น วัดผล หรือโต้ตอบกับผลลัพธ์ได้โดยตรง ยิ่งเกณฑ์ตรวจสอบวัดเป็นตัวเลขได้มาก Claude ก็ยิ่งตรวจสอบงานของตัวเองได้ง่ายขึ้น

ตัวอย่าง skill สำหรับตรวจสอบ frontend อาจมีลักษณะดังนี้:

```markdown
---
name: verify-frontend-change
description: Verify any UI change end-to-end before declaring it done.
---

# Verifying frontend changes

Never report a UI change as complete based on a successful edit alone. Verify it the way a human reviewer would:

1. Start the dev server and open the edited page in the browser.
2. Interact with the change directly. For a new control (button, input, toggle): click it, confirm the expected state change, and screenshot before/after.
3. Check the browser console: zero new errors or warnings.
4. Use the Chrome Devtools MCP, run a performance trace and audit Core Web Vitals.

If any step fails, fix the issue and rerun from step 1 — do not hand back partially verified work.
```

ประเด็นสำคัญไม่ใช่การเขียน skill สารพัดประโยชน์เพียงตัวเดียว แต่คือการระบุความหมายที่แท้จริงของคำว่า "เสร็จ" สำหรับคุณให้ชัดเจน มิฉะนั้น Claude ต้องอาศัยดุลยพินิจของตัวเองว่าควรหยุดเมื่อใด

### ลูปตามเป้าหมาย

![แผนภาพลูปตามเป้าหมาย](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a43eb603762e725a739d98f_c6fa9ae5.png)

- **เริ่มโดย**: พรอมป์ที่ผู้ใช้ส่งแบบเรียลไทม์
- **เงื่อนไขหยุด**: บรรลุเป้าหมาย หรือถึงจำนวนรอบสูงสุด
- **เหมาะที่สุดสำหรับ**: งานที่มีเกณฑ์สิ้นสุดซึ่งตรวจสอบได้
- **วิธีควบคุมการใช้งาน**: กำหนดเกณฑ์สำเร็จและเพดานจำนวนรอบให้ชัด เช่น "หยุดหลังจากลอง 5 ครั้ง"

บางครั้งรอบเดียวไม่เพียงพอ โดยเฉพาะงานซับซ้อน โดยทั่วไป AI Agent จะทำงานได้ดีขึ้นเมื่อมีโอกาสปรับแก้หลายรอบ เมื่อใช้ `/goal` คุณจะกำหนดเกณฑ์ว่างานเสร็จสมบูรณ์เมื่อใด และให้ Claude ทำงานต่อไปจนบรรลุเป้าหมายนั้น

เมื่อคุณระบุเกณฑ์สำเร็จ Claude ไม่ต้องตัดสินเองว่าอะไรคือ "ดีพอ" จึงมีโอกาสจบลูปเร็วเกินไปน้อยลง ทุกครั้งที่ Claude พยายามหยุด โมเดลผู้ประเมินจะตรวจเงื่อนไข หากยังไม่ผ่าน ก็จะส่ง Claude กลับไปทำงานต่อจนกว่าจะบรรลุเป้าหมายหรือถึงจำนวนรอบที่กำหนด

นี่คือเหตุผลที่เกณฑ์แบบกำหนดแน่นอนทำงานได้ดีเป็นพิเศษ เช่น จำนวนการทดสอบที่ผ่าน ค่าคะแนนขั้นต่ำ หรือรายการข้อผิดพลาดที่ต้องว่างเปล่า

ตัวอย่าง:

```text
/goal get the homepage Lighthouse score to 90 or above, stop after 5 tries.
```

แก่นของลูปประเภทนี้คือการย้ายอำนาจในการหยุดจากความรู้สึกเชิงอัตวิสัยของ AI Agent ไปอยู่ที่เงื่อนไขซึ่งตรวจสอบได้

### ลูปตามเวลา

- **เริ่มโดย**: ช่วงเวลาที่กำหนด
- **เงื่อนไขหยุด**: คุณยกเลิก หรือภารกิจเสร็จสิ้น เช่น PR ถูก merge หรือคิวว่าง
- **เหมาะที่สุดสำหรับ**: งานที่เกิดซ้ำ หรืองานที่ต้องโต้ตอบกับสภาพแวดล้อมและระบบภายนอก
- **วิธีควบคุมการใช้งาน**: ตั้งช่วงเวลาให้นานขึ้น หรือใช้เหตุการณ์เป็นตัวกระตุ้นแทนการ polling ตามตารางตายตัว

งานของ AI Agent บางประเภทเกิดซ้ำ: ตัวงานเหมือนเดิม แต่ข้อมูลขาเข้าเปลี่ยนไป ตัวอย่างหนึ่งคือการสรุปข้อความ Slack ทุกเช้า งานอีกประเภทขึ้นอยู่กับระบบภายนอก ซึ่งรูปแบบการโต้ตอบที่ง่ายคือคอยตรวจเป็นระยะและตอบสนองต่อความเปลี่ยนแปลง เช่น PR อาจได้รับความคิดเห็นจาก code review หรือ CI อาจล้มเหลว

กรณีเหล่านี้ใช้ `/loop` เพื่อรันพรอมป์ซ้ำตามช่วงเวลาได้ ตัวอย่าง:

```text
/loop 5m check my PR, address review comments, and fix failing CI
```

`/loop` ทำงานบนคอมพิวเตอร์ของคุณ จึงหยุดเมื่อปิดเครื่อง หากต้องการย้ายลูปไปทำงานบนคลาวด์ ให้ใช้ `/schedule` สร้าง routine

หัวใจสำคัญคืออย่ารัน routine บ่อยกว่าความถี่ที่ระบบต้นทางเปลี่ยนจริง คิวที่เปลี่ยนเพียงชั่วโมงละครั้งไม่ควรใช้โทเคนสแกนทุกนาที

### ลูปเชิงรุก

![แผนภาพลูปเชิงรุก](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a43eb603762e725a739d989_eb9e496a.png)

- **เริ่มโดย**: เหตุการณ์หรือตารางเวลา โดยไม่ต้องมีคนเข้าร่วมแบบเรียลไทม์
- **เงื่อนไขหยุด**: แต่ละงานสิ้นสุดเมื่อบรรลุเป้าหมายของตัวเอง ส่วน routine จะทำงานต่อจนกว่าคุณจะปิด
- **เหมาะที่สุดสำหรับ**: กระแสงานซ้ำที่มีขอบเขตชัด เช่น รายงาน bug, issue triage, การย้ายระบบ และการอัปเกรด dependency
- **วิธีควบคุมการใช้งาน**: ส่ง routine ไปยังโมเดลที่เล็กและเร็วกว่า และเก็บโมเดลที่เก่งที่สุดไว้สำหรับงานที่ต้องใช้ดุลยพินิจ

primitive ที่กล่าวมา เมื่อนำมารวมกับความสามารถอื่นของ Claude Code เช่น auto mode และ dynamic workflows (research preview) สามารถประกอบเป็นลูปสำหรับงานระยะยาวได้

ตัวอย่างเช่น หากต้องจัดการ feedback ที่เข้ามาต่อเนื่อง คุณสามารถรวมความสามารถเหล่านี้:

1. ใช้ `/schedule` (research preview) รัน routine เพื่อตรวจหารายงานใหม่เป็นระยะ
2. ใช้ `/goal` ระบุว่างานที่เสร็จต้องเป็นอย่างไร และใช้ skills บันทึกวิธีตรวจสอบ
3. ใช้ dynamic workflows ประสาน AI Agent หลายตัวให้คัดแยกรายงาน แก้ปัญหา และ review งานแก้ไข
4. ใช้ auto mode เพื่อให้ routine ทำงานต่อได้โดยไม่ต้องหยุดขออนุญาตทุกขั้น

เมื่อประกอบเข้าด้วยกัน พรอมป์อาจมีลักษณะดังนี้:

```text
/schedule every hour: check #project-feedback for bug reports. /goal: don't stop until every report found this run is triaged, actioned, and responded to. When fixing a bug, use a workflow to explore three solutions in parallel worktrees and have a judge adversarially review them.
```

นี่ไม่ใช่การเขียนพรอมป์ให้ยาวขึ้น แต่เป็นการนำตัวกระตุ้น เงื่อนไขหยุด การสำรวจแบบขนาน การ review และขอบเขตสิทธิ์มาอยู่ในระบบรันไทม์เดียวกัน

## การรักษาคุณภาพโค้ด

คุณภาพผลลัพธ์จากลูปขึ้นอยู่กับระบบที่ล้อมรอบมัน บทความต้นฉบับเน้นหลักการออกแบบหลายข้อ:

1. **รักษาความสะอาดของ codebase**: Claude จะทำตามรูปแบบและข้อตกลงที่มีอยู่ใน codebase หาก codebase ยุ่งเหยิง ลูปก็จะขยายรูปแบบที่ยุ่งเหยิงนั้น
2. **ให้ Claude มีวิธีตรวจสอบงานของตัวเอง**: ใช้ [skills](https://code.claude.com/docs/en/skills) ระบุให้ชัดว่าอะไรคือผลงานที่ดีสำหรับคุณและทีม
3. **ทำให้เอกสารเข้าถึงง่าย**: เอกสารของ framework และ library มีแนวปฏิบัติล่าสุด และ Claude ต้องเข้าถึงข้อมูลเหล่านั้นได้
4. **ใช้ AI Agent ตัวที่สองทำ code review**: ผู้ตรวจที่เริ่มด้วยบริบทใหม่จะมีอคติน้อยกว่า และไม่ถูกชี้นำด้วยเหตุผลของ AI Agent หลัก คุณสามารถใช้ skill `/code-review` ที่มีมาให้ หรือ [Code Review](https://code.claude.com/docs/en/code-review) ของ GitHub

เมื่อผลลัพธ์ครั้งใดไม่ถึงมาตรฐาน อย่าหยุดแค่แก้ปัญหาครั้งนั้น ควรบันทึกความล้มเหลวกลับเข้าไปในระบบ เพื่อให้การทำงานรอบถัดไปทั้งหมดได้รับประโยชน์ ความล้มเหลวควรกลายเป็น skill, การทดสอบ, สคริปต์, กฎ หรือเกณฑ์ review ไม่ใช่เพียง patch เฉพาะกิจ

## การจัดการการใช้โทเคน

หากต้องการควบคุมการใช้โทเคน ลูปต้องมีขอบเขตชัดเจน คำแนะนำจากบทความต้นฉบับสรุปได้ดังนี้:

1. **เลือก primitive และโมเดลให้เหมาะกับงาน**: งานเล็กไม่จำเป็นต้องใช้ AI Agent หลายตัวหรือลูปซับซ้อน บางงานใช้โมเดลที่ถูกและเร็วกว่าได้
2. **กำหนดเกณฑ์สำเร็จและเงื่อนไขหยุดให้ชัดเจน**: ยิ่งเฉพาะเจาะจง Claude ก็ยิ่งหาคำตอบได้เร็วโดยไม่หยุดก่อนเวลา
3. **ทดลองขนาดเล็กก่อนรันครั้งใหญ่**: dynamic workflows อาจเปิด AI Agent จำนวนมาก ควรประเมินการใช้งานจากงานส่วนเล็กก่อน
4. **ใช้สคริปต์กับงานที่ผลลัพธ์แน่นอน**: การรันสคริปต์ถูกกว่าการให้โมเดลใช้เหตุผลกับขั้นตอนเดิมทุกครั้ง เช่น PDF skill อาจมีสคริปต์กรอกฟอร์มให้ Claude เรียกใช้โดยตรง แทนการเขียนโค้ดใหม่ทุกครั้ง
5. **อย่ารัน routine บ่อยเกินความจำเป็น**: จับคู่ช่วงเวลากับอัตราการเปลี่ยนแปลงจริงของระบบที่เฝ้าดู
6. **ทบทวนการใช้งาน**: `/usage` แยกการใช้งานล่าสุดตาม skills, subagents และ MCP; `/goal` ที่ไม่ใส่อาร์กิวเมนต์จะแสดงจำนวนรอบและการใช้โทเคนปัจจุบัน; `/workflows` แสดงการใช้โทเคนของ AI Agent แต่ละตัวและให้คุณหยุดได้ทุกเมื่อ

การเลือก [โมเดลและ effort level](https://claude.com/blog/claude-model-and-effort-level-in-claude-code) ก็เป็นหนึ่งในตัวแปรสำคัญที่สุดที่กำหนดต้นทุนของลูป

สรุปสั้น ๆ ลูปไม่ใช่วิธีปล่อยให้ AI Agent ทำงานไปตลอดกาล แต่เป็นวิธีให้มันทำงานซ้ำภายในขอบเขตที่ระบุชัดเจน

## เริ่มต้นใช้งาน

บทความต้นฉบับปิดท้ายด้วยตารางเปรียบเทียบว่าส่วนใดของงานที่คุณมอบให้ลูปแต่ละแบบ:

| ลูป | สิ่งที่คุณมอบหมาย | ใช้เมื่อ | เลือกใช้ |
| --- | --- | --- | --- |
| ตามรอบสั่งงาน | ขั้นตอนตรวจสอบ | คุณกำลังสำรวจหรือตัดสินใจ | skills สำหรับตรวจสอบที่กำหนดเอง |
| ตามเป้าหมาย | เงื่อนไขหยุด | คุณรู้ว่างานที่เสร็จแล้วต้องเป็นอย่างไร | `/goal` |
| ตามเวลา | ตัวกระตุ้น | งานเกิดขึ้นนอกโปรเจกต์ตามกำหนดเวลา | `/loop`, `/schedule` |
| เชิงรุก | พรอมป์ | งานเกิดซ้ำและนิยามชัดเจน | ทั้งหมดข้างต้น รวมถึง dynamic workflows |

หากต้องการเริ่มใช้ลูป ให้ดูงานที่คุณทำอยู่แล้ว เลือกหนึ่งงานที่คุณเป็นคอขวด แล้วถามว่า: คุณเขียนเกณฑ์ตรวจสอบได้หรือไม่? เป้าหมายชัดพอที่จะตัดสินว่าเสร็จหรือยังหรือไม่? งานเข้ามาตามตารางเวลาหรือผ่านเหตุการณ์ภายนอกหรือไม่?

เมื่อมีแนวคิดแล้ว ให้รันลูป สังเกตว่ามันติดขัดหรือล้ำขอบเขตตรงไหน แล้วปรับปรุงระบบต่อไป
