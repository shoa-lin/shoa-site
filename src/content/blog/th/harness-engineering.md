> **คำศัพท์สำคัญ**
>
> **Harness** หมายถึงทุกสิ่งใน AI agent ที่อยู่นอกเหนือจากตัวโมเดล: Agent = Model + Harness สำหรับ coding agent คำนี้ครอบคลุมทั้งพรอมป์ระบบ การค้นคืนโค้ด และระบบ orchestration ที่ผู้สร้างจัดเตรียมไว้ ตลอดจนชั้นนอกที่ผู้ใช้ควบคุมเอง เช่น กฎ Skills สคริปต์ และชุดตรวจสอบ
>
> **Guides / Sensors**: Guides คือกลไกควบคุมแบบ feedforward ที่ชี้นำ agent ก่อนลงมือ ส่วน Sensors คือกลไกควบคุมแบบ feedback ที่สังเกตผลลัพธ์หลังลงมือและกระตุ้นให้เกิดการแก้ไขตนเอง
>
> **Computational / Inferential**: การควบคุมแบบ computational ใช้เครื่องมือที่ให้ผลแน่นอน เช่น tests, linters และ type checkers ส่วนการควบคุมแบบ inferential อาศัยการตัดสินเชิงความหมาย เช่น AI code review หรือ LLM-as-judge

คำว่า "harness" กลายเป็นคำย่อที่ใช้เรียกทุกสิ่งใน AI agent ยกเว้นตัวโมเดล: [Agent = Model + Harness](https://blog.langchain.com/the-anatomy-of-an-agent-harness/) แต่คำจำกัดความนี้กว้างมาก จึงควรจำกัดขอบเขตให้ชัดเมื่อพูดถึง coding agent

Harness ส่วนหนึ่งของ coding agent ถูกสร้างโดยผู้พัฒนาเครื่องมือ ผ่านพรอมป์ระบบ การค้นคืนโค้ด และบางครั้งก็รวมถึง[ระบบ orchestration ที่ซับซ้อน](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents) ขณะเดียวกัน coding agent ยังเปิดทางให้ผู้ใช้สร้าง Harness ชั้นนอกให้เหมาะกับระบบและกรณีใช้งานของตนเอง

![วงกลมซ้อนกันสามชั้น โดยมีโมเดลอยู่ตรงกลาง ล้อมด้วย Harness ของผู้สร้าง coding agent และ Harness ของผู้ใช้เป็นชั้นนอกสุด](/assets/blog/harness-engineering/harness-bounded-contexts.png)

ภาพที่ 1: คำว่า "harness" มีขอบเขตความหมายต่างกันตามบริบทที่กำลังพิจารณา

Harness ชั้นนอกที่ออกแบบมาดีมีเป้าหมายสองข้อ คือเพิ่มโอกาสให้ agent ทำงานถูกต้องตั้งแต่ครั้งแรก และสร้าง feedback loop ที่แก้ปัญหาให้ได้มากที่สุดก่อนส่งถึงมนุษย์ ผลที่ต้องการคือภาระการ review ลดลง คุณภาพระบบสูงขึ้น และยังช่วยลดโทเคนที่สูญเปล่าเป็นผลพลอยได้

## Feedforward และ Feedback

วิศวกรรม Harness ผสานการควบคุมสองรูปแบบเข้าด้วยกัน:

- **Guides (การควบคุมแบบ feedforward)** คาดการณ์พฤติกรรมที่ไม่ต้องการและชี้นำ agent **ก่อน** ลงมือ เพื่อเพิ่มโอกาสให้ผลลัพธ์แรกมีคุณภาพ
- **Sensors (การควบคุมแบบ feedback)** สังเกตผลลัพธ์ **หลัง** agent ลงมือและช่วยให้แก้ไขตนเอง กลไกนี้ทรงพลังเป็นพิเศษเมื่อออกแบบสัญญาณให้ LLM นำไปใช้ได้ทันที เช่น ข้อความจาก custom linter ที่ระบุวิธีแก้ไว้ด้วย ซึ่งเป็นการฉีดพรอมป์ในทางสร้างสรรค์รูปแบบหนึ่ง

ใช้เพียงรูปแบบใดรูปแบบหนึ่งย่อมไม่ครบถ้วน หากมี feedback แต่ไม่มี feedforward agent อาจทำผิดแบบเดิมซ้ำ ๆ แต่หากมี feedforward โดยไม่มี feedback ระบบจะบันทึกกฎไว้โดยไม่เคยรู้ว่ากฎนั้นได้ผลจริงหรือไม่

## Computational กับ Inferential

Guides และ Sensors ทำงานได้สองแบบ:

- **Computational**: ให้ผลแน่นอนและรวดเร็ว โดยทั่วไปทำงานบน CPU การทดสอบ linters, type checkers และการวิเคราะห์โครงสร้างใช้เวลาตั้งแต่มิลลิวินาทีถึงไม่กี่วินาทีและให้ผลที่เชื่อถือได้
- **Inferential**: ใช้การวิเคราะห์เชิงความหมาย เช่น AI code review และการประเมินแบบ LLM-as-judge โดยทั่วไปทำงานบน GPU หรือ NPU การควบคุมชนิดนี้ช้ากว่า แพงกว่า และผลอาจต่างกันในแต่ละครั้ง

Computational guides ใช้เครื่องมือที่ให้ผลแน่นอนเพื่อยกระดับคุณภาพตั้งแต่รอบแรก ส่วน computational sensors มีต้นทุนต่ำและเร็วพอจะทำงานคู่กับ agent ในทุกการเปลี่ยนแปลง การควบคุมแบบ inferential มีต้นทุนสูงกว่าและผลไม่คงที่ แต่ให้คำแนะนำที่ละเอียดและตัดสินเชิงความหมายได้ หากใช้โมเดลที่แข็งแรง หรือพูดให้แม่นกว่านั้นคือโมเดลที่เหมาะกับงาน inferential sensors ก็ยังช่วยเพิ่มความน่าเชื่อถือได้

### ความสัมพันธ์กับวิศวกรรมบริบท

[วิศวกรรมบริบท](https://martinfowler.com/articles/exploring-gen-ai/context-engineering-coding-agents.html) เป็นวิธีทำให้ Guides และ Sensors พร้อมใช้งานสำหรับ agent ดังนั้นการออกแบบ Harness ฝั่งผู้ใช้ให้ coding agent จึงเป็นวิศวกรรมบริบทรูปแบบหนึ่งที่มีขอบเขตชัดเจน

## วงรอบการกำกับทิศทาง

บทบาทของมนุษย์คือ **กำกับทิศทาง (steer)** ให้ agent ด้วยการปรับ Harness ซ้ำอย่างต่อเนื่อง เมื่อปัญหาเดิมเกิดขึ้นอีก ก็ควรยกระดับการควบคุมทั้ง feedforward และ feedback เพื่อลดโอกาสเกิดซ้ำหรือป้องกันไม่ให้เกิดเลย

AI ช่วยปรับปรุง Harness ของตัวเองได้เช่นกัน coding agent ทำให้การสร้างกลไกควบคุมเฉพาะทางและ static analysis มีต้นทุนต่ำลงมาก Agent สามารถเขียนการทดสอบโครงสร้าง สกัดร่างกฎจากรูปแบบที่สังเกตพบ สร้างโครง custom linter และสำรวจ codebase เพื่อจัดทำคู่มือวิธีใช้งาน

## จังหวะเวลา: ย้ายการควบคุมคุณภาพไปทางซ้าย

ทีมที่ใช้[การผสานรวมอย่างต่อเนื่อง](https://martinfowler.com/articles/continuousIntegration.html) ต้องกระจายการทดสอบ การตรวจสอบ และ human review ไปตามช่วงต่าง ๆ ของการพัฒนาโดยพิจารณาจากต้นทุน ความเร็ว และความสำคัญมาโดยตลอด ส่วนทีมที่มุ่งสู่[การส่งมอบอย่างต่อเนื่อง](https://martinfowler.com/bliki/ContinuousDelivery.html) ย่อมต้องการให้ทุกสถานะของ commit พร้อม deploy ในอุดมคติ ชุดตรวจสอบจึงควรอยู่ทางซ้ายของเส้นทางสู่ production ให้มากเท่าที่ทำได้ เพราะยิ่งพบความล้มเหลวเร็ว ต้นทุนการแก้ก็ยิ่งต่ำ

Feedback sensors รวมถึงชนิด inferential ควรถูกกระจายไปตลอด lifecycle ตามหลักเดียวกัน

**Feedforward และ feedback ใน lifecycle ของการเปลี่ยนแปลง**

- การควบคุมใดเร็วพอจะรันก่อน integration หรือแม้แต่ก่อนสร้าง commit ตัวอย่างเช่น linters, ชุดทดสอบที่รวดเร็ว และ code review agent ระดับพื้นฐาน
- การควบคุมใดมีต้นทุนสูงจนควรรันหลัง integration ใน pipeline เท่านั้น โดยรันร่วมกับชุดตรวจสอบเร็วซ้ำอีกครั้ง ตัวอย่างเช่น mutation testing และ code review ที่กว้างขึ้นซึ่งต้องเห็นภาพรวมทั้งหมด

![ตัวอย่าง feedforward guides และ feedback sensors ที่ทำงานก่อนและหลัง integration ใน lifecycle ของการเปลี่ยนแปลง](/assets/blog/harness-engineering/harness-change-lifecycle-examples.png)

**Sensors สำหรับตรวจ drift และสุขภาพระบบอย่างต่อเนื่อง**

- **Codebase drift sensors** ทำงานนอก lifecycle ของการเปลี่ยนแปลง เพื่อตรวจจับคุณภาพที่เสื่อมลงทีละน้อย เช่น dead code, test coverage ที่อ่อนลง และปัญหาด้าน dependencies
- **Runtime health sensors** ช่วยให้ agent เฝ้าดูสัญญาณจาก production เช่น SLO ที่แย่ลง คุณภาพคำตอบจากการสุ่มตัวอย่าง หรือ logs ที่ผิดปกติ แล้วเสนอแนวทางปรับปรุง

![ตัวอย่างการตรวจ codebase drift อย่างต่อเนื่องและ runtime feedback sensors หลัง integration](/assets/blog/harness-engineering/harness-continuous-feedback-examples.png)

## หมวดหมู่ของการกำกับควบคุม

Harness ของ agent ทำหน้าที่คล้ายตัวกำกับในศาสตร์[ไซเบอร์เนติกส์](https://en.wikipedia.org/wiki/Cybernetics) โดยผสาน feedforward กับ feedback เพื่อกำกับ codebase ให้เคลื่อนไปสู่สถานะที่ต้องการ สถานะนั้นมีหลายมิติ และแต่ละมิติต้องใช้ Harness คนละแบบ การแยกหมวดหมู่จึงสำคัญ เพราะแต่ละมิติมีระดับความเอื้อต่อการสร้าง Harness และความซับซ้อนต่างกันมาก

ในปัจจุบัน การแบ่งเป็นสามหมวดหมู่มีประโยชน์:

### Harness สำหรับความสามารถในการบำรุงรักษา

ตัวอย่างส่วนใหญ่ในบทความนี้กำกับคุณภาพภายในของโค้ดและความสามารถในการบำรุงรักษา Harness ประเภทนี้สร้างได้ง่ายที่สุดในตอนนี้ เพราะมีเครื่องมือที่พัฒนาเต็มที่รองรับอยู่แล้ว

หากต้องการประเมินว่าการควบคุมเหล่านี้เพิ่มความน่าเชื่อถือได้แค่ไหน ให้เทียบกับ[รูปแบบความล้มเหลวที่พบบ่อยของ coding agent](https://martinfowler.com/articles/exploring-gen-ai/13-role-of-developer-skills.html):

- **Computational sensors ตรวจจับปัญหาเชิงโครงสร้างได้อย่างสม่ำเสมอ** เช่น โค้ดซ้ำ cyclomatic complexity, coverage ที่ขาดหาย architectural drift และการผิดข้อกำหนดด้าน style กลไกเหล่านี้มีต้นทุนต่ำ ผ่านการพิสูจน์มาแล้ว และให้ผลแน่นอน
- **LLM จัดการปัญหาเชิงความหมายได้บางส่วน** เช่น โค้ดที่มีความหมายซ้ำกัน การทดสอบที่ซ้ำซ้อน การแก้แบบ brute-force และวิธีแก้ที่ออกแบบซับซ้อนเกินจำเป็น แต่ทำได้ด้วยต้นทุนสูงและเป็นเพียงความน่าจะเป็น จึงไม่ใช่ชุดตรวจสอบที่ควรรันกับทุก commit
- **ทั้งสองแบบยังตรวจไม่พบปัญหาสำคัญบางชนิดอย่างน่าเชื่อถือ** เช่น การวินิจฉัยปัญหาผิด การสร้างฟีเจอร์ที่ไม่จำเป็น การออกแบบเกินความต้องการ และการเข้าใจคำสั่งผิด บางครั้งอาจตรวจพบ แต่ยังไม่สม่ำเสมอพอจะตัด human supervision ออก หากมนุษย์ไม่เคยระบุผลลัพธ์ที่ต้องการให้ชัดเจน ความถูกต้องก็อยู่นอกขอบเขตของ Sensor ทุกชนิด

### Harness สำหรับความเหมาะสมทางสถาปัตยกรรม

หมวดหมู่นี้รวม Guides และ Sensors ที่นิยามและตรวจสอบคุณลักษณะทางสถาปัตยกรรมของแอปพลิเคชัน หรือกล่าวอีกอย่างคือ [Fitness Functions](https://www.thoughtworks.com/en-de/radar/techniques/architectural-fitness-function)

ตัวอย่าง:

- Skills ส่งข้อกำหนดด้านประสิทธิภาพไปข้างหน้า ขณะที่ performance tests รายงานกลับว่าการเปลี่ยนแปลงของ agent ทำให้ดีขึ้นหรือแย่ลง
- Skills อธิบายข้อตกลงด้าน observability เช่น มาตรฐาน logging ขณะที่คำแนะนำในการ debug ขอให้ agent ทบทวนว่าข้อมูล logs ที่มีอยู่มีคุณภาพเพียงใด

### Harness สำหรับพฤติกรรม

นี่คือหมวดหมู่ที่ยากที่สุด: เราจะชี้นำและตรวจจับได้อย่างไรว่าแอปพลิเคชันทำงานตามที่ผู้ใช้ต้องการจริง

- **Feedforward**: ข้อกำหนดเชิงฟังก์ชัน ตั้งแต่พรอมป์สั้น ๆ ไปจนถึงคำอธิบายหลายไฟล์
- **Feedback**: ชุดทดสอบที่ AI สร้างขึ้น ซึ่งผ่านการทดสอบและมี coverage ในระดับเหมาะสม บางครั้งเฝ้าตรวจด้วย mutation testing และเสริมด้วยการทดสอบด้วยมนุษย์

แนวทางนี้ฝากความไว้วางใจไว้กับการทดสอบที่ AI สร้างมากเกินไป บางทีมเริ่มได้ผลดีจากรูปแบบ [approved fixtures](https://lexler.github.io/augmented-coding-patterns/patterns/approved-fixtures/) แต่รูปแบบนี้ใช้ได้ง่ายในบางพื้นที่มากกว่าพื้นที่อื่น จึงเป็นเครื่องมือที่ควรเลือกใช้ตามบริบท ไม่ใช่คำตอบทั้งหมดของคุณภาพการทดสอบ

เรายังต้องการ Behaviour Harness ที่ดีกว่านี้ ก่อนที่ทีมจะลดการกำกับดูแลและการทดสอบด้วยมนุษย์ได้อย่างมั่นใจ

![โมเดล Harness แบบย่อที่มี Guides และ Sensors ครอบคลุมมิติด้านการบำรุงรักษา ความเหมาะสมทางสถาปัตยกรรม และพฤติกรรม](/assets/blog/harness-engineering/harness-types.png)

## ความเอื้อต่อการสร้าง Harness

ไม่ใช่ทุก codebase จะเหมาะกับการสร้าง Harness เท่ากัน ภาษาที่มี type system เข้มแข็งทำให้ type checking ทำหน้าที่เป็น Sensor ได้ ขอบเขตโมดูลที่ชัดเจนทำให้สร้างข้อจำกัดทางสถาปัตยกรรมได้ Framework อย่าง Spring ซ่อนรายละเอียดที่ agent ไม่จำเป็นต้องจัดการ และเพิ่มโอกาสสำเร็จโดยอ้อม หาก codebase ไม่มีคุณสมบัติเหล่านี้ ก็ไม่สามารถสร้างการควบคุมที่สอดคล้องกันได้

ระบบ greenfield และ legacy เผชิญข้อจำกัดต่างกัน:

- **ทีม greenfield** ออกแบบโดยคำนึงถึงความเอื้อต่อการสร้าง Harness (harnessability) ได้ตั้งแต่วันแรก การเลือกเทคโนโลยีและสถาปัตยกรรมจะกำหนดว่า codebase นั้นกำกับควบคุมได้มากเพียงใด
- **ทีม legacy** โดยเฉพาะทีมที่ทำงานกับ technical debt จำนวนมาก เผชิญปัญหาที่ยากกว่า เพราะจุดที่ต้องการ Harness มากที่สุด มักเป็นจุดที่สร้าง Harness ได้ยากที่สุดเช่นกัน

## Harness templates

องค์กรส่วนใหญ่ใช้ service topology ทั่วไปเพียงไม่กี่แบบรองรับความต้องการส่วนมาก เช่น บริการธุรกิจที่มี API, event processors และ data dashboards องค์กรที่มีวุฒิภาวะมักเข้ารหัส topology เหล่านี้ไว้ใน service templates อยู่แล้ว

Templates เหล่านั้นอาจพัฒนาไปเป็น **Harness templates** ซึ่งรวม Guides และ Sensors เป็นชุดเดียว เพื่อจำกัดให้ coding agent ทำงานภายในโครงสร้าง ข้อตกลง และ technology stack ของ topology ที่กำหนด ในอนาคต ทีมอาจเลือกเทคโนโลยีโดยพิจารณาจาก Harness ที่มีให้ใช้ด้วย

![ตัวอย่าง service topologies ที่แต่ละ topology มี Harness template ซึ่งประกอบด้วย Guides และ Sensors](/assets/blog/harness-engineering/harness-templates.png)

### กฎของ Ashby

[กฎความหลากหลายที่จำเป็นของ Ashby](https://en.wikipedia.org/wiki/Variety_%28cybernetics%29#Law_of_requisite_variety) สนับสนุนเหตุผลของการกำหนด topology ล่วงหน้าให้ชัดยิ่งขึ้น ตัวกำกับต้องมีความหลากหลายอย่างน้อยเท่ากับระบบที่ตนควบคุม และควบคุมได้เฉพาะสิ่งที่มีแบบจำลองรองรับ coding agent ที่ใช้ LLM สามารถสร้างได้แทบทุกอย่าง การยึด topology หนึ่งแบบช่วยลดพื้นที่ของความเป็นไปได้ และทำให้การสร้าง Harness ที่ครอบคลุมเป็นไปได้มากขึ้น การกำหนด topology จึงเป็นการลดความหลากหลายของระบบที่ต้องควบคุม

Harness templates สืบทอดปัญหาการบำรุงรักษาแบบเดียวกับ service templates เมื่อสร้าง instance แล้ว มันจะค่อย ๆ เบี่ยงออกจากการปรับปรุงของต้นแบบ upstream การจัดการเวอร์ชันและการส่งการปรับปรุงกลับอาจยากยิ่งกว่าเดิม เมื่อ Guides และ Sensors ให้ผลไม่แน่นอนและทดสอบได้ยาก

## บทบาทของมนุษย์

นักพัฒนามนุษย์นำทักษะและประสบการณ์เข้ามาในทุก codebase ในฐานะ Harness โดยนัย เราซึมซับข้อตกลงและแนวปฏิบัติที่ดี รับรู้ต้นทุนทางความคิดของความซับซ้อน และรู้ว่าชื่อของเราผูกอยู่กับ commit เรายังมีบริบทขององค์กร ทั้งสิ่งที่ทีมกำลังพยายามทำให้สำเร็จ technical debt ใดที่ยอมรับได้ด้วยเหตุผลทางธุรกิจ และคำว่า "ดี" ในที่นี้หมายถึงอะไร การทำงานทีละขั้นด้วยจังหวะของมนุษย์เปิดพื้นที่ให้ประสบการณ์เหล่านี้ปรากฏและส่งผลต่อการตัดสินใจ

Coding agent ไม่มีสิ่งเหล่านี้ มันไม่ต้องรับผิดชอบต่อสังคม ไม่มีสัญชาตญาณต่อต้านฟังก์ชันยาว 300 บรรทัด ไม่มีความรู้สึกว่า "ทีมเราไม่ทำแบบนี้" และไม่มีความทรงจำขององค์กร มันแยกไม่ได้ว่าข้อตกลงใดเป็นข้อจำกัดสำคัญที่ค้ำจุนระบบ และข้อใดเป็นเพียงความเคยชิน รวมถึงไม่รู้ว่าวิธีแก้ที่ถูกต้องทางเทคนิคสอดคล้องกับเจตนาของทีมหรือไม่

Harness ดึงประสบการณ์บางส่วนของมนุษย์ออกมาเขียนให้ชัดเจน แต่ทำได้เพียงระดับหนึ่ง ระบบ Guides, Sensors และวงรอบการแก้ไขตนเองที่สอดคล้องกันมีต้นทุนสูง เป้าหมายของ Harness ที่ดีจึงไม่จำเป็นต้องกำจัดมนุษย์ออกจากกระบวนการ แต่คือการพาความสนใจของมนุษย์ไปยังจุดที่สำคัญที่สุด

## จุดเริ่มต้นและคำถามที่ยังเปิดอยู่

กรอบคิดนี้รวบรวมเทคนิคที่เริ่มปรากฏในการใช้งานจริง และวางกรอบให้เห็นว่ายังมีอะไรที่แก้ไม่ตก มันยกระดับบทสนทนาเหนือฟีเจอร์รายชิ้นอย่าง Skills หรือ MCP servers ไปสู่การออกแบบระบบควบคุมเชิงกลยุทธ์ที่สร้างความมั่นใจต่อผลลัพธ์ของ agent ได้จริง

ตัวอย่างจากการใช้งานในปัจจุบัน ได้แก่:

- [ทีม OpenAI ได้บันทึก Harness ของตนไว้](https://openai.com/index/harness-engineering/): ใช้ custom linters และการทดสอบโครงสร้างบังคับสถาปัตยกรรมแบบแบ่งชั้น พร้อม "garbage collection" ที่ทำงานเป็นระยะเพื่อตรวจ drift และให้ agents เสนอการแก้ไข ข้อสรุปของทีมคือความท้าทายที่ยากที่สุดในตอนนี้อยู่ที่การออกแบบสภาพแวดล้อม feedback loops และระบบควบคุม
- [บทความของ Stripe เกี่ยวกับ minions](https://stripe.dev/blog/minions-stripes-one-shot-end-to-end-coding-agents) อธิบาย pre-push hooks ที่เลือก linters ที่เกี่ยวข้องด้วย heuristics เน้นการย้าย feedback ไปทางซ้าย และใช้ "blueprints" เพื่อรวม feedback sensors เข้ากับ workflows ของ agent
- Mutation testing และการทดสอบโครงสร้างเป็น computational feedback sensors ที่เคยถูกใช้น้อยเกินไป แต่กำลังกลับมาได้รับความสนใจ
- การผสาน LSP และ code intelligence เป็นตัวอย่างของ computational feedforward guides
- ทีม Thoughtworks กำลังผสาน computational และ inferential sensors เพื่อจัดการ architectural drift รวมถึงการจับคู่ agents กับ custom linters และแนวทางแบบ "janitor army" สำหรับดูแลคุณภาพโค้ด

ยังมีคำถามอีกมาก เมื่อ Harness เติบโตขึ้น จะรักษา Guides และ Sensors ให้สอดคล้องกันได้อย่างไร เราจะไว้ใจให้ agent ชั่งน้ำหนักคำสั่งและสัญญาณ feedback ที่ขัดแย้งกันได้มากเพียงใด หาก Sensor ไม่เคยส่งสัญญาณเลย แปลว่าคุณภาพดีหรือความสามารถในการตรวจจับอ่อนเกินไป เราต้องมีวิธีประเมิน coverage และคุณภาพของ Harness ที่เทียบเคียงได้กับ code coverage และ mutation testing ปัจจุบัน feedforward กับ feedback ยังกระจัดกระจายอยู่ตามขั้นตอนต่าง ๆ ของการส่งมอบ จึงยังมีพื้นที่สำหรับเครื่องมือที่กำหนดค่า ประสาน และให้เหตุผลเกี่ยวกับกลไกเหล่านี้ในฐานะระบบเดียว

การสร้าง Harness ชั้นนอกกำลังกลายเป็นแนวปฏิบัติทางวิศวกรรมที่ต้องทำอย่างต่อเนื่อง ไม่ใช่การตั้งค่าเพียงครั้งเดียว
