## บทนำ

วิศวกรบางคนพูดกันว่าแคชควบคุมทุกสิ่งรอบตัว สำหรับ Agent ที่ทำงานต่อเนื่องเป็นเวลานาน เรื่องนี้ก็จริงเช่นกัน

ผลิตภัณฑ์อย่าง Claude Code คุ้มค่าพอที่จะให้บริการได้ เพราะการแคชพรอมป์ (Prompt Caching) ช่วยให้คำขอในรอบหลังนำการประมวลผลจากรอบก่อนกลับมาใช้ซ้ำ การใช้ซ้ำนี้ช่วยลดทั้งเวลาแฝงและต้นทุน โดยเฉพาะเมื่อบทสนทนายาวขึ้นเรื่อย ๆ

## สถาปัตยกรรมแคชของ Claude Code

Claude Code ได้รับการออกแบบโดยมีการแคชพรอมป์เป็นแกนหลัก อัตรา cache hit ของพรอมป์ที่สูงช่วยลดต้นทุนการดำเนินงาน และทำให้บริการแบบสมัครสมาชิกกำหนด rate limit ได้ผ่อนคลายขึ้น ทีมติดตามอัตรานี้อย่างใกล้ชิดจนการลดลงอย่างรุนแรงอาจถูกจัดการในระดับ incident

หัวข้อต่อไปนี้สรุปบทเรียนจาก production ที่ทีมได้รับระหว่างปรับแต่งการแคชพรอมป์ในระบบขนาดใหญ่

![แผนภาพสถาปัตยกรรมการแคชพรอมป์](https://pbs.twimg.com/media/HBipHa1boAAXD_A?format=jpg&name=large)

## การแคชพรอมป์ทำงานอย่างไร

### การจับคู่ prefix

การแคชพรอมป์ทำงานด้วย **การจับคู่ prefix** หากเนื้อหาตั้งแต่ต้นคำขอไปจนถึง cache breakpoints ไม่เปลี่ยน API ก็สามารถนำผลการประมวลผลของส่วนนั้นกลับมาใช้ซ้ำได้

ดังนั้นลำดับจึงสำคัญ ยิ่งคำขอจำนวนมากมีส่วนเริ่มต้นเหมือนกัน ก็ยิ่งนำงานที่แคชไว้กลับมาใช้ได้มากขึ้น

### ลำดับที่เป็นมิตรต่อแคชของ Claude Code

Claude Code ใช้โครงสร้างที่เป็นมิตรต่อแคช: **วางเนื้อหาที่เสถียรไว้ก่อน และเนื้อหาแบบไดนามิกไว้ท้ายสุด**

คำขอถูกจัดลำดับโดยประมาณดังนี้:

1. **พรอมป์ระบบและเครื่องมือที่เสถียร** (ใช้ร่วมกันได้ในวงกว้าง)
2. **บริบทของโปรเจกต์** (ใช้ร่วมกันภายในโปรเจกต์)
3. **บริบทของ session** (ใช้ร่วมกันภายใน session)
4. **ข้อความในบทสนทนา**

การจัดเรียงนี้เพิ่มโอกาสให้คำขอและ session ต่าง ๆ มี prefix ที่ใช้ซ้ำร่วมกันได้

### เหตุใดลำดับนี้จึงเปราะบาง

การเปลี่ยนแปลงที่ดูไม่เป็นอันตรายก็อาจทำให้ prefix ขาดได้ ตัวอย่างเช่น:

- ใส่ timestamp ที่ละเอียดลงในพรอมป์ระบบส่วนที่ควรเสถียร
- ส่งคำจำกัดความของเครื่องมือออกมาในลำดับที่ไม่แน่นอน
- เปลี่ยนพารามิเตอร์ของเครื่องมือ เช่น กำหนดใหม่ว่าเครื่องมือ Agent สามารถเรียก Agent ใดได้บ้าง

## การรักษาแคชให้ใช้ได้

### การอัปเดตข้อมูลที่ล้าสมัย

ข้อมูลบางอย่างในพรอมป์ย่อมล้าสมัยตามธรรมชาติ วันที่เปลี่ยน ไฟล์ถูกแก้ไข หรือสถานะอื่นใน runtime เปลี่ยนไป

การแก้พรอมป์ระบบก่อนหน้าอาจดูเป็นระเบียบ แต่จะทำให้ prefix เปลี่ยน และทำให้ทุกสิ่งที่ตามมาเกิด cache miss

รูปแบบที่ Claude Code ใช้คือส่งข้อมูลอัปเดตในข้อความภายหลัง ตัวอย่างเช่น ข้อความผู้ใช้หรือผลลัพธ์จากเครื่องมือรายการถัดไปอาจมี `<system-reminder>` ระบุว่าตอนนี้เป็นวันพุธ วิธีนี้ยังคงใช้ prefix เดิมซ้ำได้ ขณะเดียวกันโมเดลก็ได้รับข้อมูลปัจจุบัน

## กับดักของการเปลี่ยนโมเดล

### แคชผูกกับโมเดลแต่ละตัว

แคชพรอมป์ผูกกับโมเดลแต่ละตัว ทำให้การคำนวณต้นทุนไม่ตรงไปตรงมาอย่างที่เห็นในตอนแรก

หากบทสนทนามีโทเคน 100k ที่แคชไว้สำหรับ Opus แล้ว การถามคำถามง่าย ๆ กับ Opus อีกหนึ่งข้ออาจมีต้นทุนต่ำกว่าการเปลี่ยนไปใช้ Haiku เพราะ Haiku ต้องสร้างแคชพรอมป์ใหม่สำหรับประวัติทั้งหมด

### ใช้ subagent สำหรับการส่งต่องานข้ามโมเดล

เมื่ออีกโมเดลเหมาะกับงานมากกว่า Claude Code จะเลือกส่งต่องานผ่าน **subagent** แทนการเปลี่ยนโมเดลของบทสนทนาปัจจุบัน Opus สามารถจัดเตรียมคำอธิบายงานแบบกระชับสำหรับโมเดลอีกตัวได้

Explore agents เป็นตัวอย่างที่พบบ่อย พวกมันใช้ Haiku ได้โดยไม่ต้องทิ้งแคชที่ผูกกับโมเดลของบทสนทนาหลัก

## เหตุใดการเปลี่ยนเครื่องมือจึงมีต้นทุนสูง

การเปลี่ยนชุดเครื่องมือกลางบทสนทนาเป็นอีกวิธีที่พบบ่อยในการทำลายการใช้แคชพรอมป์ซ้ำ

การเปิดให้ใช้เฉพาะเครื่องมือที่จำเป็นในขณะนั้นอาจดูมีประสิทธิภาพ แต่คำจำกัดความของเครื่องมือเป็นส่วนหนึ่งของ prefix ที่แคชไว้ ดังนั้นการเพิ่มหรือนำเครื่องมือออกจะทำให้ prefix ของบทสนทนาหลังจากจุดนั้นใช้ไม่ได้

### Plan Mode: แสดงสถานะโดยไม่เปลี่ยนเครื่องมือ

Plan Mode แสดงให้เห็นว่า Claude Code ออกแบบฟีเจอร์ภายใต้ข้อจำกัดนี้อย่างไร

วิธีที่เห็นได้ชัดคือเปลี่ยนชุดเครื่องมือปกติเป็นเครื่องมือแบบอ่านอย่างเดียวเมื่อผู้ใช้เข้าสู่ Plan Mode แต่การเปลี่ยน tool schema เช่นนั้นจะทำลายแคช

Claude Code จึงคงเครื่องมือไว้เหมือนเดิม และรวม `EnterPlanMode` กับ `ExitPlanMode` เป็นเครื่องมือปกติ จากนั้นข้อความระบบที่เพิ่มภายหลังจะบอก Agent ว่าขณะนี้อยู่ใน Plan Mode: ให้ตรวจดู codebase ห้ามแก้ไขไฟล์ และเรียก `ExitPlanMode` เมื่อวางแผนเสร็จ คำจำกัดความของเครื่องมือไม่เปลี่ยน

เนื่องจาก `EnterPlanMode` เป็นเครื่องมือ โมเดลจึงเข้าสู่ Plan Mode ได้เองเมื่อเห็นว่าปัญหาต้องอาศัยการวางแผนเชิงลึก โดยไม่ทำให้ prefix ที่แคชไว้ใช้ไม่ได้

### Tool Search: ใช้ deferred loading แทนการนำเครื่องมือออก

หลักการเดียวกันใช้กับ Tool Search ได้ Claude Code อาจมีเครื่องมือ MCP หลายสิบรายการ แต่การส่ง tool schema ฉบับเต็มทั้งหมดในทุกคำขอย่อมมีต้นทุนสูง ขณะเดียวกันการนำเครื่องมือออกกลางบทสนทนาก็ยังทำลายแคช

วิธีแก้คือ `defer_loading` ซึ่งหมายถึงการเลื่อนโหลด schema ฉบับเต็มออกไป Claude Code จะส่ง tool stub ที่เสถียรและมีขนาดเล็ก พร้อมกำกับด้วย `defer_loading: true` เมื่อจำเป็น โมเดลจึงใช้ `ToolSearch` เพื่อโหลด tool schema ฉบับเต็ม stub ชุดเดิมยังคงอยู่ในลำดับเดิม จึงรักษา prefix ไว้ได้

API เปิดให้ใช้ `ToolSearch` เพื่อให้แอปพลิเคชันอื่นนำรูปแบบเดียวกันไปใช้ได้

## Compaction และการแคช

![แผนภาพ compaction และการแคช](https://pbs.twimg.com/media/HBitEdRbUAMVSnM?format=jpg&name=large)

Compaction คือการย่อบริบทเมื่อบทสนทนาใกล้ถึงขีดจำกัดของ context window ระบบจะสร้างสรุป แล้วสนทนาต่อโดยใช้ตัวแทนข้อมูลที่เล็กลง

กระบวนการนี้ทำให้เกิดกรณีขอบของการแคชพรอมป์หลายอย่าง

### ปัญหา

โมเดลต้องใช้ประวัติบทสนทนาเพื่อสร้างสรุป วิธีทำแบบตรงไปตรงมาอาจส่งคำขอแยกต่างหาก โดยใช้พรอมป์ระบบคนละชุดและไม่มีเครื่องมือ คำขอนั้นจะไม่ตรงกับ prefix ของบทสนทนาหลักอีกต่อไป โทเคนขาเข้าทั้งหมดจึงต้องประมวลผลด้วยราคาเต็ม

### วิธีแก้: fork ที่ไม่ทำลายแคช

Claude Code ปฏิบัติต่อ compaction เสมือนเป็น fork ที่ไม่ทำลายแคช คำขอสำหรับ compaction ใช้พรอมป์ระบบ บริบทของผู้ใช้และระบบ คำจำกัดความของเครื่องมือ และประวัติบทสนทนาชุดเดียวกับคำขอหลัก แล้วจึงเพิ่มคำสั่ง compaction เป็นข้อความผู้ใช้ใหม่ไว้ท้ายสุด

หากพิจารณาเฉพาะการคำนวณ input prefix คำขอนี้ใช้ prefix เครื่องมือ และประวัติร่วมกับคำขอหลัก จึงนำ prefix ที่แคชไว้กลับมาใช้ได้ โดยหลักแล้วมีเพียง input tokens ของคำสั่ง compaction ที่เพิ่มใหม่เท่านั้นที่ต้องประมวลผลใหม่ อย่างไรก็ตาม โมเดลยังต้องสร้างสรุปจริง ดังนั้นการคำนวณส่วนนั้นและ output tokens ยังคงมีค่าใช้จ่าย

แนวทางนี้ยังต้องมี compaction buffer ซึ่งคือพื้นที่ว่างใน context window ที่เพียงพอสำหรับคำสั่ง compaction ที่เพิ่มเข้ามาและสรุปที่โมเดลจะสร้าง

## ห้าบทเรียนที่ได้รับ

รายละเอียดของ compaction มีความละเอียดอ่อน แต่บทเรียนในภาพรวมใช้ได้กับ Agent ทุกตัวที่สร้างบนการแคชพรอมป์

<div class="info-box">

**1. การแคชพรอมป์คือการจับคู่ prefix**

การเปลี่ยนแปลงใด ๆ ภายใน prefix จะทำให้ทุกสิ่งหลังจากนั้นใช้ไม่ได้ จึงควรออกแบบคำขอโดยยึดลำดับที่เสถียรตั้งแต่ต้น

</div>

<div class="tip-box">

**2. ส่งข้อมูลอัปเดตเป็นข้อความ**

เมื่อข้อมูลใดข้อมูลหนึ่งเปลี่ยนไป ไม่ว่าจะเป็นวันที่ สถานะของ runtime หรือโหมดการทำงาน ให้เพิ่มข้อความใหม่แทนการเขียนพรอมป์ระบบก่อนหน้าทับใหม่

</div>

<div class="warning-box">

**3. อย่าเปลี่ยนโมเดลหรือเครื่องมือกลางบทสนทนา**

ใช้การส่งต่องานเมื่อต้องเปลี่ยนโมเดล ใช้เครื่องมือสำหรับการเปลี่ยนสถานะ และใช้ deferred loading สำหรับแค็ตตาล็อกเครื่องมือขนาดใหญ่

</div>

<div class="info-box">

**4. ติดตามอัตรา cache hit ของพรอมป์เหมือนติดตาม uptime**

ความเปลี่ยนแปลงเพียงไม่กี่เปอร์เซ็นต์อาจส่งผลต่อต้นทุนและเวลาแฝงอย่างมีนัยสำคัญ การถดถอยของแคชจึงควรมีการแจ้งเตือนด้านปฏิบัติการ

</div>

<div class="tip-box">

**5. งานที่ fork ออกไปควรรักษา prefix ของงานหลัก**

Compaction การสรุป และการคำนวณเสริมอื่น ๆ ควรนำรูปแบบคำขอที่ไม่ทำลายแคชของงานหลักกลับมาใช้ทุกครั้งที่ทำได้

</div>
