## 들어가며

엔지니어들은 캐시가 시스템의 성패를 좌우한다고 말하곤 합니다. 장시간 실행되는 에이전트도 마찬가지입니다.

Claude Code 같은 제품이 경제적으로 운영될 수 있는 이유는 프롬프트 캐싱 덕분입니다. 이후 요청이 앞선 대화에서 이미 수행한 계산을 재사용할 수 있어, 대화가 길어질수록 지연 시간과 비용을 크게 줄일 수 있습니다.

## Claude Code의 캐싱 아키텍처

Claude Code는 프롬프트 캐싱을 중심으로 설계됐습니다. 프롬프트 캐시 적중률이 높으면 운영 비용이 낮아지고 구독 요금제에서 더 넉넉한 사용량 제한을 제공할 수 있습니다. 팀은 이 지표를 면밀히 모니터링하며, 심각한 하락은 장애로 취급할 정도입니다.

아래에서는 대규모 프롬프트 캐싱 최적화 과정에서 얻은 운영상의 교훈을 정리합니다.

![프롬프트 캐싱 아키텍처 다이어그램](https://pbs.twimg.com/media/HBipHa1boAAXD_A?format=jpg&name=large)

## 프롬프트 캐싱의 작동 방식

### 프리픽스 매칭

프롬프트 캐싱은 **프리픽스 일치** 방식으로 작동합니다. 요청 시작부터 캐시 중단점까지의 프리픽스가 바뀌지 않았다면 API는 그 구간에서 이미 수행한 계산을 재사용할 수 있습니다.

따라서 순서가 중요합니다. 더 많은 요청이 같은 시작 부분을 공유할수록 재사용할 수 있는 캐시도 많아집니다.

### Claude Code의 캐시 친화적 순서

Claude Code는 **안정적인 콘텐츠를 앞에, 동적인 콘텐츠를 뒤에 배치**하는 캐시 친화적 구성을 사용합니다.

요청은 대략 다음 순서로 구성됩니다.

1. **안정적인 시스템 프롬프트와 도구**(여러 요청에서 폭넓게 공유)
2. **프로젝트 컨텍스트**(프로젝트 안에서 공유)
3. **세션 컨텍스트**(세션 안에서 공유)
4. **대화 메시지**

이 배치는 여러 요청과 세션이 재사용 가능한 프리픽스를 공유할 가능성을 높입니다.

### 이 순서가 쉽게 깨지는 이유

겉보기에는 사소한 변경도 프리픽스를 깨뜨릴 수 있습니다. 예를 들면 다음과 같습니다.

- 안정적인 시스템 프롬프트에 정확한 타임스탬프를 넣는 경우
- 도구 정의를 비결정적인 순서로 내보내는 경우
- 에이전트 도구가 호출할 수 있는 에이전트 목록처럼 도구 파라미터를 변경하는 경우

## 캐시를 유효하게 유지하기

### 오래된 정보 갱신하기

프롬프트 안의 일부 정보는 자연스럽게 오래됩니다. 날짜가 바뀌고, 파일이 수정되며, 다른 런타임 상태도 계속 변합니다.

앞쪽 시스템 프롬프트를 직접 고치는 방식은 깔끔해 보이지만 프리픽스를 변경해 그 뒤의 모든 캐시를 무효화합니다.

Claude Code는 업데이트를 뒤쪽 메시지로 보냅니다. 예를 들어 다음 사용자 메시지나 도구 결과에 이제 수요일이라는 `<system-reminder>`를 넣을 수 있습니다. 기존 프리픽스는 재사용하면서 모델에는 최신 정보를 전달할 수 있습니다.

## 모델 전환의 함정

### 캐시는 모델별로 구분된다

프롬프트 캐시는 모델별로 분리되기 때문에 비용 계산이 처음 생각보다 직관적이지 않을 수 있습니다.

Opus와의 대화에 이미 100k 토큰이 캐시돼 있다면, 간단한 질문 하나를 더 할 때 Haiku로 바꾸는 것보다 Opus를 그대로 쓰는 편이 저렴할 수 있습니다. Haiku는 전체 대화 기록에 대해 새로운 프롬프트 캐시를 만들어야 하기 때문입니다.

### 모델 인계에는 서브에이전트를 사용한다

다른 모델이 적합할 때 Claude Code는 기존 대화의 모델을 바꾸기보다 **서브에이전트**로 작업을 인계합니다. Opus가 다른 모델을 위한 간결한 작업 설명을 준비할 수 있습니다.

Explore 에이전트가 대표적인 예입니다. 상위 대화에 쌓인 모델별 캐시를 버리지 않으면서 Haiku를 사용할 수 있습니다.

## 도구 변경의 비용이 큰 이유

대화 도중 도구 집합을 변경하는 것도 프롬프트 캐시 재사용을 무너뜨리는 흔한 원인입니다.

지금 필요한 도구만 노출하면 효율적으로 보일 수 있습니다. 하지만 도구 정의는 캐시된 프리픽스의 일부이므로, 도구를 추가하거나 제거하면 그 뒤의 대화 프리픽스가 무효화됩니다.

### Plan Mode: 도구를 바꾸지 않고 상태를 표현한다

Plan Mode는 Claude Code가 이 제약을 고려해 기능을 설계한 사례입니다.

가장 단순한 구현은 사용자가 Plan Mode에 들어갈 때 일반 도구 집합을 읽기 전용 도구로 교체하는 것입니다. 그러나 이런 도구 스키마 변경은 캐시를 깨뜨립니다.

대신 Claude Code는 도구를 안정적으로 유지하고 `EnterPlanMode`와 `ExitPlanMode`를 일반 도구로 포함합니다. 이후 시스템 메시지로 현재 Plan Mode이며, 코드베이스를 살펴보되 파일은 수정하지 말고 계획이 끝나면 `ExitPlanMode`를 호출하라고 에이전트에 알립니다. 도구 정의 자체는 바뀌지 않습니다.

`EnterPlanMode` 자체가 도구이므로 모델이 더 깊은 계획이 필요하다고 판단했을 때도 캐시된 프리픽스를 무효화하지 않고 스스로 Plan Mode에 들어갈 수 있습니다.

### Tool Search: 도구를 제거하지 않고 지연 로딩한다

Tool Search에도 같은 원칙이 적용됩니다. Claude Code에는 수십 개의 MCP 도구가 있을 수 있지만, 요청마다 전체 도구 스키마를 모두 보내는 것은 비쌉니다. 그렇다고 대화 중간에 도구를 제거하면 여전히 캐시가 깨집니다.

해법은 `defer_loading`입니다. Claude Code는 `defer_loading: true`로 표시한 안정적이고 가벼운 도구 스텁을 전송합니다. 필요할 때 모델이 `ToolSearch`를 사용해 전체 도구 스키마를 불러옵니다. 같은 스텁이 같은 순서로 유지되므로 프리픽스도 보존됩니다.

API 역시 `ToolSearch`를 제공하므로 애플리케이션도 같은 패턴을 사용할 수 있습니다.

## 컴팩션과 캐싱

![컴팩션과 캐싱 다이어그램](https://pbs.twimg.com/media/HBitEdRbUAMVSnM?format=jpg&name=large)

컴팩션은 대화가 컨텍스트 윈도 한도에 가까워졌을 때 일어납니다. 시스템이 요약을 만들고 더 작은 표현으로 대화를 이어 갑니다.

이 과정에는 프롬프트 캐싱과 관련된 몇 가지 경계 사례가 있습니다.

### 문제

요약을 만들려면 모델이 대화 기록을 봐야 합니다. 단순한 구현에서는 다른 시스템 프롬프트를 사용하고 도구를 제외한 별도 요청을 만들 수 있습니다. 그러면 메인 대화의 프리픽스와 일치하지 않아 모든 입력 토큰을 정상 비용으로 다시 처리해야 합니다.

### 해법: 캐시를 유지하는 분기

Claude Code는 컴팩션을 캐시를 유지하는 분기로 취급합니다. 컴팩션 요청은 상위 요청과 동일한 시스템 프롬프트, 사용자 및 시스템 컨텍스트, 도구 정의, 대화 기록을 사용하고, 마지막에 컴팩션 지시를 새 사용자 메시지로 추가합니다.

입력 프리픽스 처리 관점에서는 캐시된 프리픽스를 재사용할 수 있어, 새로 추가한 컴팩션 지시 입력을 주로 다시 처리하면 됩니다. 다만 요약 출력은 여전히 생성해야 하며, 그 계산과 출력 토큰에는 비용이 듭니다.

이를 위해서는 컴팩션용 여유 공간도 필요합니다. 컨텍스트 윈도에 추가된 컴팩션 지시와 모델이 생성할 요약을 담을 충분한 공간을 남겨 둬야 합니다.

## 다섯 가지 교훈

컴팩션의 세부 동작은 까다롭지만, 여기서 얻은 교훈은 프롬프트 캐싱을 사용하는 모든 에이전트에 적용됩니다.

<div class="info-box">

**1. 프롬프트 캐싱은 프리픽스 일치로 작동한다**

프리픽스 안의 내용이 바뀌면 그 뒤의 캐시가 모두 무효화됩니다. 처음부터 순서가 안정적으로 유지되도록 요청 구조를 설계해야 합니다.

</div>

<div class="tip-box">

**2. 업데이트는 메시지로 보낸다**

날짜, 런타임 상태, 모드가 바뀌면 앞쪽 시스템 프롬프트를 다시 쓰지 말고 메시지를 추가합니다.

</div>

<div class="warning-box">

**3. 대화 중간에 모델이나 도구를 바꾸지 않는다**

모델 변경에는 인계, 상태 전환에는 도구, 대규모 도구 목록에는 지연 로딩을 사용합니다.

</div>

<div class="info-box">

**4. 프롬프트 캐시 적중률을 가용성 지표처럼 모니터링한다**

몇 퍼센트포인트의 변화도 비용과 지연 시간에 실질적인 영향을 줄 수 있으므로 캐시 회귀에는 운영 알림이 필요합니다.

</div>

<div class="tip-box">

**5. 분기된 작업도 상위 요청의 프리픽스를 유지한다**

컴팩션, 요약, 그 밖의 부수 계산은 가능한 한 상위 요청의 캐시 친화적 형태를 재사용해야 합니다.

</div>

## 맺음말

Claude Code는 처음부터 프롬프트 캐싱을 중심으로 만들어졌습니다. 실질적인 교훈은 모든 에이전트가 하나의 고정된 구성을 그대로 복사해야 한다는 것이 아닙니다. 캐시 안정성을 핵심 아키텍처 제약으로 다뤄야 한다는 것입니다.
