많은 Agent 기사에서는 **모델이 작업을 완료하도록 만드는 방법**과 **시스템을 안정적으로 작성하는 방법**이라는 두 가지 사항을 함께 설명합니다. 그러다보니 명사도 많이 기억나고, 실제로 디자인을 시작하게 되면 어떤 문제를 먼저 해결해야 할지 모르겠더라고요.

이 문서에서는 에이전트가 실제 환경에서 판단하고, 행동하고, 협력하고, 제어 가능한 상태를 유지하는 방법에 대해서만 답변합니다. 단일 에이전트의 추론 루프에서 시작하여 점차적으로 여러 사람의 협업, 메모리 및 지식을 입력하고 마지막으로 생산 작업에 필요한 안전 및 내결함성을 추가하는 작업 맵이라고 생각할 수 있습니다.

읽을 때 스키마 이름을 기억할 필요는 없습니다. 각 섹션에서 실제로 답변하고 싶은 내용은 다음과 같습니다. **이런 종류의 작업에서 어떤 종류의 불확실성을 제거해야 합니까? ** 때로는 다음에 무엇을 할 것인지, 때로는 누가 할 것인지, 때로는 기억하는 방법, 때로는 돌이킬 수 없는 결과를 피하는 방법입니다.

## 첫 번째 계층: 단일 에이전트 코어 모드

### 01 ReAct — 추론과 행동이 얽혀 있음

**논문**: Yao et al., 2022 — _ReAct: 언어 모델에서 추론과 행동의 시너지 효과_

💡 핵심 아이디어

먼저 생각하고 나서 하지 말고, 하면서 생각해보세요. LLM은 **생각**과 **행동**을 번갈아 생성하고 결과를 관찰한 후 계속 추론합니다.

> 생각 : 사용자는 오늘 베이징의 날씨를 알고 싶어합니다. 검색해야 해요.
> Action: search("2026년 4월 9일 베이징 날씨")
> 관측 : 맑음, 18°C, 북풍 3도
> 생각: 날씨 데이터가 있으니까 대답할 수 있겠네요.
> 답: 오늘 베이징은 날씨가 맑고 기온은 18°C, 북풍은 3도입니다.

**작동하는 이유**:

- **추론이 행동을 안내합니다**: 각 행동 단계에는 맹목적인 시도를 줄이기 위해 "이 작업을 수행해야 하는 이유"를 설명하는 생각이 있습니다.
- **행동 피드백 추론**: 관찰은 실제 정보를 제공하여 후속 추론을 더욱 정확하게 만듭니다.
- **설명 가능성**: Thought 링크 자체는 자연스러운 "결정 로그"입니다.

**적용 가능한 시나리오**: 다단계 추론 + 도구 호출(질문 및 답변, 분석, 디버깅)이 필요한 작업

**제한 사항**: 긴 체인은 "편차"가 발생하기 쉽고 후속 관찰이 원래 목표에서 벗어날 수 있습니다. 방향을 제한하려면 Reflection 또는 Planning 모드와 협력해야 합니다.

### 02 계획 및 실행 — 행동을 취하기 전에 명확하게 생각하십시오.

💡 핵심 아이디어

단계별로 지켜보는 대신 완전한 실행 계획을 생성한 다음 단계별로 실행해 보세요. ReAct의 "생각과 행동"과 달리 **계획**과 **실행**은 명확하게 두 단계로 구분됩니다.

> [기획단계]
> 사용자: 경쟁 제품 HireGo의 최신 개발 분석을 도와주세요.
>계획:
> 1. HireGo 공식 홈페이지를 검색하여 제품정보를 확인하세요.
> 2. 구글 플레이/앱스토어 리뷰 검색
> 3. 최근 30일간 관련 보도에 대한 검색 기술 매체
> 4. LinkedIn에서 채용 업데이트 검색 (팀 확장 방향 결정)
> 5. 위 정보를 바탕으로 경쟁제품 인텔리전스 브리핑 작성
>
> [실행단계]
> 1단계 수행 → 관찰 → 2단계 수행 → 관찰 → ... → 최종 보고서

**주요 디자인**:

- **계획은 동적으로 조정될 수 있습니다**: 2단계를 수행할 때 새로운 단서를 발견하면 다시 돌아가서 1단계의 검색을 보완할 수 있습니다.
- **ReAct 보완**: 계획 및 실행은 "글로벌 비전"을 제공하고 ReAct는 "지역적 유연성"을 제공합니다.
- **실제 프로젝트에서 종종 혼합됨**: 먼저 대략적인 계획을 생성하고 각 단계에서 ReAct 모드로 실행합니다.

**적용 가능한 시나리오**: 명확한 프로세스가 포함된 다단계 작업(연구 보고서, 데이터 처리 파이프라인, 코드 리팩토링)

**제한사항**: 초기 계획은 완벽하지 않을 수 있습니다. 환경이 급격하게 변하면 정적 계획은 구식이 됩니다.

### 03 반성 — 마무리 후 되돌아보기

💡 핵심 아이디어

에이전트가 **자체 출력을 평가**하고 결함이 발견되면 스스로 수정하도록 합니다. 이것은 가장 "인간적인" 모드일 수 있습니다. 문제를 끝내고 다시 확인한 다음 실수를 발견하고 수정하세요.

> [1차 출력]
> "NVIDIA의 2026년 1분기 매출은 미화 280억 달러로 전년 대비 78% 증가했습니다."
>
> [자기성찰]
> 잠깐, 확인해 보겠습니다. 작년 1분기 280억 대 260억...
> 280/260 - 1 = 7.7%, 78%가 아님. 78%는 2024년 1분기 대비 전년 대비입니다.
> 비교 기준선을 명확하게 표시해야 합니다.
>
> [보정 출력]
> "NVIDIA의 2026년 1분기 매출은 280억 달러로 전년 동기 대비 7.7%(분기 대비), 전년 동기 대비 78% 증가했습니다."

**Andrew Ng의 전형적인 사례**: LLM에서 코드 작성 → 테스트 실행 → LLM에 오류 피드백 → LLM에서 코드 수정 → 다시 테스트합니다. 이 주기는 통과될 때까지 여러 번 반복될 수 있습니다.

**구현 포인트**:

- 피드백 신호는 **외부**(테스트 결과, 사용자 피드백, 도구 반환 값) 또는 **내부**(LLM 자체 평가)일 수 있습니다.
- 반영 빈도를 제어해야 합니다. 매번 반영하는 것은 시간 낭비이며, 핵심 노드에서만 반영합니다(코드 작성 및 보고서 생성 후).
- 명확한 '완료 기준'이 필요합니다. 그렇지 않으면 에이전트가 무한 루프에 '반영'합니다.

**적용 가능한 시나리오**: 코드 생성, 작성, 데이터 분석 - 모든 "품질이 중요"한 작업

### 04 도구 사용 - 에이전트가 팔과 다리를 성장시키도록 합니다.

💡 핵심 아이디어

LLM 자체는 **함수 호출/도구 사용**을 통해 실제 도구에서 작동할 수 있게 해주는 텍스트 생성기일 뿐입니다. 이는 모든 에이전트 시스템의 **인프라 구조**입니다. 도구를 사용하지 않으면 에이전트도 존재하지 않습니다.

> 사용자: "내일 아침 오전 9시에 회의가 있다는 알림을 만들어 주세요."
>
> LLM 내부 의사결정:
> 전화가 필요합니다 → cron_schedule(
> 메시지: "알림: 9시에 모임",
> 시간: "2026-04-10T09:00:00+08:00"
> )
>
> → 통화 성공 → "알림이 설정되었습니다: 내일 9:00 - 회의를 상기시키기 위해"

**도구 분류**:

| 카테고리 | 예 | 특징 |
| --- | --- | --- |
| **정보검색** | 웹\_검색, 웹\_페치, 읽기\_파일 | 읽기 전용, 부작용 없음 |
| **작업 실행** | send\_message, write\_file, exec | 부작용이 있으니 주의해주세요 |
| **대화형 도구** | 브라우저, 메시지, 캔버스 | 양방향 커뮤니케이션 |
| **계산기** | 계산기, 코드\_인터프리터 | 결정적 출력 |

**엔지니어링 실습**:

- **도구 설명의 질이 통화의 질을 결정합니다**: LLM이 도구를 올바르게 선택할 수 있는지 여부는 도구 설명이 잘 작성되었는지에 90% 달려 있습니다.
- **최소 권한의 원칙**: 에이전트는 현재 작업에 필요한 도구만 필요하며 한꺼번에 제공하지 않습니다.
- **오류 처리가 핵심**: 도구 호출 실패가 일반적이며 에이전트는 실패를 정상적으로 처리할 수 있어야 합니다(재시도, 대체, 보고).

### 05 생각의 사슬 & 생각의 나무

**생각의 사슬**: LLM에게 직접 답변을 제공하는 대신 "사고 과정을 작성"하도록 요청하세요.

> 나쁜 습관: "정답은 42"(블랙박스)
> 모범 사례: "x를...로 두고 공식에 대입하면...을 얻으므로 답은 42입니다."(과정이 표시됨)

이는 독립적인 에이전트 모드는 아니지만 ReAct의 사고, 계획 및 실행의 계획, Reflection의 평가 등 모든 모드의 기본 인프라가 모두 CoT에 의존합니다.

**생각의 나무**: CoT의 업그레이드 버전 - 단지 하나의 사고 방식이 아니라 **다양한 경로를 탐색하고 최선의 경로를 선택**하세요.

> 질문: "검색 API 대기 시간을 2초에서 200ms로 줄이는 방법은 무엇입니까?"
>
> 경로 A: 더 빠른 API 제공업체로 변경 → 비용 3배 증가 → 권장하지 않음
> 경로 B: 캐시 레이어 추가 → 적중률 알 수 없음 → 평가 필요
> 경로 C: 동시에 여러 API를 요청하고 가장 빠른 반환을 얻음 → 복잡성 증가 → 시도해 볼 가치가 있음
>
> 평가: 비용을 통제할 수 있으면 경로 C가 목표를 달성할 수 있음 → 경로 C 선택

**적용 가능한 시나리오**: 여러 솔루션(아키텍처 설계, 솔루션 선택) 탐색이 필요한 복잡한 결정

## 두 번째 수준: 다중 에이전트 공동 작업 모드

단일 에이전트의 역량이 충분하지 않은 경우 여러 전문 에이전트가 협력해야 합니다. 이것이 오케스트레이션의 영역에 들어갑니다.

### 06 감독자 — 감독자가 작업을 할당합니다.

"감독 에이전트"는 사용자의 의도를 이해하고, 작업을 세분화하여 전문 에이전트에게 배포하여 실행하는 역할을 담당합니다.

> 사용자: "AI 산업 주간 보고서 작성을 도와주세요"
>
> 감독 에이전트:
> → 정보수집요원 : "이번주 AI업계 주요 뉴스 모아보세요"
> → 금융대리점 : "이번주 AI 관련 기업 주가동향을 모아보세요"
> → 연구요원 : "이번 주 중요한 논문을 모아라"
> ← 모든 Agent의 출력을 요약 → 주간 보고서로 통합 → 사용자에게 전송

💡 **일반적인 관행**은 감독 에이전트가 정보 수집, 금융, 연구 등 전문 에이전트를 조정하는 것입니다.

**장점**: 명확한 책임, 각 에이전트는 자신의 영역에 집중합니다.
**단점**: 감독자는 단일 지점 병목 현상이 발생합니다. 작업 분해가 잘못되면 전체 파이프라인이 궤도를 벗어나게 됩니다.

### 07 계층적 — 계층적 관리

Supervisor의 업그레이드 버전 - **다단계 명령 체계**.

```text
协调 Agent
├── 产品负责人 Agent — 负责产品相关任务
│   ├── 竞品分析 Agent — 数据搜集
│   └── 用户研究 Agent — 数据搜集
├── 技术负责人 Agent — 负责技术任务
│   ├── 前端开发 Agent
│   └── 后端开发 Agent
└── 运营负责人 Agent — 负责运营任务
    ├── 数据分析 Agent
    └── 内容创作 Agent
```

**감독자와의 차이점**: 감독자는 "단순"합니다. 한 명의 상사가 모든 사람을 직접 관리합니다. 계층 구조는 더 큰 작업을 처리할 수 있는 중간 관리 계층이 있는 "트리 모양"입니다.

### 08 Swarm — 분산형 꿀벌 떼

중앙 통제가 없으며 상담원 간 협업은 **Handoff**를 통해 완료됩니다.

> 사용자 : "베이징에서 상하이로 가는 항공권을 예약하고 싶어요"
>
> 라우팅 에이전트:
> 의도 분석 → '항공편 예약'으로 식별
> → 매표소 직원에게 카드 건네주기
>
> 티켓 판매원:
> 항공편 검색 → 로그인이 필요한 것으로 확인됨
> → 인증원에게 핸드카드를 건네주세요
>
> 인증 에이전트:
> 로그인 안내 → 로그인 성공
> → 핸드카드 반납 티켓 판매원
>
> 티켓 판매원:
> 예약 진행 → 완료

**인계**: 에이전트 A가 해당 부분을 완료한 후 계속 진행하기 위해 컨텍스트를 에이전트 B에 전달합니다. 글로벌 관리자는 없으며 각 에이전트는 자신의 책임만 담당합니다.

**장점**: 유연성, 확장성, 단일 지점 병목 현상 없음
**단점**: 프로세스를 추적하기 어렵습니다. 핸드오프 논리가 제대로 설계되지 않으면 작업은 에이전트 간에 "핑퐁"됩니다.

**일반적인 구현**: OpenAI Swarm 프레임워크는 이 패턴을 기반으로 설계되었습니다.

### 09 칠판 — 공유 칠판

여러 에이전트가 공유된 칠판을 중심으로 작업하며 각자 칠판에 정보를 쓰고 다른 사람의 정보를 읽습니다.

```text
[共享黑板]
┌─────────────────────────────────┐
│ 用户需求: "做一个天气预报 App"    │
│                                 │
│ [搜索 Agent 写入]               │
│ 天气API: OpenWeatherMap 免费    │
│ 地理API: Nominatim              │
│                                 │
│ [设计 Agent 写入]               │
│ 技术栈: React + Node.js        │
│ 架构: 三层（UI/API/数据）        │
│                                 │
│ [代码 Agent 读取黑板后开始编码]  │
│ ...                            │
└─────────────────────────────────┘
```

**감독자와의 차이점**: 감독자는 "명령 중심"입니다. 감독자는 누가 무엇을 하는지 알려줍니다. Blackboard는 "데이터 기반"입니다. 즉, 정보를 보는 사람은 누구나 사전에 처리할 수 있습니다.

### 10 파이프라인/DAG — 파이프라인

작업은 고정된 단계로 분류되며 각 단계는 에이전트에 의해 처리되며 데이터는 이를 통해 순서대로 흐릅니다.

> 입력 → [에이전트 수집] → [에이전트 분석] → [쓰기 에이전트] → [감사 에이전트] → 출력
> 원시 데이터 구조적 분석 보고서 생성 품질 확인

- 각 단계는 병렬로 처리될 수 있습니다(단계 간 종속성이 없는 경우).
- 명확한 인터페이스를 통해 단계 간 데이터 전달
- 모니터링 및 디버그 용이(각 단계의 입력 및 출력이 결정됨)

**제한 사항**: 유연성 부족 - 프로세스가 고정되어 있으며 중간 결과에 따라 동적으로 조정할 수 없습니다.

## 세 번째 수준: 기억 및 지식 모델

에이전트에는 "실제 메모리"가 없으며 모든 대화가 새로 시작됩니다. 메모리 모드는 세션 전반에 걸친 정보 보존 문제를 해결합니다.

### 11 단기 기억 - 현재 대화 맥락

**Essence**: LLM의 컨텍스트 창입니다.

> 현재 대화:
> 사용자: "NVIDIA 분석을 도와주세요."
> 상담원 : [데이터를 얻기 위해 검색툴을 호출 중]
> 사용자: "AMD는 어떻습니까?" ← 상담원은 단기 기억을 통해 "분석"이 재무 분석을 의미함을 알고 있습니다.

**엔지니어링 과제**:

- **제한된 컨텍스트 창**: 긴 대화는 이전 콘텐츠를 "잊어버립니다"(GPT-4 128K, Gemini 1M, GLM 128K)
- **노이즈 축적**: 대화가 길어질수록 관련 없는 정보가 많아져 추론의 질에 영향을 미칩니다.
- **비용**: 토큰 수 = 돈, 긴 컨텍스트 = 높은 비용

**엔지니어링 실습**: 대화 압축(정기 요약), 슬라이딩 윈도우(가장 최근 N 라운드만 유지), 검색 향상(요청 시 관련 조각 검색)

### 12 장기 기억 – 세션 전반에 걸쳐 지식 보유

에이전트가 세션 사이에 정보를 보관하도록 하여 귀하가 누구인지, 무엇을 했는지, 선호 사항을 기억하도록 하십시오.

| 방법 | 원리 | 장점 | 단점 |
| --- | --- | --- | --- |
| **파일 메모리** | MEMORY.md / .learnings/ 읽기 및 쓰기 | 단순하고 투명하며 감사 가능 | 대략적인 수동 유지 관리 |
| **벡터 메모리** | 삽입 후 정보는 벡터 데이터베이스에 저장됩니다. 의미 검색, 자동 연관 | 추가 인프라가 필요합니다 |
| **구조적 메모리** | 지식 그래프, 관계형 데이터베이스 | 정확한 질의, 강력한 추론능력 | 높은 건설 비용 |

💡 **일반적인 문서 기억 연습**: 장기 기억 문서를 사용하여 안정적인 사실을 저장하고, 원본 컨텍스트를 날짜별로 정리된 기록으로 저장한 다음, 학습 로그를 사용하여 오류와 개선 사항을 축적합니다.

### 13 RAG — 검색 향상 생성

LLM이 모든 지식을 "기억"하도록 하는 대신 필요할 때 외부 지식 기반에서 관련 정보를 검색하여 프롬프트 단어에 삽입합니다.

> 사용자 : "2026-04-02에 우리가 무슨 얘기를 나눴나요?"
>
> → 벡터 데이터베이스 검색: query="2026-04-02 Discussion"
> → 검색됨: 해당 메모리 조각/2026-04-02.md
> → 프롬프트에 삽입: "다음 컨텍스트를 기반으로 사용자 질문에 답변: [검색 결과]"
> → LLM이 답변을 생성합니다

💡 **RAG는 메모리 모드의 "검색 엔진"입니다** - "저장 방법"이 아니라 "필요한 정보를 빠르게 찾는 방법"을 해결합니다.

## 네 번째 계층: 생산 수준 엔지니어링 모드

학문적 프로토타입과 사용 가능한 생산 시스템 사이에는 엔지니어링 실습이 있습니다.

### 14 가드레일 - 안전 가드레일

에이전트의 동작에 **경계 제약**을 설정하여 에이전트가 해서는 안 되는 일을 하지 않도록 하세요.

```text
[输入护栏]                          [输出护栏]
用户输入 → ┌──────────┐ → LLM → ┌──────────┐ → 最终输出
          │ 过滤敏感词  │         │ 验证事实   │
          │ 检测注入    │         │ 检查格式   │
          │ 限制话题    │         │ 过滤有害内容│
          └──────────┘         └──────────┘
```

| 레이어 | 예 |
| --- | --- |
| **입력 확인** | 신속한 주입 감지, 민감한 지침 필터링 |
| **도구 권한** | exec 명령 화이트리스트, 파일 쓰기 작업에 확인이 필요함 |
| **출력 감사** | 사실 확인, 형식 검증, 민감한 정보 필터링 |
| **행동 제약** | 에이전트가 특정 데이터 소스에만 액세스하고 외부 메시지를 보낼 수 없도록 제한 |

### 15 인간 참여형(Human-in-the-Loop) — 루프에 참여하는 사람들

에이전트는 완전히 자율적이지 않으며 **주요 결정에는 사람의 확인이 필요합니다**.

> 에이전트: "총 342개의 파일이 있는 /tmp/old-data/ 디렉터리를 삭제하려고 합니다. 삭제를 확인하시겠습니까?"
> 사람: [확인] / [거부] / [수정: .log 파일만 삭제]

| 결정 유형 | 자동화 정도 | 이유 |
| --- | --- | --- |
| 파일 읽기 및 검색 | 완전 자동 | 부작용 없음 |
| 지정된 위치에 파일 쓰기 | 반자동 | 기존 콘텐츠를 덮어쓸 수 있음 |
| 다른 사람에게 메시지 보내기 | 확인이 필요합니다 | 외부 영향이 있음 |
| 데이터 삭제 | 확인 필요 | 되돌릴 수 없는 작동 |
| 쉘 명령 실행 | 위험별로 분류 | 시스템 보안에 영향을 미칠 수 있음 |

**엔지니어링 구현**: 승인 게이트는 Human-in-the-Loop의 일반적인 구현입니다.

### 16 대체 및 재시도 — 내결함성 및 다운그레이드

에이전트의 모든 단계는 실패할 수 있으며 시스템은 실패를 적절하게 처리할 수 있어야 합니다.

```text
搜索请求 → web_search(Gemini)
              ↓ 429 限流
           unified-search(Tavily)
              ↓ 无结果
           unified-search(Exa)
              ↓ 全部失败
           返回提示: "搜索服务暂时不可用，请稍后重试"
```

💡 **강력한 검색 레이어**는 기본 검색 소스에 결과가 없을 때 자동으로 백업 소스로 전환하고 사용자에게 의미 있는 다운그레이드 결과를 제공할 수 있습니다.

**내결함성 설계 원칙**:

1. **빠른 실패**: 실패할 작업이 완료될 때까지 너무 오래 기다리지 마세요.
2. **의미 있는 다운그레이드**: 대체 솔루션은 "아무 것도 할 수 없음"이 될 수 없지만 일부 기능을 제공해야 합니다.
3. **재시도에는 상한이 있습니다**: 무한 재시도 = 무한 루프, 최대 재시도 횟수를 설정해야 합니다.
4. **실패 이유 기록**: 후속 분석 및 개선을 위해 구조화된 오류 대기열에 실패를 기록합니다.

### 17 자기 개선 — 자기 진화

상담원은 자신의 실수로부터 배우고 지속적으로 개선할 수 있습니다.

```text
执行任务 → 出错 → 记录错误 → 分析模式 → 提炼规则 → 下次避免
                                                    ↑
                                              注入到行为中
```

**일반적인 구현**:

- 도구 호출 후 자동으로 오류를 감지하고 구조화된 오류 대기열에 기록합니다.
- 프롬프트 단어를 만들기 전에 보류 중인 학습 알림을 삽입합니다.
- 정기적인 검토를 통해 오류를 보관하고, 반복 패턴을 감지하고, 규칙을 안정적인 구성으로 승격합니다.

💡 **주요 통찰력**: 자기 진화의 핵심은 본질적으로 "학습"이 아닙니다. LLM은 본질적으로 상황을 통해 학습합니다. 핵심은 수동 알림에 의존하는 대신 지속적이고 체계적이며 자동화된 학습을 만드는 것입니다.

```text
[生产级工程层]
├ Guardrails
├ Human-in-Loop
├ Fallback
└ Self-Improvement

[多 Agent 编排层]
├ Supervisor

├ 长期记忆

```

완전한 에이전트 시스템은 일반적으로 **ReAct**(생각하는 동안 도구 호출), **도구 사용**, **감독자**(전문 에이전트 조정), **파일 메모리**, **가드레일**, **Human-in-the-Loop**, **폴백** 및 **자체 개선**을 사용합니다. 이러한 패턴은 함께 작동하고 관리 가능한 시스템을 형성합니다.

## 선택 가이드

| 업무특성 | 권장 모드 | 이유 |
| --- | --- | --- |
| 간단한 Q&A | **ReAct + 도구 사용** | 한두 단계로 해결 가능, 복잡한 계획 필요 없음 |
| 다단계 연구 | **계획 및 실행 + 대응** | 글로벌 비전이 필요하지만 각 단계의 유연성 |
| 코드 생성 | **도구 사용 + 반성** | 실제로 코드를 실행하고 수정해야 함 |
| 다분야 협업 | **감독자/계층** | 전문적인 분업이 필요함 |
| 높은 동시 처리 | **스웜/파이프라인** | 중앙 조정이 필요하지 않으며 병렬화 가능 |
| 지식집약적 | **RAG + 장기 기억** | 많은 양의 지식을 검색해야 함 |
| 보안에 민감 | **가드레일 + Human-in-Loop** | 제약 조건 및 확인 필요 |
| 장기운영 | **자기 개선 + 대체** | 실수로부터 배우고 내결함성이 필요함 |

## 요약: 작업 모드를 먼저 선택한 다음 코드 구조에 대해 이야기하세요.

이 시점에서 에이전트가 어떻게 작동해야 하는지 결정할 수 있어야 합니다. 즉 에이전트를 수행하는 동안 고려해야 하는지, 아니면 계획하고 실행해야 하는지를 결정할 수 있어야 합니다. 감독자의 조정이 필요한지, 아니면 전문 에이전트가 자율적으로 전달해야 하는지 여부; 무엇을 기억해야 하는지, 확인을 위해 어떤 주요 작업을 중지해야 하는지.

이러한 선택은 시스템의 행동 경계를 결정합니다. 다음 단계는 이러한 기능을 교체 가능, 관찰 가능, 복구 가능 코드 구조로 구성하는 것입니다.
