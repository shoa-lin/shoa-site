**루프 엔지니어링(loop engineering)은 에이전트에 직접 프롬프트를 보내던 자신의 역할을, 그 일을 대신 수행하는 시스템 설계로 바꾸는 것입니다.** 여기서 루프는 재귀적인 목표로 이해할 수 있습니다. 목적을 정의하면 AI가 완료될 때까지 반복합니다. 저는 이것이 코딩 에이전트와 협업하는 미래의 방식일 수 있다고 봅니다. 하지만 아직 초기 단계이고 여전히 회의적인 부분이 있으며, 토큰 예산에 따라 사용 패턴이 크게 달라지므로 토큰 비용을 반드시 [신중하게 살펴야](https://x.com/weswinder/status/2063700289710964906) 합니다. 그래서 루프 엔지니어링이 무엇이고 어떤 의미인지 풀어 보려 합니다.

---

Peter Steinberger는 최근 [이렇게 말했습니다](https://x.com/steipete/status/2063697162748260627). "이제 코딩 에이전트에 직접 프롬프트를 보내지 말아야 합니다. 대신 에이전트에 프롬프트를 보내는 루프를 설계해야 합니다." 마찬가지로 canonical 원문은 Anthropic의 Claude Code 책임자 Boris Cherny의 발언을 [다음과 같이 전합니다](https://addyosmani.com/blog/loop-engineering/). "저는 더 이상 Claude에 직접 프롬프트를 보내지 않습니다. Claude에 프롬프트를 보내고 해야 할 일을 파악하는 루프를 실행합니다. 제 일은 루프를 작성하는 것입니다."

그렇다면 이 말은 실제로 무엇을 뜻할까요?

지난 약 2년 동안 코딩 에이전트에서 결과를 얻는 방법은 좋은 프롬프트를 작성하고 충분한 맥락을 제공하는 것이었습니다. 무언가를 입력하고, 돌아온 결과를 읽고, 다음 내용을 다시 입력했습니다. 에이전트는 사용자가 처음부터 끝까지 손에 쥐고 한 턴씩 사용하는 도구였습니다. 이제 그 방식은 사실상 끝났거나, 적어도 일부 사람들은 그렇게 생각합니다.

이제는 일을 발견하고, 배정하고, 확인하고, 완료 내역을 기록하고, 다음 단계를 결정하는 작은 시스템을 구축합니다. 사용자가 직접 에이전트를 움직이는 대신 그 시스템이 에이전트를 구동하게 합니다. 저는 이와 가까운 두 개념을 이전에 다뤘습니다. 하나는 단일 에이전트가 실행되는 환경을 설계하는 [에이전트 하네스 엔지니어링](https://addyosmani.com/blog/agent-harness-engineering/)이고, 다른 하나는 소프트웨어를 만드는 시스템인 [팩토리 모델](https://addyosmani.com/blog/factory-model/)입니다. 루프 엔지니어링은 하네스보다 한 단계 위에 있습니다. 타이머에 따라 실행되고, 작은 보조 에이전트를 만들며, 결과를 다음 사이클에 다시 전달하는 하네스라고 볼 수 있습니다.

놀라운 점은 이것이 더 이상 도구 수준의 문제만은 아니라는 사실입니다. 1년 전 루프가 필요했다면 bash 스크립트를 잔뜩 작성하고 계속 유지해야 했으며, 그 시스템은 오직 자신만을 위한 일회성 도구였습니다. 이제는 구성 요소가 제품에 직접 들어 있습니다. Steinberger의 목록은 거의 그대로 Codex 앱에 대응하고, Claude Code에도 거의 동일하게 대응합니다. 구조가 같다는 사실을 알아차리면 어느 도구가 더 나은지 논쟁하는 대신 어떤 도구 안에서도 작동할 루프를 설계하게 됩니다.

## 다섯 가지 구성 요소와 상태에 관한 설명

하나의 [루프](https://x.com/reach_vb/status/2063713960495558940)에는 다섯 가지 요소와 상태를 기억할 장소가 필요합니다. 세부 대응 관계를 보기 전에 먼저 목록부터 살펴보겠습니다.

1. **자동화(Automations)**: 일정에 따라 자동으로 실행되며 발견과 분류를 스스로 처리합니다.
2. **워크트리(Worktrees)**: 병렬로 작업하는 두 에이전트가 서로의 파일을 건드리지 않게 합니다.
3. **스킬(Skills)**: 에이전트가 추측하지 않도록 프로젝트 지식을 기록합니다.
4. **플러그인과 커넥터(Plugins and connectors)**: 에이전트를 이미 사용 중인 도구에 연결합니다.
5. **서브에이전트(Sub-agents)**: 하나는 해법을 제안하고 다른 하나는 이를 확인하게 합니다.

여기에 여섯 번째 요소인 메모리가 있습니다. Markdown 파일이나 Linear 보드처럼 단일 대화보다 오래 유지되면서 완료된 일과 다음 할 일을 기록하는 것이면 무엇이든 됩니다. 너무 단순해서 중요하지 않아 보일 수도 있습니다. 하지만 모든 장시간 실행 에이전트가 의존하는 방식이기도 합니다. [long-running agents](https://addyosmani.com/blog/long-running-agents/)에서 자세히 다뤘듯 모델은 실행 사이의 내용을 모두 잊기 때문에 메모리는 맥락이 아니라 디스크에 있어야 합니다. 에이전트는 잊지만 저장소는 잊지 않습니다.

두 제품 모두 이제 이 다섯 구성 요소를 제공합니다.

| 프리미티브 | 루프에서 맡는 역할 | Codex 앱 | Claude Code |
| --- | --- | --- | --- |
| **자동화** | 예약된 발견 + 분류 | [Automations tab](https://developers.openai.com/codex/app/automations): 프로젝트, 프롬프트, 실행 주기, 환경을 선택하고 결과는 Triage 받은편지함으로 전달되며 `/goal`로 완료 시까지 실행 | Scheduled tasks와 cron, `/loop`, `/goal`, hooks, GitHub Actions |
| **워크트리** | 병렬 기능 격리 | 스레드마다 내장 워크트리 제공 | `git worktree`, `--worktree`, 서브에이전트의 `isolation: worktree` |
| **스킬** | 프로젝트 지식 체계화 | [Agent Skills](https://developers.openai.com/codex/skills)(`SKILL.md`), `$name`으로 호출하거나 암시적으로 시작 | [Agent Skills](https://addyosmani.com/blog/agent-skills/)(`SKILL.md`) |
| **플러그인 / 커넥터** | 실제 도구 연결 | Connectors(MCP) + 배포용 plugins | MCP servers + plugins |
| **서브에이전트** | 해법 제안과 검증 | [Subagents](https://developers.openai.com/codex/subagents), `.codex/agents/`에 TOML로 정의 | `.claude/agents/`의 subagent와 agent teams |
| **상태** | 완료된 작업 추적 | 커넥터를 통한 Markdown 또는 Linear | Markdown(`AGENTS.md`, 진행 파일) 또는 MCP를 통한 Linear |

이름은 조금씩 다르지만 제공하는 기능은 같습니다. 세부 사항이 중요한 이유는 루프가 제대로 맞물려 돌아갈지, 아니면 곳곳에서 조용히 상태를 흘릴지를 결정하기 때문입니다.

## 자동화: 루프의 심장 박동

자동화는 한 번 실행하고 끝나는 작업을 실제 루프로 바꾸는 요소입니다. Codex 앱에서는 Automations 탭에서 자동화를 만들고 프로젝트, 실행할 프롬프트, 실행 주기, 로컬 체크아웃과 백그라운드 워크트리 중 실행 환경을 선택합니다. 무언가를 발견한 실행 결과는 Triage 받은편지함으로 들어오고, 아무것도 발견하지 못한 실행은 자동으로 보관됩니다. OpenAI 내부에서는 일일 이슈 분류, CI 실패 요약, 커밋 브리핑 작성, 지난주에 누군가 도입한 버그 추적처럼 단조로운 작업에 자동화를 사용합니다. 자동화는 스킬도 호출할 수 있습니다. 재사용할 동작을 유지보수 가능한 형태로 두고, 아무도 갱신하지 않을 긴 지침을 일정에 붙여 넣는 대신 `$skill-name`을 시작하면 됩니다.

Claude Code는 스케줄링과 훅을 통해 같은 목적지에 도달합니다. `/loop`로 프롬프트나 명령을 일정 간격마다 실행하고, cron 작업을 예약하고, 에이전트 생명주기의 특정 시점에 훅으로 셸 명령을 실행할 수 있습니다. 노트북을 닫은 뒤에도 계속 실행하려면 전체 작업을 GitHub Actions로 옮길 수 있습니다. 발상은 완전히 같습니다. 자율 작업과 주기를 정의하면 사용자가 모든 시스템을 직접 확인하지 않아도 발견된 문제가 사용자에게 찾아옵니다.

이 글의 핵심에 더 가까운 세션 내 프리미티브도 알아둘 만합니다. `/loop`는 정해진 주기로 반복 실행합니다. `/goal`은 사용자가 작성한 조건이 실제로 참이 될 때까지 계속 실행합니다. 매 턴이 끝나면 별도의 작은 모델이 완료 여부를 확인하므로 코드를 작성한 에이전트가 자신의 결과를 직접 평가하지 않습니다. "test/auth의 모든 테스트가 통과하고 lint가 깨끗할 것" 같은 조건을 주고 자리를 비우면 됩니다. Codex에도 이름이 같은 `/goal`이 있습니다. 검증 가능한 종료 조건이 성립할 때까지 여러 턴에 걸쳐 작업을 이어 가며 일시 중지, 재개, 지우기를 지원합니다. 두 도구에 같은 프리미티브가 있다는 점은 이 글 전체에서 반복되는 패턴입니다.

여기까지가 일을 발견하는 부분입니다. 루프의 나머지 부분은 발견한 일에 조치합니다.

## 워크트리: 병렬 작업이 혼란으로 번지는 것을 막기

에이전트를 둘 이상 실행하는 순간 파일 충돌이 시작되며, 이것이 주요 실패 지점입니다. 두 에이전트가 같은 파일을 수정하는 상황은 두 엔지니어가 서로 조율하지 않은 채 같은 줄을 변경하는 것과 똑같은 골칫거리입니다. git worktree는 이 기계적인 문제를 해결합니다. 동일한 저장소 이력을 공유하지만 별도 브랜치에서 작동하는 독립 작업 디렉터리이므로 한 에이전트의 수정이 다른 에이전트의 체크아웃에 닿을 수 없습니다.

Codex는 워크트리 지원을 앱에 직접 내장해 여러 스레드가 같은 저장소에서 동시에 작업하더라도 서로의 체크아웃을 수정하지 않게 합니다. Claude Code는 `git worktree`, 자체 체크아웃에서 세션을 여는 `--worktree` 플래그, 각 보조 에이전트에 새 체크아웃을 제공하고 작업 후 정리하는 서브에이전트의 `isolation: worktree` 설정으로 같은 격리를 제공합니다. 저는 [orchestration tax](https://addyosmani.com/blog/orchestration-tax/)에서 사람 측면의 비용을 다뤘습니다. 워크트리는 기계적인 충돌을 없애지만 한계는 여전히 **사용자**입니다. 실제로 몇 개의 에이전트를 실행할 수 있는지는 도구가 아니라 사용자의 리뷰 처리량이 결정합니다.

## 스킬: 매번 프로젝트를 다시 설명하지 않게 하기

스킬을 사용하면 매 세션마다 같은 프로젝트 맥락을 다시 설명하지 않아도 됩니다. 두 도구는 동일한 형식을 사용합니다. 지침과 메타데이터를 담은 `SKILL.md`가 있는 폴더에 선택적으로 스크립트, 참고 자료, 리소스를 추가합니다. Codex는 `$`나 `/skills`로 호출할 때 스킬을 실행하거나, 작업이 스킬 설명과 일치하면 자동으로 시작합니다. 그래서 영리해 보이는 설명보다 짧고 직설적인 설명이 더 효과적입니다. Claude Code도 같은 방식으로 작동하며, 저는 [agent skills](https://addyosmani.com/blog/agent-skills/)에서 이 패턴을 설명했습니다.

스킬은 의도를 반복해서 설명하는 비용을 없애는 곳이기도 합니다. 저는 [intent debt](https://addyosmani.com/blog/intent-debt/)에서 에이전트가 매 세션을 처음부터 시작하고, 사용자의 의도에 빈틈이 있으면 확신에 찬 추측으로 채운다고 설명했습니다. 스킬은 의도를 외부에 기록한 것입니다. 규칙, 빌드 단계, "그 사고 때문에 우리는 이 방식을 사용하지 않는다" 같은 내용을 한 번 적어 두면 에이전트가 실행될 때마다 읽습니다. 스킬이 없으면 루프는 매 사이클마다 프로젝트 전체를 처음부터 다시 추론합니다. 스킬이 있어야 프로젝트 지식이 사이클을 거치며 축적될 수 있습니다.

한 가지 구분은 분명히 해야 합니다. 스킬은 작성 형식이고 플러그인은 배포 방식입니다. 여러 저장소에 스킬을 공유하거나 몇 개를 묶고 싶다면 플러그인으로 패키징합니다. Codex와 Claude Code 모두 같습니다.

## 플러그인과 커넥터: 루프를 실제 도구에 연결하기

파일 시스템만 볼 수 있는 루프는 활동 범위가 매우 좁습니다. MCP 기반 커넥터를 사용하면 에이전트가 이슈 트래커를 읽고, 데이터베이스를 조회하고, 스테이징 API를 호출하고, Slack에 메시지를 보낼 수 있습니다. Codex와 Claude Code 모두 MCP를 지원하므로 한쪽을 위해 만든 커넥터를 다른 쪽에서도 대체로 사용할 수 있습니다. 플러그인은 커넥터와 스킬을 함께 패키징하므로 팀원이 전체 구성을 기억에 의존해 다시 만드는 대신 한 번에 설치할 수 있습니다.

이 차이가 "수정 방법은 이렇습니다"라고 말하는 에이전트와, PR을 자동으로 열고 Linear 티켓을 연결하고 CI가 통과하면 채널에 알리는 루프를 가릅니다. 커넥터가 있어야 루프가 실제 환경에서 행동할 수 있으며, 할 수만 있다면 무엇을 하겠다고 설명하는 데 그치지 않습니다.

## 서브에이전트: 작성자와 검증자 분리하기

루프에서 지금까지 가장 유용한 구조적 기법은 작성자와 검증자를 분리하는 것입니다. 코드를 작성한 모델은 자신의 결과를 평가할 때 지나치게 관대합니다. 다른 지침을 받고 때로는 다른 모델을 사용하는 두 번째 에이전트는 첫 번째 에이전트가 스스로 납득해 버린 문제를 찾아낼 수 있습니다.

Codex는 사용자가 요청할 때만 서브에이전트를 생성하고, 이들을 동시에 실행한 뒤 결과를 하나의 답변으로 통합합니다. `.codex/agents/`에 이름, 설명, 지침, 선택적인 모델과 reasoning effort를 담은 TOML 파일로 사용자 정의 에이전트를 정의할 수 있습니다. 보안 리뷰어는 높은 effort의 강력한 모델로, 탐색기는 빠른 읽기 전용 모델로 구성할 수 있습니다. Claude Code는 `.claude/agents/`의 서브에이전트와 서로 작업을 전달하는 agent teams로 같은 일을 합니다. 두 도구에서 흔한 분리는 한 에이전트가 탐색하고, 다른 에이전트가 구현하며, 또 다른 에이전트가 명세와 대조해 검증하는 방식입니다.

저는 이 주장을 두 차례 다뤘습니다. 한 번은 [code agent orchestra](https://addyosmani.com/blog/code-agent-orchestra/)에서, 또 한 번은 [agentic code review](https://addyosmani.com/blog/agentic-code-review/)에서 설명했습니다. 루프 안에서 특히 중요한 이유는 사용자가 보고 있지 않을 때도 루프가 실행되기 때문입니다. 실제로 신뢰할 수 있는 검증자가 있어야 잠시 자리를 비울 수 있습니다. 각 서브에이전트가 자체 모델 작업과 도구 호출을 수행하므로 더 많은 토큰을 사용합니다. 두 번째 의견에 비용을 지불할 가치가 있는 곳에 토큰을 써야 합니다. Claude Code의 `/goal`도 내부적으로 같은 구조를 사용합니다. 작업을 수행한 모델이 아니라 새로운 모델이 루프 완료 여부를 판단합니다. 즉, 작성자와 검증자의 분리를 종료 조건 자체에도 적용합니다.

## 루프는 어떤 모습인가

구성 요소를 하나로 묶으면 단일 스레드가 작은 제어판으로 바뀝니다. 다음은 제가 계속 사용해 온 패턴입니다.

자동화 하나가 매일 아침 저장소에서 실행됩니다. 프롬프트는 분류 스킬을 호출해 전날의 CI 실패, 열린 이슈, 최근 커밋을 읽고 발견 사항을 Markdown 파일이나 Linear 보드에 기록합니다. 처리할 가치가 있는 발견 사항마다 스레드는 격리된 워크트리를 열고, 서브에이전트 하나에 수정안 작성을 맡기며, 두 번째 서브에이전트가 프로젝트 스킬과 기존 테스트를 기준으로 초안을 리뷰합니다.

커넥터는 루프가 PR을 열고 티켓을 갱신하게 합니다. 루프가 처리할 수 없는 항목은 제 Triage 받은편지함으로 들어옵니다. 상태 파일은 전체 시스템의 중심축입니다. 무엇을 시도했고, 무엇이 통과했으며, 무엇이 아직 열려 있는지 기억하므로 다음 날 아침 실행은 오늘 멈춘 지점에서 이어집니다.

제가 실제로 한 일을 보십시오. 저는 프로세스를 한 번 설계했을 뿐, 개별 단계에는 직접 프롬프트를 보내지 않았습니다. 이것이 Steinberger의 주장을 구체화한 모습이며, 구성 요소가 같기 때문에 Codex와 Claude Code에서도 같은 루프입니다.

## 루프가 여전히 대신할 수 없는 일

루프는 일을 바꾸지만 사용자를 일에서 제거하지는 않습니다. 루프가 강력해질수록 다음 세 가지 문제는 쉬워지는 것이 아니라 오히려 더 날카로워집니다.

**검증은 여전히 사용자의 책임입니다.** 무인으로 실행되는 루프는 아무도 지켜보지 않을 때도 실수할 수 있습니다. 검증 서브에이전트와 작성자를 분리하는 이유는 루프가 말하는 "완료"에 더 나은 근거를 부여하기 위해서입니다. 그래도 "완료"는 여전히 주장이지 증명은 아닙니다. 저는 [code review in the age of AI](https://addyosmani.com/blog/code-review-ai/)에서 같은 말을 반복해 왔습니다. 실제로 작동한다고 확인한 코드를 출시하는 것이 사용자의 일입니다.

**방치하면 이해도는 계속 떨어집니다.** 루프가 사용자가 작성하지 않은 코드를 더 빠르게 전달할수록 실제 시스템과 사용자가 이해하는 시스템 사이의 격차는 커집니다. 이것이 [이해 부채(comprehension debt)](https://addyosmani.com/blog/comprehension-debt/)입니다. 루프가 만든 내용을 주의 깊게 읽지 않으면 매끄러운 루프는 이 부채를 더 빠르게 늘릴 뿐입니다.

**지나치게 편안한 태도는 위험합니다.** 루프가 스스로 실행되면 사람이 판단을 형성하지 않은 채 돌아온 결과를 그대로 받아들이기 쉽습니다. 저는 이를 [인지적 포기(cognitive surrender)](https://addyosmani.com/blog/cognitive-surrender/)라고 부릅니다. 판단력을 갖고 루프를 설계하는 것은 해독제가 될 수 있습니다. 생각을 피하려고 루프를 설계하면 문제를 가속합니다. 같은 행동이 정반대의 결과를 만듭니다.

## 루프를 구축하되 엔지니어로 남기

저는 이것이 우리의 일이 어떻게 변할지 보여 주는 예고편이라고 생각합니다. 하지만 제가 직접 코드를 리뷰하지 않거나 수정 작업을 자동화 루프에 전적으로 의존한다면 제품 품질은 나빠질 것입니다. 결국 계속해서 더 깊은 구덩이를 파는 하강 나선에 빠질 수도 있습니다.

그렇더라도 루프를 계속 구축하십시오. 다만 에이전트에 직접 프롬프트를 보내는 방식도 여전히 효과적이라는 점을 잊지 마십시오. 핵심은 적절한 균형을 찾는 것입니다.

루프는 사용하는 사람에 따라 전혀 다른 결과를 만들 수 있습니다. 두 사람이 완전히 같은 루프를 구축해도 정반대의 결과를 얻을 수 있습니다. 한 사람은 자신이 깊이 이해하는 일을 더 빠르게 진행하는 데 루프를 사용합니다. 다른 사람은 일 자체를 이해하지 않기 위해 루프를 사용합니다. 루프는 그 차이를 모르지만, 당신은 압니다.

그래서 루프 설계는 프롬프트 엔지니어링보다 쉬운 것이 아니라 더 어렵습니다. Cherny의 요지는 일이 단순해졌다는 뜻이 아닙니다. 지렛대의 받침점이 옮겨졌다는 뜻입니다.

루프를 구축하십시오. 단순히 시작 버튼을 누르는 사람이 아니라, 엔지니어로 남으려는 사람처럼 구축하십시오.
