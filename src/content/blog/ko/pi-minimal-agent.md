*2026년 1월 31일 작성*

OpenClaw는 ClawdBot, MoltBot 등 여러 이름으로 빠르게 화제가 됐습니다. 본질적으로는 통신 채널에 연결돼 코드를 실행할 수 있는 에이전트입니다.

OpenClaw 내부에는 **Pi**라는 작은 코딩 에이전트가 사용됩니다. Armin Ronacher는 이제 거의 Pi만을 코딩 에이전트로 사용한다고 말하며, 의도적으로 작게 설계된 구조가 왜 매력적인지 설명합니다.

Pi는 **Mario Zechner**가 만들었습니다. Mario의 현실적인 접근은 Peter의 "약간의 광기가 섞인 SF"와 다르지만, Pi와 OpenClaw는 같은 전제를 공유합니다. LLM은 코드를 작성하고 실행하는 데 매우 능숙하므로 시스템은 그 능력을 적극적으로 활용해야 한다는 것입니다.

## Pi란 무엇인가

코딩 에이전트는 많지만 Pi를 돋보이게 하는 특징은 두 가지입니다. Armin은 **AMP** 역시 그저 세련된 인터페이스로 감싼 제품이 아니라, 에이전트 기반 프로그래밍을 진지하게 시험해 온 사람들이 만든 제품이라고 평가합니다.

Pi가 흥미로운 이유는 크게 두 가지입니다.

- **코어가 매우 작습니다.** 시스템 프롬프트가 이례적으로 짧고, 코어가 제공하는 도구는 Read, Write, Edit, Bash 네 개뿐입니다.
- **강력한 확장 시스템이 있습니다.** 확장은 동작을 추가하고 자체 상태를 세션에 영속화할 수 있습니다.

실용적인 장점도 있습니다. Pi는 세심하게 만든 소프트웨어처럼 안정적이고 메모리를 적게 사용하며, 화면 깜빡임이나 무작위 오류로 사용자의 집중을 흐트러뜨리지 않습니다.

Pi는 다른 에이전트를 만들기 위한 작은 컴포넌트 모음이기도 합니다. OpenClaw는 이 컴포넌트 위에 구축됐고, Armin은 Telegram 봇에, Mario는 `mom`에 사용했습니다. Pi에게 자체 코드와 `mom` 같은 예시를 보여 주면 원하는 통합 방식에 맞는 다른 에이전트를 조립하도록 도울 수 있습니다.

## Pi에 없는 것

Pi를 이해하려면 의도적으로 제외한 것도 알아야 합니다. 코어에는 **내장 MCP 지원이 없습니다**. 그렇다고 MCP를 사용할 수 없는 것은 아닙니다. 확장으로 추가하거나, MCP 호출을 CLI 또는 TypeScript 바인딩으로 노출하는 **mcporter**를 에이전트가 사용할 수 있습니다.

이 생략은 Pi의 철학을 반영합니다. 에이전트에 어떤 기능이 없을 때 기본 행동은 마켓플레이스에서 미리 만들어진 확장을 찾는 것이 아닙니다. 코드를 작성하고 실행해 스스로를 확장하도록 에이전트에 요청하는 것입니다.

확장 다운로드도 지원합니다. 차이는 문화에 가깝습니다. 기존 확장을 바꿀 수 없는 의존성으로 취급하는 대신, 에이전트가 로컬 요구에 맞게 재구성할 참고 자료로 사용할 수 있습니다.

## 에이전트를 만드는 에이전트를 위한 설계

스스로 형태를 바꾸는 소프트웨어에는 몇 가지 기반 능력이 필요합니다.

첫째, Pi의 AI SDK에서는 하나의 세션에 서로 다른 모델 제공자의 메시지를 담을 수 있습니다. 세션이 완벽하게 이식 가능하지 않다는 현실은 인정하면서도 제공자 고유 기능에 불필요하게 의존하지 않습니다.

둘째, 세션 파일에는 모델 메시지와 함께 사용자 정의 메시지를 저장할 수 있습니다. 확장은 이를 이용해 상태를 영속화하고, 시스템은 일부 정보를 모델에 전혀 보내지 않거나 일부만 포함하도록 결정할 수 있습니다.

셋째, 확장 상태를 디스크에 저장할 수 있고 확장은 핫 리로드를 지원합니다. 에이전트가 확장을 작성하고, 다시 불러오고, 테스트한 뒤 계속 반복할 수 있습니다. 넷째, Pi에는 에이전트가 자기 확장 과정에서 읽을 수 있는 문서와 예제가 포함됩니다. 다섯째, 세션은 트리 구조입니다. 사용자는 별도 작업으로 분기해 메인 브랜치의 컨텍스트를 소비하지 않고 고장 난 도구를 고친 다음 돌아올 수 있으며, Pi는 다른 브랜치에서 일어난 일을 요약해 줄 수 있습니다.

이런 선택은 도구에도 중요합니다. 많은 모델 제공자에서는 MCP 도구와 다른 LLM 도구가 세션 시작 시 시스템 컨텍스트나 도구 영역에 로드됩니다. 이후 정의를 완전히 교체하면 캐시를 파괴하거나 이전 호출 방식에 대한 상충된 기억을 모델에 남길 수 있습니다.

## 모델 컨텍스트 밖의 도구

Pi 확장은 LLM이 호출할 수 있는 도구를 등록할 수 있고 Armin도 가끔 이 방식을 사용합니다. 로컬에서 만든 이슈 추적기가 한 예입니다. 에이전트가 직접 할 일을 관리해야 하므로 CLI 대신 추가 도구 하나를 노출합니다. 현재 그가 모델 컨텍스트에 로드하는 유일한 추가 도구입니다.

추가 기능 대부분은 도구 스키마 형태로 모델 컨텍스트를 차지할 필요가 없습니다. 대개 스킬이거나 사람의 작업 흐름을 개선하는 TUI 확장입니다. Pi 확장은 스피너, 진행 표시줄, 파일 선택기, 표, 미리보기 창을 터미널에 직접 그릴 수 있습니다. Mario는 TUI에서 Doom을 실행하는 데모까지 보여 줬습니다. 실용적이지는 않지만 인터페이스의 유연성을 잘 보여 줍니다.

아래 확장들은 고정된 패키지가 아니라 예시입니다. 에이전트에게 하나를 참고하게 하고 필요한 동작에 맞춰 재구성하도록 하는 것이 의도된 사용법입니다.

### `/answer`

Armin은 Plan Mode를 사용하지 않습니다. 엄격한 구조화 질문 대화보다 에이전트의 자연스러운 문장 속에 설명과 그림을 섞어 생산적으로 주고받는 방식을 선호합니다.

본문 안의 질문이 많아지면 깔끔하게 답하기 어려워집니다. `/answer`는 에이전트의 마지막 응답을 읽고 질문을 추출해 답하기 쉬운 하나의 입력 상자로 다시 구성합니다.

![/answer 확장이 질문 대화 상자를 보여 주는 화면](https://lucumr.pocoo.org/static/pi-answer.png)

### `/todos`

Armin은 Beads의 구현을 비판하지만 에이전트용 할 일 목록은 유용하다고 봅니다. `/todos`는 `.pi/todos` 아래에 Markdown 파일로 저장된 항목을 엽니다. 사용자와 에이전트 모두 편집할 수 있고, 세션이 작업을 맡아 진행 중으로 표시할 수도 있습니다.

### `/review`

에이전트가 더 많은 코드를 작성할수록 미완성 작업은 사람에게 넘기기 전에 에이전트 리뷰를 거쳐야 합니다. Pi 세션은 트리 구조이므로 Armin은 새로운 리뷰 컨텍스트로 분기해 지적 사항을 모으고 수정 사항을 메인 세션으로 가져올 수 있습니다.

![/review 확장이 리뷰 프리셋 옵션을 보여 주는 화면](https://lucumr.pocoo.org/static/pi-review.png)

인터페이스는 Codex를 본떠 만들었으며 커밋, 변경 사항, 커밋되지 않은 변경, 원격 풀 리퀘스트 등을 리뷰 대상으로 지원합니다. 리뷰 프롬프트는 새로 추가된 의존성을 명시적으로 지적하는 등 Armin이 중요하게 여기는 피드백을 강조합니다.

### `/control`

Armin의 일상적인 작업 흐름에 포함된 기능이 아니라 실험적인 확장입니다. 하나의 Pi 에이전트가 다른 Pi 에이전트에 프롬프트를 보내 복잡한 오케스트레이션 계층 없이 소규모 멀티 에이전트 구성을 만들 수 있습니다.

### `/files`

이 확장은 세션에서 변경하거나 참조한 파일을 나열합니다. Finder에서 표시하고, VS Code에서 변경 사항을 확인하고, Quick Look으로 열거나, 프롬프트에서 참조할 수 있습니다. `shift+ctrl+r`을 누르면 가장 최근에 언급된 파일을 Quick Look으로 볼 수 있어 에이전트가 PDF를 만들었을 때 편리합니다.

다른 개발자들도 확장을 만들었습니다. Nico의 서브에이전트 확장과 Pi가 동작을 확인할 수 있는 TUI 오버레이 안에서 대화형 CLI를 자율적으로 실행하게 해 주는 `interactive-shell` 등이 있습니다.

## 소프트웨어가 소프트웨어를 만든다

핵심은 Armin이 이 확장들을 직접 작성하지 않았다는 점입니다. 원하는 것을 설명하면 Pi가 만들었습니다. Pi 코어에는 MCP도, 번들로 제공되는 커뮤니티 스킬도 없지만 에이전트는 소유자에게 맞는 기능을 만들고 유지할 수 있습니다. 한 예로 브라우저 자동화 CLI나 MCP 통합 대신 CDP와 직접 통신하는 스킬을 사용합니다.

그의 에이전트에는 많은 스킬이 있지만 모두 필요할 때 버릴 수 있습니다. 다른 엔지니어가 공유한 Pi 세션을 코드 리뷰용으로 읽는 것도 있고, 커밋 메시지, 커밋 동작, 변경 기록 업데이트 방식을 다듬는 것도 있습니다. 이전 슬래시 명령 일부를 스킬로 옮기는 중이며, `uv` 사용을 권하는 스킬과 `pip`, `python` 호출을 `uv`로 리디렉션하는 확장을 함께 사용하기도 합니다.

이것이 Pi 같은 미니멀 에이전트의 매력입니다. 소프트웨어가 소프트웨어를 만드는 일을 일상적인 업무 방식으로 자리 잡게 합니다. OpenClaw는 로컬 UI를 없애고 에이전트를 채팅에 연결해 이 아이디어를 더 밀어붙입니다. Armin의 결론은 모든 세부 사항이 확정됐다는 것이 아닙니다. 다만 이 방향이 점점 소프트웨어 미래의 일부처럼 보인다는 것입니다.
