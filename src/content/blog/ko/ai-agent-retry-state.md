## 결론부터 말하면

**AI가 텍스트만 생성할 때 Retry는 다른 답변을 다시 뽑는 기능입니다. 그러나 AI가 현실의 환경에 행동하기 시작하면 Retry는 작업 전체를 다시 실행하는 일이 될 수 있습니다.**

이 차이는 Codex 같은 Agent 제품에서 전통적인 ‘재생성’ 버튼이 더 이상 중심 기능으로 보이지 않는 배경을 설명합니다.

- Chatbot의 Retry는 보통 화면의 텍스트 하나만 바꿉니다.
- Agent의 한 번의 실행은 이미 파일을 수정하고, 명령을 실행하고, 외부 도구를 호출했을 수 있습니다.
- 두 번째 실행이 시작될 때 세계는 첫 번째 실행 이전과 같은 상태가 아닙니다.
- 따라서 막연한 Retry보다 피드백을 이어 가거나, 실패한 단계만 다시 실행하거나, 명확한 상태에서 Fork하는 편이 안전합니다.

다만 Codex가 “상태가 복잡해져서 Retry를 없앴다”고 공식적으로 밝힌 적은 없습니다. 이 글은 공개된 Thread, Turn, Item, Fork 모델을 바탕으로 한 아키텍처적 해석입니다.

![다른 답변을 만드는 Retry에서 이력을 보존하는 분기로](/assets/blog/ai-agent-retry-state/retry-becomes-fork-ko.png)

예전 Chatbot을 사용할 때 저는 **Retry**, 즉 재시도나 재생성 기능에 익숙했습니다.

질문을 보내면 AI가 답하고, 답변이 마음에 들지 않으면 질문을 다시 쓰지 않아도 Retry 한 번으로 다른 답변을 받을 수 있었습니다.

너무 자연스러운 기능이었기 때문에 Codex를 쓰기 시작하며 오히려 궁금해졌습니다. **왜 Codex에서는 익숙한 ‘재생성’ 버튼이 두드러지지 않을까요?**

단순히 빠진 기능일까요? 아니면 Agent 제품에는 다른 상호작용 방식이 필요한 걸까요?

제가 지금 가장 타당하다고 보는 설명은 이렇습니다.

> AI가 텍스트만 생성할 때 Retry는 다른 답변을 다시 뽑는 기능입니다. 그러나 AI가 현실의 환경에 행동하기 시작하면 Retry는 작업 전체를 다시 실행하는 일이 될 수 있습니다.

겉으로는 버튼 하나의 차이지만, 그 뒤에는 완전히 다른 두 시스템이 있습니다.

## Chatbot 시대의 Retry는 왜 단순했을까

전통적인 Chatbot의 상호작용은 대략 다음과 같습니다.

```text
사용자 질문 → 모델 생성 → 답변 A
                          └→ Retry → 답변 B
```

첫 답변이 만족스럽지 않으면 같은 대화 맥락을 유지한 채 모델에 다시 생성을 요청할 수 있습니다. 이전 답변을 버려도 심각한 결과가 생기는 경우는 드뭅니다.

대부분 Chatbot이 바꾼 것은 화면의 텍스트뿐이기 때문입니다.

- 로컬 파일을 수정하지 않았습니다.
- 명령을 실행하지 않았습니다.
- Git 브랜치를 바꾸지 않았습니다.
- 외부 시스템에 데이터를 쓰지 않았습니다.
- 취소할 수 없는 이메일을 보내지 않았습니다.

같은 문제를 다른 방식으로 다시 설명해 달라고 부탁하는 것과 비슷합니다. 첫 종이가 마음에 들지 않으면 구겨 버리고 새로 쓰면 됩니다.

따라서 Chatbot의 Retry는 **입력은 유지하고 출력은 버린 뒤, 새로운 후보를 생성하는 것**으로 이해할 수 있습니다.

## Codex가 돌려주는 것은 ‘답변’만이 아니다

Codex 같은 Agent는 최종 답변을 쓰기 전에 이미 많은 작업을 수행했을 수 있습니다.

Codex App Server 모델에서 Thread는 여러 Turn으로 구성되고, Turn은 여러 Item을 포함합니다. Item에는 사용자와 AI의 메시지뿐 아니라 명령 실행, 파일 변경, 도구 호출도 들어갑니다.

즉 화면에 보이는 마지막 문장은 전체 작업을 요약한 보고에 불과할 수 있습니다.

```text
사용자 작업
   │
   ▼
저장소와 컨텍스트 이해
   │
   ▼
파일 읽기 → 명령 실행 → 도구 호출 → 코드 수정 → 테스트 실행
   │
   ▼
최종 응답: “다음 내용을 수정했습니다…”
```

사용자가 최종 답변에 만족하지 않을 때 Retry는 정확히 무엇을 반복해야 할까요?

마지막 요약만 다시 쓸까요? 추론부터 다시 할까요? 모든 명령을 다시 실행할까요? 아니면 이미 발생한 변경을 먼저 되돌린 다음 처음부터 실행해야 할까요?

이 순간부터 Retry는 단순한 기능이 아닙니다.

## 두 번째 실행이 시작될 때 세계는 이미 달라져 있다

Codex에 다음과 같이 요청했다고 가정해 보겠습니다.

> 이 버그를 수정하고, 테스트를 실행한 뒤, Pull Request를 만들어 주세요.

첫 실행에서 Codex는 이미 다음 작업을 끝냈을 수 있습니다.

1. 코드를 읽고 문제를 찾았습니다.
2. 파일 세 개를 수정했습니다.
3. 테스트를 실행했습니다.
4. 새 브랜치를 만들었습니다.
5. 변경을 커밋하고 push했습니다.
6. Pull Request를 만들었습니다.

이제 Retry를 누르면 어떤 일이 생길까요?

- 텍스트만 다시 생성하면 새 설명이 실제 수행된 작업과 어긋날 수 있습니다.
- 추론부터 다시 하면 이미 수정된 저장소에서 두 번째 작업을 시작합니다.
- 도구를 다시 실행하면 중복 커밋, 충돌하는 브랜치, 두 번째 PR이 생길 수 있습니다.
- 먼저 rollback하려 해도 일부 외부 작업은 되돌릴 수 없습니다.

파일은 복원할 수 있지만 이메일은 이미 전송됐을 수 있습니다. 로컬 브랜치는 지울 수 있어도 외부 시스템의 승인, 메시지, 거래를 흔적 없이 되돌릴 수 있다고 보장할 수 없습니다.

진짜 문제는 단순히 “중간 정보가 복잡하다”는 것이 아닙니다.

> 한 번의 Agent 실행은 현실의 인과관계를 만듭니다. 두 번째 실행은 첫 번째 실행 이전의 세계에서 시작하지 않습니다.

![텍스트는 다시 쓸 수 있지만 Agent가 바꾼 현실의 상태는 사라지지 않는다](/assets/blog/ai-agent-retry-state/text-vs-world-state-ko.png)

## Agent의 Retry는 최소 네 가지 상태와 연결된다

시스템 관점에서 하나의 Agent 작업은 네 종류의 상태에 영향을 줍니다.

### 1. 대화 상태

사용자 요청, 이전 메시지, 확인된 제약, 추론 맥락, 도구 결과가 포함됩니다.

Retry하려면 어떤 이력을 유지하고 어떤 이력을 버릴지 결정해야 합니다.

### 2. 실행 상태

로컬 파일, Git 작업 트리, 실행 중인 프로세스, 테스트 산출물, 임시 파일, 설치된 의존성이 포함됩니다.

첫 실행에서 이미 모두 달라졌을 수 있습니다.

### 3. 외부 상태

GitHub Pull Request, 데이터베이스 레코드, 전송된 메시지, 클라우드 작업, 제출된 폼, 외부 시스템이 포함됩니다.

완전히 되돌릴 수 없는 경우가 많아 가장 위험한 상태입니다.

### 4. 권한과 감사 상태

사용자가 승인한 작업, 호출된 도구, 발생 시각, 책임을 추적하기 위한 기록입니다.

Retry가 일련의 작업을 조용히 재생한다면 두 번째 실행은 첫 승인 범위에 포함될까요, 아니면 새 승인이 필요할까요? 감사 로그에서는 두 실행의 관계를 어떻게 표현해야 할까요?

이 네 상태가 겹치면 Retry는 단순한 버튼이 아니라 rollback, 중복 실행 방지, 분기 관리를 위한 메커니즘이 됩니다.

## Retry는 실제로 무엇을 의미할 수 있을까

Agent 제품에서 “다시 시도”는 최소 네 가지 서로 다른 동작을 의미합니다.

```text
다시 작성       → 완료된 작업은 유지하고 최종 응답만 다시 작성
다시 계획       → 현재 환경은 유지하되 추론 및 실행 경로를 변경
다시 실행       → 실패한 명령 또는 도구 호출을 다시 실행
Fork로 돌아가기 → 원래 이력을 유지하고 특정 상태에서 새 분기 생성
```

이 동작들의 위험은 완전히 다르지만 하나의 Retry 버튼은 그 차이를 감춥니다.

Agent 제품에도 재시도는 필요합니다. 다만 의미가 더 분명한 동작으로 나누어야 합니다.

- 최종 답변만 다시 생성한다.
- 현재 상태를 유지하며 수정을 계속한다.
- 실패한 단계만 다시 실행한다.
- 이 지점에서 새로운 분기를 만든다.
- 체크포인트로 복원한 뒤 다시 실행한다.

사용자와 시스템 모두 무엇을 반복하는지 알아야 Retry를 신뢰할 수 있습니다.

## Codex는 하나의 Retry 대신 Fork를 제공한다

공식 문서에서 확인할 수 있는 사실은 Codex가 작업을 Thread, Turn, Item으로 모델링하고 `fork` 메커니즘을 제공한다는 것입니다. Codex CLI의 `/fork`는 현재 작업을 새로운 작업으로 복사합니다. App Server의 `thread/fork`도 원래 이력을 보존하면서 새 Thread를 만듭니다.

Fork와 Retry는 비슷해 보이지만 서로 다른 의미를 전달합니다.

- Retry: “방금 시도는 없었던 것으로 하고 다시 해.”
- Fork: “방금 시도는 실제로 일어났다. 여기서 다른 길을 탐색해.”

텍스트만 만드는 Chatbot이라면 첫 번째 의미로 충분한 경우가 많습니다.

하지만 파일을 바꾸고, 명령을 실행하고, 외부 도구를 호출하는 Agent에서는 두 번째 의미가 더 정직하고 추적하기 쉽습니다.

Fork는 역사가 존재한다는 사실을 인정합니다. 새 시도는 이전 과정을 덮어쓰지 않고 독립된 분기와 식별자를 가집니다.

## 많은 경우 Retry보다 피드백이 더 효과적이다

Agent 결과가 마음에 들지 않는다고 해서 전체 작업을 다시 하고 싶은 것은 아닙니다. 보통은 무엇이 잘못됐는지 알려 주고 싶습니다.

- 조사는 유지하되 결론을 더 직접적으로 작성한다.
- API는 바꾸지 말고 내부 구현만 수정한다.
- 테스트 결과는 유지하고 근본 원인을 다시 조사한다.
- 현재 코드는 되돌리지 말고 다른 UI 방향을 탐색한다.
- 실행을 멈추고 현재 변경부터 보여 준다.

이런 피드백은 유용한 작업을 보존하고 이전 실행이 목표에서 벗어난 지점을 정확히 알려 줍니다.

전통적인 Retry는 무작위성에 기대어 다음 답변이 더 좋기를 바라는 방식이었습니다. 좋은 Agent 협업은 동료와 일하는 방식에 가깝습니다. 어긋난 지점을 말하고, 맞는 부분을 남기며, 현재 상태에서 결과를 수렴시킵니다.

## 작은 버튼이 보여 주는 제품의 경계

Codex에서 전통적인 Retry가 눈에 띄지 않는다고 해서 기능이 사라졌다는 뜻은 아닙니다.

그것은 AI가 **답변 기계**에서 **행동 시스템**으로 이동하는 경계를 보여 줍니다.

AI가 텍스트만 생성할 때는 이력을 교체할 수 있습니다. 하지만 AI가 행동하기 시작하면 이력은 시스템 상태의 일부가 됩니다.

중요한 질문은 더 이상 “다시 답할 수 있는가?”에 그치지 않습니다.

- 이전 실행이 무엇을 했는지 정확히 볼 수 있는가?
- 맞는 결과는 보존하고 잘못된 부분만 고칠 수 있는가?
- 명확한 상태에서 다른 길로 분기할 수 있는가?
- 도구와 외부 작업의 중복 실행을 막을 수 있는가?
- 각 시도 사이의 인과관계를 추적할 수 있는가?

> Retry 버튼이 희미해지는 것은 익숙한 상호작용 하나가 사라지는 일이 아닙니다. AI가 현실을 바꿀 수 있게 되면 “다시 시도”는 어느 상태에서 어떤 작업을 반복할 것인지라는 더 무거운 질문에 답해야 합니다.
